摘要（中文翻译，忠实原文）

越来越多场景期待工具使用型 agent能在真实的专业工作流中运行——它们必须解释多模态输入、协调外部工具、检视中间产物、并在最终输出之前修正自己的行为。但现有 benchmark 往往把tool use、computer use、multimodal reasoning 拆开评测，留下了 benchmark 设置与现实端到端全模态工具使用之间的能力断层。

为了弥合这个断层，我们提出 MM-ToolBench——一个面向任务的全模态工具使用 benchmark 与评测 harness。MM-ToolBench 包含 100 个可执行任务，来自两个 macro 任务族——Customer Service 与 Intelligent Creation——覆盖 20 个子类切片，由 27 个 MCP servers / 324 个工具支撑。其核心设计是闭环多模态验证（closed-loop multimodal verification）：agent 必须执行工具、检视渲染或转换后的产物、并在输出不满足任务要求时自我修正。

为了让这种评测可扩展且可验证，MM-ToolBench 把基于 MCP 的执行与任务专属的 grounded 评估器耦合，并辅以一条半自动化的构建管线（覆盖场景发现、任务实例化、评估器合成、人类审核）。

在 15 个当代 agent 模型上的实验显示：MM-ToolBench 高度有挑战性——Claude Opus 4.6（被普遍视为最强 coding-agent 模型之一）task success 仅 32.0%，远低于人类基线的 94.0%。我们将 MM-ToolBench 视为通过闭环多模态验证评估并推动下一代全模态工具使用 agent 的实用基础。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去三周 benchmark 主线一直在做"更接近真实业务"的迁移——5-12 价值评测的 Agent-ValueBench、5-13 长上下文 RAG 的 TRIVIA+、5-14 BenchJack 的 reward-hack 审计、5-15 ClawForge 的"前任烂摊子"、5-18 SaaS-Bench 的真实业务系统（CUA e2e < 4%）。所有这些 benchmark 都共享同一个未解决的痛点：它们要么测 tool use（一次性工具调用），要么测 computer use（GUI 操作），要么测 multimodal reasoning（图片理解），但没有一个把三者打成一个闭环。MM-ToolBench 正是把"agent 自己看自己生成的产物 + 自我修正"做成 first-class——这是从"open-loop benchmark"到"closed-loop benchmark"的范式跃迁。

核心方法：MCP-native + closed-loop verify + grounded evaluator

  • MCP 原生执行：直接采用 27 个 Model Context Protocol servers / 324 个真实工具——这意味着 benchmark 不是"工具的桩"，而是"真实的工具生态"。任何对 MCP 协议的能力差距会立刻被暴露。
  • Closed-Loop 多模态验证：每个任务要求 agent 做"调用工具 → 检视产物（图像/文件/状态）→ 自我修正"的闭环。这把单纯的"输出对不对"升级为"agent 能不能识别自己输出错了，并改正"——更贴近真实人类工作流。
  • Grounded Evaluator + 半自动构建：每个任务都有专属的 grounded evaluator（不是 LLM-as-judge 的笼统打分），构建管线包含场景发现、任务实例化、evaluator 合成、人工审核四步——保证评测器自身可解释、可重现。
  • 双 macro 任务族：Customer Service（更多 tool 协调与用户对话）+ Intelligent Creation（更多多模态生成与产物迭代）。覆盖 20 个子类，确保评测面足够宽。
  • 15 模型 head-to-head + 人类基线：15 个当代 agent 模型直接对比，并提供 94.0% 的人类基线作为天花板——给"agent 距离人类有多远"提供可比刻度。

核心数据亮点 / 关键论点
  • 32.0% vs 94.0%（Claude Opus 4.6 vs 人）：3× 差距——明确说明"看上去强"的 coding-agent 模型在闭环多模态自校验任务上仍是早期阶段。
  • 27 MCP servers × 324 tools：直接接行业标准协议，不是合成工具——意味着任何 MCP 兼容 agent 都可立刻拿来跑分，benchmark 与真实生态零适配成本。
  • per-task grounded evaluator：每个任务都有专属判分器（不靠 LLM-as-judge 的笼统打分），可重现性比 LLM-judge 类 benchmark 高出一个量级。
  • 半自动构建管线：场景发现 → 任务实例化 → evaluator 合成 → 人工审核四步——这是 benchmark 工程化（不是"手工写题"）的范本，可被其他 benchmark 直接复用。
  • 20 个子类切片：覆盖足够宽，避免"在某一类工具上过拟合"——既能给整体打分，也能 per-slice 暴露 agent 的强弱项。
  • 双任务族：Customer Service（对话密集 + 工具协调）vs Intelligent Creation（多模态生成密集 + 产物迭代）——这两类任务对 agent 的能力剖面完全不同，能给出"agent 在不同 workflow 类型下退化模式"的对比。
  • 方法学外推：closed-loop multimodal verification 的设计可直接搬到任意 agent benchmark——只要任务输出能被渲染或转换成可视化产物，就能要求 agent 自校验自修正。

Safety Benchmark（直击主线）

MM-ToolBench 把 benchmark 设计的"真实工具生态"和"闭环自校验"两条主线焊在一起——前者解决"工具有真假"，后者解决"产物有对错"。结合昨天 SaaS-Bench 的"真实业务系统"和 5-15 ClawForge 的"前任烂摊子"，整个 agent benchmark 已经收敛到三轴：（1）真实生态轴（MCP / SaaS / 真用户痕迹）；（2）闭环自校验轴（agent 看自己产物 + 自纠正）；（3）能力剖面轴（task family 切片）。建议在你的"agent benchmark 综述"里把这三轴作为必报维度，并把 MM-ToolBench 的 32.0% vs 94.0% gap 与 SaaS-Bench < 4% e2e 联读——它们一起证明 agent 离"真实工作流交付"还有 1～2 个数量级的工程差。

Harness Engineering

"agent 必须看自己的产物"是一种新型 harness 信号源。当前你的 KV-cache offload runtime 只用 cache hit / latency 这种低级信号驱动决策；MM-ToolBench 提示一种更高级的信号——从输出产物倒推 runtime 决策是否合理。例如：当 prefetch 策略导致下游 logits 出现微小漂移时，可以用一个"产物 verifier"（小 LLM 比较 baseline vs offload 的输出）来判定本轮 offload 是否实际"成功"。这把 hit-rate 这种 cache-级信号升级为"语义级"自校验。结合今日 harness 的 ANNEAL（typed patch + canary）与 agent-safety 的 SPG，整个 runtime 就能：cache 决策 → SPG 测语义损失 → ANNEAL 写回 typed policy patch → MM-ToolBench 风格的产物对比作为最终 ground truth。

Agent Skills Safety

对 skill safety 的启示：skill 不只要能用，更要"自己能识别自己用错了"。MM-ToolBench 的 closed-loop verification 给出了一种结构化做法——任何 skill 输出（包括对 file system 的修改、API 调用结果、生成的多模态产物）都必须可被 agent 自校验。结合今日 safety 选定的 memory laundering 与 5-13 SKILL.md 供应链，可以推断：self-verifying skill 是阻止 laundered 状态污染下游决策的关键防线——只要每条 skill 的输出能被独立 evaluator grounded 验证，错误产物就不会被压入 memory。值得作为 agent skill SDK 的标准接口。
