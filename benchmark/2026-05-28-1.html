一句话核心贡献
把 multi-agent 安全评测的核心度量从"攻击成功率"升级为"伤害放大比 Hglobal/Hlocal"——同一攻击点的局部偏差与全 trace 偏差之比，使"编排如何放大伤害"成为一等可量化指标。
摘要（中文翻译，忠实原文）

多智能体 LLM 系统将工作流分散到多个 agent、工具、共享 context、内存、决策门。这种模块化提升了可解释性，但也产生了传播风险——对一个组件的有限扰动，可以被其他 agent 反复重用并放大为系统级伤害。

我们提出 HARP（Harm Amplification by Role Perturbation），一种追踪优先方法论，用于研究 multi-agent LLM 系统中"局部到全局"的伤害放大。HARP 配对 clean / perturbed 两次执行，记录 expert output、tool call、memory read/write、guard event、oracle log、延迟、token 成本和决策。定义局部伤害为目标 agent 或受损通道的偏差、全局伤害为整 trace 的偏差、伤害放大为 H_global/H_local——这一指标补充 ASR，提供"编排在攻击点之外如何继续放大伤害"的强度度量。

我们在一个面向金融的 7-agent 系统中实例化 HARP，配以可配置攻击工具：expert compromise / collusion / shared-context corruption / 时间或内存持久攻击。在 5 种防御中：仅 prompt 防御保留良性效用、但放任高 ASR + 高隐蔽；tool-pre / step-level guard减少部分故障、但带效用或延迟代价；IntegrityGuard（trace 一致性防御）实现最低 ASR + 最低全局伤害，但引入效用/成本权衡。结论：单点 expert compromise 放大最强、shared-context corruption ASR 最高、time-persistence 恶意影响最大。HARP 主张multi-agent 安全评测必须测量"传播"，不能只测"绕过"。

核心内容解读（伤害形式化 · trace-first 评测 · 4 类攻击 vs 5 种防御）

问题背景：当前 multi-agent 安全 benchmark（如 AgentBench、AgentDojo、HarmBench）大多沿用 single-LLM 的 jailbreak 框架——以 ASR（Attack Success Rate）为唯一一阶指标。但 multi-agent 的本质风险不在"是否绕过"，而在"一个被绕过的点，会不会通过工具调用 / 共享内存 / 决策门，被其他 agent 重复使用、滚雪球放大"。这种"传播性伤害"在 ASR 框架下被压扁为一个 0/1 标签，无法刻画。HARP 是首个把这个问题形式化为可量化指标的 benchmark 工作。

三个核心定义：

量定义含义
Hlocal受攻击 agent / 通道在 perturbed run 与 clean run 的输出偏差"攻击点本身造成多大伤害"
Hglobal整条 trace（含所有下游 agent / tool / memory）在 perturbed 与 clean run 的总偏差"所有 agent 加起来一共造成多大伤害"
Harm AmplificationHglobal / Hlocal"编排把单点伤害放大到全局的倍数"——本论文一阶指标

4 类攻击工具：论文设计 7-agent 金融系统作为靶台，对每个工具配上独立攻击器——

  • Expert Compromise：单个 agent 被注入恶意 system prompt，输出仍合法但偏向损害。
  • Collusion：两个 agent 串通，组合输出绕过单 agent 守卫。
  • Shared-Context Corruption：共享 memory 中的元数据被篡改，下游 agent 在不知情的情况下用受污染的 context。
  • Temporal / Memory Persistence：攻击在某一步注入，但通过持久化写入影响后续 N 步——评测"持久化攻击的长尾效应"。

5 种防御机制对比：

防御ASR全局伤害良性效用延迟代价
Prompt-only高高保留低
Tool-pre Guard中中轻度损失中
Step-level Guard中中中度损失高
Output Verifier中中轻度损失中
IntegrityGuard（trace 一致性）最低最低损失高

这一对比表给出 multi-agent 安全防御的"效用-安全 Pareto 前沿"——单纯 prompt-level 防御（同 5-25 #2 SafeHarbor 的 L1 层）无效，必须做 trace 一致性级别的横向校验，但代价是延迟与效用都要付。

三个反直觉发现：

  • "单点 expert compromise 放大最强"——比看起来"更协调"的 collusion 还要严重，因为没有任何 guard 设计预期的"内部 agent 是恶意的"假设。
  • "shared-context corruption ASR 最高"——shared memory 是 multi-agent 系统最薄弱的环节，污染一次、整链中招。
  • "time persistence 恶意影响最大"——一次注入贯穿后续 trace 的所有步，是单步 guard 完全看不见的盲区。

与 5-25 #2 PoisonForge 的关系：PoisonForge 是单 agent / 单维 attack benchmark；HARP 是 multi-agent / 全 trace 系统级 benchmark。两者形成"单 agent → 多 agent"演进：PoisonForge 测"1% 投毒预算如何击穿单模型"，HARP 测"击穿后伤害如何在 7 agent 之间放大"。把这两个 benchmark 串起来即可构成一个完整的"注入 → 放大 → 系统级伤害"评测流水线。

与今日 #2 Snyk Skill Ecosystem 的关系：Snyk 给出"76 个真实恶意 skill 已经在线"的事实；HARP 给出"这些恶意 skill 一旦进入 multi-agent 系统会放大几倍"的方法论。这两篇拼起来正好给毕设/未来研究一个完整的 threat-to-impact 评测路径。

核心数据亮点 / 关键论点
  • 新一阶指标 Hglobal/Hlocal——把"编排放大伤害的倍率"从隐喻变成可量化数字，补足 ASR 的盲点。
  • 7-agent 金融系统——在风控、报价、撮合、清算、合规、审计、客服 7 个 agent 上测试，覆盖工具链丰富的产业级场景。
  • 4 攻击 × 5 防御 = 20 组矩阵——给出 multi-agent 安全防御的效用-延迟-安全 Pareto 前沿。
  • "单点 expert compromise 放大最强"——反直觉地比 collusion 更危险，因为缺少"内部 agent 也可能恶意"的假设。
  • "shared-context corruption ASR 最高"——shared memory 是当前 multi-agent 系统最薄弱的攻击面。
  • "time persistence 恶意影响最大"——单步 guard 对持久化注入完全失效，必须做 trace 级一致性校验。
  • 仅 prompt 防御无效——再次实证 5-25 #2 SafeHarbor 的"分层 guardrail 必要性"。
  • IntegrityGuard 是当前最优防御——但代价是 utility 与 latency 都要付出，意味着 multi-agent 安全无 free lunch。
对你三个研究方向的启发
Harness Engineering

HARP 的"trace-first"评测路径给 harness 设计提供了一个新维度——你今日 #1 那篇 Policy-Driven Runtime 的 9 策略表里"safe execution"原本是孤立条目，HARP 给出了它的具体评测协议：每条 trace 要记 agent 输出 / tool 调用 / memory 读写 / guard 事件 / oracle 日志 / 延迟 / token / 决策 7 维。把这一协议接入 CacheSage 的 observe 原语，就可以让 KV cache 调度同时承担 safety trace 收集，在生产环境 zero-overhead 完成 multi-agent 安全审计。

Agent Skills Safety

HARP 的 4 类攻击恰好对应今日 #2 Snyk 论文里的真实恶意 skill 模式——expert compromise = SKILL.md 注入 / collusion = 同作者 40+ skill 串联 / shared-context corruption = SKILL.md 写入共享 memory / time persistence = auto-update 持久化。建议：把 Snyk 的 76 个真实恶意 skill 直接灌进 HARP 的 7-agent 系统，跑一次端到端伤害放大测试——这会成为第一份"真实 skill × multi-agent 系统级伤害"实测数据，高度有发表价值。
