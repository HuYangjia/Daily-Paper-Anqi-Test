摘要（中文翻译，忠实原文）

企业 Agent 越来越多地在受范围限制的检索系统、被委托的工作流、和受策略约束的证据环境中运行。在这些场景下，访问控制（access control）可以被正确执行，但系统仍可能给出"看上去完整"的答案，尽管关键证据其实落在调用方授权边界之外。本文提出 Partial Evidence Bench，一个用于测量这一失败模式的 deterministic benchmark。

该 benchmark 提供 3 个场景族——尽职调查（due diligence）、合规审计（compliance audit）、安全事件响应（security incident response）——共 72 个任务。每个任务自带：① ACL-分区的语料；② oracle complete answer；③ oracle authorized-view answer；④ oracle 完整性判定；⑤ 结构化 gap-report oracle。

评测沿 4 个评估面展开：①答案正确性；②完整性意识（系统是否承认证据缺失）；③gap-report 质量（缺口报告的结构化程度）；④不安全的完整性行为（是否在授权受限时仍宣称完整）。已签入仓库的 baseline 显示：沉默过滤（silent filtering）在所有场景族中都是灾难性不安全；而显式的 fail-and-report 行为可以消除"不安全完整性"，且不会让任务退化为"什么都拒答"。初步真模型试跑显示，不同模型在过度声称完整、保守式低估或以企业可用形式报告不完整之间存在模型相关、场景敏感的差异。

该 benchmark 更广义的贡献是：让一种治理关键的 Agent 失败模式变得可度量，而不需要人工 judge 或易污染的静态语料。

核心内容解读（背景·方法·差异）

解决了什么问题：当下 agent benchmark 普遍假设"所有证据都对 Agent 开放"，所以评测点是"答案是否正确"。但企业部署 Agent 的真实场景几乎都带 ACL（Access Control List）：一个 PM 角色看不到 HR 的薪酬表，一个外审会计看不到公司加密通讯，一个 SOC 分析师看不到 HR 调查档案。问题不在 ACL 错了，而在于 Agent 在 ACL 正常工作的前提下，仍然把"我能看到的那一半证据"包装成"完整答案"——这是一种被作者命名为"silent filtering"的治理级失败。它过去之所以没被严肃评测，是因为它需要：① 同一份语料的两套答案 oracle（complete vs authorized-view）；② 一种"完整性意识"的形式化定义；③ deterministic 评测，避免 LLM-as-judge 污染。Partial Evidence Bench 第一次把这三件事一起搞定。

核心方法：

  • 3 个场景族 = 现实最常见的 ACL-敏感工作流：① due diligence（投资尽调，部分文档非授权方不可见）；② compliance audit（合规审计，跨部门隔离）；③ security incident response（事件响应，HR/法务证据天然隔离）。
  • 72 个任务，ACL-partitioned 语料：每任务的语料 split 成 authorized 与 restricted 两份；question 一致；正确行为依赖 Agent 是否承认 restricted 部分存在。
  • 5 件套 oracle：① complete answer（基于全语料的真值）；② authorized-view answer（基于授权部分的真值）；③ completeness judgment（这两份答案是否一致——决定 ground-truth 的"是否需要承认证据缺失"）；④ structured gap-report（什么类目缺、缺多少）；⑤ ACL 划分本身。
  • 4 评测面：① answer correctness（答案正确性）；② completeness awareness（"我看到的够不够"的元认知正确性）；③ gap-report quality（缺口报告结构化程度）；④ unsafe completeness behavior（在缺证据时仍宣称完整的"治理级"严重错误）。
  • Deterministic 评测：每条 oracle 都是结构化数据 + 程序化匹配，不依赖 LLM-as-judge——避开了今日 benchmark 之痛。
  • Baseline 启示：silent-filter Agent 在所有场景族都灾难级不安全；fail-and-report Agent 能消除 unsafe completeness 而不会退化为"什么都拒答"。

这篇是典型的"小而尖"benchmark。72 任务规模不大，但因为它把"授权 vs 完整"这一类企业语境下高频但被忽视的失败原型形式化、结构化、可程序判定，因此对企业 Agent 部署具有非常直接的工程价值。它和今日 Agent Safety 报告 Chain of Risk 是互补的——后者把 reasoning trace 和 final answer 作为评测两段，前者把"是否承认证据不全"作为答案正确性的第三个维度；二者一同把 agent 安全评测从"做对没做对"的二分推到"做对、知道做对了多少、报告做对到哪儿了"的三分。

核心数据亮点 / 关键论点
  • 3 场景族 × 72 任务：覆盖企业 Agent 最常见、最 ACL-敏感的三类工作流；规模虽小但每任务结构高度形式化。
  • 5 套 oracle：complete / authorized-view / completeness judgment / structured gap-report / ACL split——任意工程师都能 deterministic 地复现评分。
  • 4 评测面：answer correctness / completeness awareness / gap-report quality / unsafe completeness behavior，第一次把"完整性意识"从"答案对错"中切出来作为独立维度。
  • Silent filtering 在所有 3 场景族都灾难级不安全——这是 benchmark 给业界的最直接结论。
  • Fail-and-report 能消除 unsafe completeness 而不退化为 trivial abstention——给"安全 vs 实用"trade-off 提供了非平凡的中间解。
  • Deterministic + 合成语料：抗污染（不会被新的 LLM 训练数据吞掉）、不依赖人工 judge——这两点恰好回应 Judge Sensitivity 与 BenchGuard 上周指出的两类 benchmark 失效来源。
  • 真模型初探显示模型相关分布差异：有的模型 over-claim，有的 under-claim，有的真的写出企业可用的 gap report——给"可信赖 Agent vs 自信幻觉"的工业选型提供量化依据。
对你三个研究方向的启发
Harness Engineering

Partial Evidence Bench 给 evaluation harness 提供了一个很好的"oracle 多重视角"模板：每条任务带多套 ground-truth（complete / authorized-view / gap-report），评测器同时输出多面分数，而不是合成一个"总分"。这可以直接映射到你 ZipperQuant 的 e2e benchmark：除了"端到端正确率"以外，还应该附 "FP16 reference / W4A4 quantized / W4A4+sparse" 三套答案的差异 vector，把"量化是否破坏哪一类问题"做成 per-category gap-report——这样审稿人能一眼看出量化退化集中在哪一类 prompt 上。它也补上昨日 CCI 的另一面：评测 harness 不仅要做"组件子集 × 性能"二维表，还应记录"每个子集的失败原型分布"，这就需要 Partial Evidence Bench 这种 multi-oracle 设计。

Agent Skills Safety

"silent filtering 在所有场景都灾难级不安全"是这篇给 agent safety 社区的一个核心观点。它直接对应今日 Agent Safety 报告 Chain of Risk 中的 "leak case"：当 reasoning trace 看到了不该看的、final answer 给出了"看似安全的拒答"，本质上就是一种 silent filtering。两篇拼起来等于：不论是 ACL 切走的"我没权限看到的"还是 safety policy 切走的"我不该说的"，Agent 都倾向于"用看起来完整的答案掩盖证据残缺"，这是 alignment 一直没被严肃评测的一类元认知失败。建议把 Partial Evidence Bench 的 4 评测面平移到 agent-safety guardrail 评测里。

Safety Benchmark

Partial Evidence Bench 给下一代 safety benchmark 设计提供了 3 个可立即复用的设计原则：① 多 oracle 同时评分（消除"单一总分"导致的维度坍缩）；② deterministic 评测（不被 judge LLM 漂移影响）；③ 合成语料（抗训练污染）。这三件加起来正好是过去一年 benchmark 危机（judge 敏感性、live snapshot、benchguard）的解药雏形。下一步自然是把 Partial Evidence Bench 和 XL-SafetyBench 的语言/文化轴、DTap 的 14 域 × 50 环境拼成"权限 × 域 × 文化 × 攻击面"的四维 agent safety benchmark。
