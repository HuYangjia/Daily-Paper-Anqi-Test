摘要（中文翻译，忠实原文）

本文提出 SGR-Bench，一个用于评估 search agent 在 "状态门控检索（State-Gated Retrieval, SGR）" 任务上的 benchmark。SGR 任务 是指：要拿到答案，agent 必须先发现合适的网站，并把该网站的检索状态（filter、view、scope、hierarchy）配置对——只有当状态配对了，对应的结构化答案才会浮现出来。这类任务在政府数据门户、统计局、专利库、学术数据库、地图服务等公共数据生态中极为常见，但传统 search benchmark 都把它绕开。

SGR-Bench 包含 100 个专家策划任务，覆盖 6 个源族 与 12 个公共数据生态。每个任务需要 agent 同时完成"找到正确网站 + 配置正确状态 + 产出结构化答案"三件事。同时提供双重问题表述——约束引导（明示要满足哪些约束）vs 目标导向（只给目标，约束需自行推断）——支持显式 vs 隐式指导对比。

评测了 8 个 CLI 代理 LLM 系统 与 3 个商业 search-agent 产品。最强系统的项级 F1 仅 66.18%。对 156 条可分析失败轨迹的人工审计揭示了主导失败模式：检索范围漂移（37.2%） 与 标准不匹配（27.6%）——agent 容易在非靶网站、错误层级或错误过滤器下交付"看似合理但全错"的答案。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去半年的 agent benchmark 已经把"能不能完成最终任务"这件事打磨得很细：5-14 BenchJack（reward-hack taxonomy）、5-15 ClawForge（state conflict）、5-19 MM-ToolBench（closed-loop tool）、5-20 POLAR-Bench（5×5 诊断面）、5-21 GrandGuard（context risk）。SGR-Bench 把这条线推到一个更微观的失败族：在很多公共数据生态里，"找对网站" 还不够——你必须把过滤器、视图、scope、hierarchy 这些状态都配置对，结构化答案才会显形。这是被既有 benchmark 系统性绕过的一类任务，但在政府开放数据 / 统计局 / 专利库等场景里恰恰是真实工作量的最大头。

核心方法：state-gated 任务族 + 双问题表述 + 项级 F1 + 人工审计

  • 定义新任务族 SGR。把"状态配置"列为搜索任务的一等公民——和"找网站"、"查询关键词"并列。SGR 任务的成功定义为状态被配对 + 结构化答案被取出，而不是任意能给出文字答案就算赢。
  • 100 专家策划任务 × 6 源族 × 12 数据生态。源族包含政府开放数据门户、统计局、专利库、地图服务、学术 API、文档系统等。每个任务由领域专家策划——保证状态门控的真实性而非合成。
  • 双重问题表述：(a) 约束引导——明示"请在这些约束下完成"，agent 只需把约束翻译成状态；(b) 目标导向——只给目标，agent 必须自己推断该开哪些过滤器。两条 setup 让"显式指引 vs 隐式自主"的能力差被分离测量。
  • 项级 F1 评分 + 156 轨迹人工审计。用 item-level F1 计算"对几项答对几项"，再对所有可分析失败轨迹做人工归因——产出 7 类失败模式，其中两类占 64.8%：检索范围漂移（37.2%） + 标准不匹配（27.6%）。

核心数据亮点 / 关键论点
  • 项级 F1 上限 66.18%——最强 search-agent 系统在 SGR 任务上仍然有约 1/3 项无法答对。
  • 检索范围漂移 37.2%——agent 经常在非靶网站、错误 scope 或错误层级下"假装在搜"。
  • 标准不匹配 27.6%——agent 找到了正确网站，但没有把过滤器调到正确组合，给出"看起来对、其实错"的答案。
  • 双问题表述——约束引导 vs 目标导向，让"显式监督 vs 自主推断" 的能力差被显式分离。
  • 12 公共数据生态 / 6 源族——政府数据门户、统计局、专利库、地图、学术、文档；覆盖真实工作量主流。
  • 156 失败轨迹人工审计——把"为什么失败"从模型评判变成可校验的人工归因。
  • 已开源数据集——Hugging Face PKUAIWeb/SGR-BENCH，可直接复现。
对你三个研究方向的启发
Harness Engineering（毕设可借鉴）

把"state-gated"这一概念跨域迁移到 KV-cache 仿真评测：你毕设里"prefetch + 双水位线驱逐"也是一种"state-gated"——只有当 GPU 缓存的水位、热度图、轨迹簇等多状态变量同时配对，hit 才会发生；评测时不能只看 hit rate，还要看"状态配对成功率"和"失败时是哪一项状态没对齐"。具体可以借鉴 SGR-Bench 的"项级 F1"思想——把 hit 拆成"token 条目级是否被预取对" 和 "预取时机是否对" 两条独立指标，再像 SGR-Bench 一样做失败轨迹归因。这正好对应你毕设附录补充的 CNN/Linear 学习型预取器对照实验需要展示的"不仅有结果，还能解释为什么"。

Agent Skills Safety

对 agent safety 的迁移：把 SGR 的"状态配对错"翻译成 safety 域的"权限范围错"——很多 agent 安全失败并不是"做了不该做的事"，而是"在错误的 scope 上做了对的事"——这跟 5-21 CUGA Policy 的 Intent Guard、5-13 SKILL.md 的 governance 绕过、5-20 ECA 的 authorization-failure 同源。可以做一个"Safety-SGR" 类比 benchmark：让 agent 必须先配置好 ACL / scope / persona，才能拿到合法答案——失败归因 7 类直接套用。这给"agent skills safety"补一个非常缺的权限-状态评测面。

Safety Benchmark（直击）

对 benchmark 设计的方法论是直接弹药：(1) 把"项级 F1" 写进所有 benchmark 标准报告项——单值正确率太粗、不利于诊断；(2) 双问题表述协议（约束引导 vs 目标导向）是任意 agent benchmark 都该补上的——它把"显式监督蒸出能力"（呼应 5-21 AgentAtlas）量化得更细；(3) 156 轨迹人工审计 + 7 失败族是细粒度诊断的范式——既不是单跑分也不是全自动 LLM-judge，而是"跑分 + 抽样人工归因"两层；(4) 状态配对作为新评测维度——所有跨网站 / 跨 SaaS / 跨 MCP 的 benchmark 都该加上，跟 5-18 SaaS-Bench 的 weighted checkpoint 互补。
