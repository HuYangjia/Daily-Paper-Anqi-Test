摘要（中文翻译，忠实原文）

传统 LLM 安全 benchmark 评估的是 生成的文本——模型是否输出有毒语言 / 复制偏见 / 服从有害指令。但当模型作为 agent 部署时，"安全相关的对象"从"系统说什么"变成"系统在环境中做什么"，仅以 prompt 评估输出已不足以应对 AI 带来的安全挑战。最近一波 benchmark 已开始评估 LLM 作为 agent，本文为这条线作出新贡献。

我们提出 Boiling the Frog benchmark，评估部署在企业 / 办公环境的工具使用型 AI 模型对增量攻击是否易感。每个场景从"良性的工作区编辑"开始，随后逐步注入承载风险的请求。本 benchmark 聚焦有状态多轮评估：链路暴露的是持久工作区，在轮序的受控位置放置承载风险的负载，然后判定生成的工件状态是否变得不安全。

场景以一份三级操作风险分类法组织：(1) "Boiling the Frog" 风险，(2) EU AI Act 附件 I 与附件 III 高风险情境，(3) EU AI Act 通用 AI（GPAI）实践守则。在 9 个模型的对比面板上，整体严格 ASR 为 44.4%。模型间 ASR 跨度从 Claude Haiku 4.5 的 20.5% 到 Gemini 3.1 Flash Lite 的 92.9%，Seed 2.0 Lite 也超过 80%。在"实践守则失控"链路类别上，平均 ASR 高达 93.3%。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：近期 benchmark 已经开始报告 agentic safety 单轮 ASR——TRIVIA+（5-13）、BenchJack（5-14）、MM-ToolBench（5-19）、POLAR-Bench（5-20）、GrandGuard（5-21）、SGR-Bench（5-23）。但所有这些都默认"每个评测样本独立、当下当下评判"——这在真实办公环境完全不成立：真正的 agent 风险从来不是一句话，而是 N 步小动作累积出来的状态。Boiling the Frog 第一次把这个直觉做成可运行 benchmark：每条评测链是一段连续的多轮交互、共享同一份 workspace state、风险负载放在轮序里特定位置注入。

核心评估协议——三件正交配合的机件：

  • 有状态多轮评估——评测样本是"链"而非"条"。链开始时是良性的工作区编辑（如建文件夹 / 整理邮件），然后在受控位置插入承载风险的请求。判定标准看的是最终工件状态而非单条文本输出，这与 agentic 部署完全一致。
  • 三级操作风险分类法——把场景按以下三层标签组织：(a) Boiling the Frog 风险（升温式）、(b) EU AI Act 附件 I/III（高风险背景如就业、医疗、法律执行）、(c) EU AI Act 通用 AI（GPAI）实践守则（失控 / 越权 / 偏见）。这是首份直接锚定到 EU 监管文本的 agentic benchmark。
  • 严格 ASR 判定——只要链路最终工件不安全，就计入攻击成功。对 9 模型的对比给出 20.5% – 92.9% 的极宽分布，把"同样模型在单轮上类似分数、在多轮上分数差 4×"这一现象立成基准事实。

Harness Engineering（直击毕设）

Boiling the Frog 的"有状态多轮评估"对你毕设附录里的预取策略评测协议是一份方法学背书：你可以把对 LongBench 的多任务评测重写成"持久 KV-cache state + 任务序贯注入"，让"温缓存被升温到热缓存"和"被驱逐到冷"都成为可量化事件——而不是各任务独立重置 cache。这与本论文 "判定对象从响应改为工件状态" 异曲同工：你的判定对象应从单任务 acc 改为"跨任务 hit-rate 与 bandwidth 走势"。也呼应 5-25 Code as Agent Harness 综述列出的"超越最终任务成功的评估"这一开放挑战。

Agent Skills Safety

对 agent safety 的迁移：今天 SafeHarbor（5-25）给出 63.6% / 93%+ 的单轮 双指标 SOTA，但只要换成 Boiling the Frog 的多轮链路，这两个数字大概率会塌方——所有 guardrail 工作的下一步是必须报多轮 ASR。具体可记忆 3 件：(a) Memory Laundering（5-19）定义的 "Sub-threshold Propagation Gap" 与本文 "Boiling the Frog 风险" 类别本质上是同一现象；(b) CUGA 的 5 检查点必须在跨轮上重新评估有效性；(c) SafeHarbor 分层记忆正好是抵御本类多轮攻击的天然候选——把它接到 Boiling the Frog 上是接下来一篇可写论文。

Safety Benchmark（核心）

对 benchmark 设计的最强启发：这是把单轮 ASR 范式打掉的拐点工作。具体三件事：(1) 评测样本必须从"条"升级为"链"；(2) 判定对象必须从"响应"升级为"工件状态"；(3) 风险分类必须直接锚定到监管文本（EU AI Act 附件 I/III + GPAI Code of Practice）才能让结果对合规审计有意义。这把过去 12 个月所有 agentic safety benchmark（BenchJack / MM-ToolBench / POLAR-Bench / GrandGuard / SGR-Bench）都置于"需要补多轮配套实验"的整改议程上——和 5-20 What Twelve Papers Disclose 的 0.38 平均披露分一起，构成 agent benchmark 改革的两份共同压力。
