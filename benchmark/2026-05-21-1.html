摘要（中文翻译，忠实原文）

随着老年人越来越多地使用基于 LLM 的聊天机器人来获得陪伴与帮助，一个安全鸿沟正在浮现。老年人可能因社会孤立、数字素养有限、认知衰退而面临脆弱性，但现有的安全 benchmark 主要瞄准普适性危害，忽视了老年人专属的风险。例如，"如何在黑暗里独自修理天花板上的灯"对大多数用户是良性的，但对行动不便的老年人构成严重的跌倒风险。

我们提出 GrandGuard——首个用于评估并缓解 LLM 交互中"老年人专属 contextual 风险"的综合框架：

  • 构建一份3 级 taxonomy，包含 50 类细粒度风险，覆盖 5 大领域：心理健康 / 财务 / 医疗 / 毒性 / 隐私——基于真实事件、社区讨论、利益相关者研究分析。
  • 基于该 taxonomy 构建一份 10,404 条 prompts/responses 的 benchmark，揭示多个 leading LLM 在 50%+ 案例中误处理老年人专属上下文风险。
  • 提出两种 safeguard：fine-tuned Llama-Guard-3（96.2% 不安全 prompt 检出准确率）与 policy-enhanced gpt-oss-safeguard-20b（90.9%）。

GrandGuard 为"超越普适安全、走向支持老龄人口的 AI 系统"奠定基础。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去三周的 benchmark 主线把"评测面拓宽"做到了一定饱和——5-12 Agent-ValueBench 价值观、5-13 TRIVIA+ 视觉幻觉、5-14 BenchJack reward-hack、5-15 ClawForge CLI agent、5-18 SaaS-Bench、5-19 TOBench、5-20 POLAR-Bench privacy-aware adversarial。GrandGuard 把这条线推到了一个全新的正交方向："population-specific risk"——同样一句 prompt 对不同人群安全意义截然不同，这是过去所有 safety benchmark 的共同盲区（AgentAtlas 同日的 5-21 报告也指出了"benchmark coverage 审计"的必要性）。

核心方法：双轴 grounding × 3 级 taxonomy × 双 safeguard

  • 双轴 grounding 数据源：风险类型不是凭空臆造——来自真实事件 / 社区讨论 / 利益相关者研究分析的三元 grounding，避免"benchmark 反映 benchmark 团队偏见"。
  • 3 级 taxonomy（粗→中→细）× 50 fine-grained 风险：跨 5 大领域分布——心理健康、财务、医疗、毒性、隐私——确保每条风险都有清晰的类别归属。
  • 10,404 prompts/responses 标注集：每条 prompt 都附带"在该上下文下应被如何处理"的金标——比"该不该回答"更细，是"该如何安全地回答"。
  • 双 safeguard 基线：fine-tuned Llama-Guard-3（96.2%）走 supervised 路径；policy-enhanced gpt-oss-safeguard-20b（90.9%）走 declarative policy 路径——给后续工作提供两条对照路径。
  • "contextual risk"概念正式入册：风险不是 prompt 的固有属性，而是prompt × 人群的二元乘积——这是一个对 safety benchmark 设计有根本性影响的概念升级。

核心数据亮点 / 关键论点
  • 10,404 labeled prompts/responses × 3 级 × 50 类风险——首份关于老年人专属风险的工业级规模 benchmark。
  • 5 大风险域：心理健康 / 财务 / 医疗 / 毒性 / 隐私——覆盖老年人面临的核心 contextual 危害。
  • 多个 leading LLM 误处理 > 50% 案例——这是 safety benchmark 文献里少见的"主流模型严重失败"信号，意味着 contextual risk 是 LLM 安全的真盲区。
  • Fine-tuned Llama-Guard-3 达到 96.2% 不安全 prompt 检出——比通用 safety guard 在该域有显著提升，证明population-specific 微调是有效的。
  • Policy-enhanced gpt-oss-safeguard-20b 达 90.9%——declarative policy 路径仅以 5 pp 落后，但无需重训，这是部署灵活性的关键证据。
  • "独自修灯泡"案例——同一句 prompt 对不同人群安全语义截然不同，这是 contextual risk 概念的最佳教学样本。
  • "contextual risk" 是 GrandGuard 留给社区的一个 first-class 概念——未来 KidGuard / DisabilityGuard 都将沿用这套定义。
对你三个研究方向的启发
Harness Engineering（直击毕设）

GrandGuard 的"contextual risk"概念可以直接对位你 KV-cache 评测的"contextual hit/miss"——同一个 prefetch 决策对不同 working set 大小、不同 sequence 长度、不同 batch 形状下"是否命中"语义截然不同。借鉴 GrandGuard 的方法，把毕设第三章的策略评测从"策略 × benchmark"二元升级为"策略 × benchmark × 上下文配置"三元——例如分别报告 "短序列 + 大缓存 / 长序列 + 小缓存 / 多轮对话场景" 三种 contextual 切面下的 hit rate。这跟同日 5-21 AgentAtlas（harness）的"双 prompt 模式"是同向方法学——揭示"看似一致的指标其实在掩盖上下文敏感的失败"。

Agent Skills Safety

对 agent safety 的迁移路径：把 SKILL.md 安全也做成"contextual skill risk"——同一个 skill 对不同用户角色（admin / dev / end-user）、不同环境（生产 / 沙箱 / 演示）、不同上下文（在 IDE 内调用 / 在 CI 流水线内调用）有截然不同的安全语义。这和 5-21 同日的 CUGA 5 检查点架构互补：CUGA 给出"在哪些点拦截"，GrandGuard 给出"按什么 context 来判定该不该拦"。两者结合，agent 安全系统才能既有结构化拦截又有 context-aware 判定。

Safety Benchmark（核心）

GrandGuard 是一份直接可复制的benchmark 设计模板：(a) 三元 grounding（真实事件 + 社区 + 利益相关者）避免单一来源偏见；(b) 3 级 taxonomy 让风险细到"独自修灯泡"层级——粒度足够暴露 LLM 真实失败；(c) 双 safeguard 路径（fine-tune vs declarative policy）作为baseline 比较——这一条直接映射到 5-21 CUGA 的 policy-as-code 路线。同时，"population-specific safety benchmark" 这条新赛道未来会催生大量子方向工作——这是一份 high-leverage 的 benchmark 论文。它跟 5-20 POLAR-Bench 和 5-19 TOBench 一起，构成 2026-05 整月 benchmark 方向的"概念升级浪潮"——从单维度 ASR 走向 contextual / policy-aware / population-specific 三向同时进化。
