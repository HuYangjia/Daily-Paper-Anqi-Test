摘要（中文翻译，忠实原文）

Computer-Using Agent（CUA）正在快速把大语言模型（LLM）从基于文本的推理扩展到更复杂环境中的动作执行，例如浏览器与图形界面（GUI）。然而，现有的 web 与 GUI agent benchmark 通常依赖简化场景、孤立任务或短 horizon 交互，使得我们难以评估 agent 在真实专业工作流里的能力。Software-as-a-Service（SaaS）环境是 CUA 评测的天然选择——它承载现代数字工作的大部分，并自然涉及动态系统状态、跨应用协作、领域专门知识与长 horizon 依赖。

为此我们提出 SaaS-Bench：基于 23 个 deployable SaaS 系统、覆盖 6 大专业域、包含 106 个真实工作场景任务。这些任务需要长 horizon 执行、覆盖纯文本与多模态两种设置，并用 加权 verification checkpoints 评分——同时衡量严格的任务完成度与部分进展。

实验显示：代表性 LLM-based agent 在 SaaS-Bench 上挣扎——最强模型端到端完成率不足 4%，暴露在规划、状态追踪、跨应用上下文维护与错误恢复方面的局限。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去三周 benchmark 主线把"评测面要看什么"几乎讨论了一遍——5-08 XL-SafetyBench 多语言（5-08）、5-12 Agent-ValueBench 28 价值系统（5-12）、5-13 TRIVIA+ desiderata（5-13）、5-14 BenchJack 红队 benchmark 自身（5-14）、5-15 ClawForge 让 agent 处理"前任的烂摊子"（5-15）。但这些 benchmark 几乎都构造自合成或半合成环境。SaaS-Bench 是第一份大规模把真实部署的商业 SaaS当 benchmark 后端的工作——它把"评测真实性"这个变量从 simulator-style 推到 production-backed。结果震撼：在真业务流上，最强模型的端到端完成率不足 4%——这把 OSWorld、WebArena、AgentBench 等 benchmark 上的 30%~70% 数字"打回原形"，意味着主流 CUA 评测可能严重高估了 agent 在生产里的可用度。

核心方法：四个关键设计选择

  • 真 SaaS 后端：23 个 deployable SaaS 系统，覆盖 6 大专业域——不是 mock 环境。这意味着 agent 必须面对真实的 latency、动态 UI、版本演化、跨应用集成等"工程现实"，而不是合成场景。
  • 106 任务 grounded in realistic work scenarios：任务由专业用户写，而不是众包标注——直接指向"真实业务流"而非"演示用例"。
  • 加权 verification checkpoints：这是评测面的关键创新——不是 binary 完成/未完成，而是把每个任务拆成多个 weighted checkpoint，分别报告 strict completion 与 partial progress。这允许 benchmark 既给出严格分数、又能追踪 agent 在长 horizon 里的"卡在哪一步"。
  • 长 horizon + 多模态：纯文本 + 多模态都覆盖；任务涉及 cross-application coordination 与 state tracking——这两个维度恰好是过去 benchmark 的盲点。

核心数据亮点 / 关键论点
  • e2e < 4%：最强模型端到端完成率不足 4%——这是 CUA benchmark 历史上最低的 frontier 数字之一，意味着真实 SaaS 环境的难度被严重低估。
  • 23 SaaS × 106 任务 × 6 域：覆盖宽度足以撑住"行业 benchmark"定位；不是单一 vertical 的玩具。
  • weighted checkpoint 评分：把"长 horizon 任务"评分从二值改成连续——这给后续 benchmark 提供了一份可继承的协议，让"50% 进度"和"完全失败"得到区分。
  • 4 个失败模式被显式命名：planning / state tracking / cross-application context maintenance / error recovery——这把"agent 不会做"拆成了 4 条可独立改进的研究方向，对后续 RL / 训练数据合成研究有指导价值。
  • 开源 deployable：GitHub: UniPat-AI/SaaS-Bench——这意味着学界可直接对接，不需要再花数月自己接 SaaS API。这把 SaaS-Bench 从"调查报告"升级为可作 leaderboard 的基础设施。

对你三个研究方向的启发
Harness Engineering

SaaS-Bench 的"weighted checkpoint"评分协议是 harness 工程的可借鉴范式：不要把任务结果折成单一标量。这与今日选定的 CAX-Agent 的 recovery ladder 同源——都在"把过程信息显式化"。具体落到你 KV-cache offload 实验：可以把"prefetch 是否成功"做成 weighted checkpoint——①top-k 命中、②被预取层在 PP overlap 内 finalize、③decoder 正确读取、④无 stall——分别给权重而非二值评估。这会让你 v3 实验章的论证更细粒、更难被审稿人挑刺。

Agent Skills Safety

SaaS-Bench 间接给 agent safety 一个尖锐警告：当模型在真实业务流上 e2e 完成率 < 4% 时，"安全 vs 功能"的张力图景被改写——大量"安全失败"其实是"功能失败的副产物"。今日选定的 Ensemble Monitoring 给出的"diversity 比 scale 更重要"在这里有直接对应：在 SaaS-Bench 这种长 horizon 场景里，safety monitor 不能只盯 final action，必须沿 weighted checkpoint 多点同时审视——而每个 checkpoint 的 monitor 应当是 diverse 的（一个看 state 一致性、一个看 action permission、一个看 cross-app 凭证）。这把 5-12 PRISM 的"传播放大"分析自然地搬到长 horizon。

Safety Benchmark（直击主线）

这是过去三周 benchmark 方向最重要的"reality check"。它对前面所有 benchmark 提一个共同质疑：当我们把任务从合成 simulator 换到真实 SaaS，绝对数字塌缩到 1/10——这意味着已发表 leaderboard 的相对排序也可能反转。建议把 SaaS-Bench 的 weighted checkpoint 协议反向接入既有的 5-12 Agent-ValueBench、5-13 TRIVIA+、5-14 BenchJack，看看哪些是 simulator-friendly 偏置、哪些在真实环境下还成立。同时，weighted checkpoint 与 5-15 ClawForge 的 strict/partial/step-efficiency 评分构成更通用的 "fine-grained progress benchmarking"协议族——值得在毕设论文里单列一节"评测协议演化"。
