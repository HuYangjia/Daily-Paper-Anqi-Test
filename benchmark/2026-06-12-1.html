一句话核心贡献
把 LLM unlearning 评测从单模态 + 一次性 升级到 多模态 + 终身顺序请求—— 9 类共 127 个实体 的顺序遗忘请求，揭示"持续从一个模态遗忘可能降级整个模型" 的多模态独有挑战，并提出 LUMoE 显著缓解累积退化。
摘要（中文翻译，忠实原文）

多模态大语言模型（MLLMs）训练于海量多模态数据之上，使数据遗忘（data unlearning）变得日益重要——数据所有者可能要求移除特定内容。在实际操作中，这类请求往往随时间顺序到达，引发了一个具挑战性的问题：MLLM 终身遗忘（MLLM Lifelong Unlearning）。

然而，大多数现有 benchmark 在规模与范围上受限，无法捕捉 MLLM 终身遗忘的复杂性。为填补这一空白，我们引入 MLUBench——一个大规模综合 benchmark，包含9 类共 127 个实体 的终身遗忘请求。

我们使用 MLUBench 进行了广泛实验，揭示现有遗忘方法存在严重的累积退化。更关键的是，我们识别了该问题的独有挑战：不同于单模态模型，MLLM 终身遗忘受制于保持多模态对齐（preserve multimodal alignment）的需求——持续从一个模态遗忘可能降级整个模型。

为缓解这一挑战，我们提出 LUMoE，一种有效的方法。实验证明 LUMoE 显著缓解 baseline 面临的退化问题。

核心内容解读（终身遗忘 / 多模态对齐 / 累积退化 / LUMoE）

立论：unlearning 评测必须从"一次性" 升级到"终身顺序"。过去两年 LLM unlearning 研究的主流 benchmark（TOFU / WMDP / MUSE 等）几乎都是一次性遗忘评测——给定一个遗忘集 / 一次操作 / 评估保留性。但现实部署的核心挑战是：用户隐私请求是随时间陆续到达的——GDPR 第 17 条规定的"被遗忘权" 在 6 个月后可能再次被新用户触发；DMCA 投诉是持续流入的；版权方维权也是顺序到达。这意味着遗忘是流式问题而非批处理问题。MLUBench 的第一个核心贡献是把 unlearning "从批处理升级为流式"。这一升级与 5-19 ANNEAL（arXiv 2605.16309）"把 recurring failure 翻译为对 process knowledge graph 的 governed 符号补丁" 思想同源——都是把"一次性问题" 重写为"持续治理问题"。

方法学贡献一：MLLM unlearning 的独有结构性挑战是多模态对齐。论文最重要的发现是identify 了 MLLM unlearning 的本质难点——不在 遗忘本身，而在遗忘后多模态对齐是否还能保持。具体来说：

挑战单模态情形多模态情形（独有）
遗忘单一概念仅删除该概念的文本表征需同时从视觉、文本两个表征空间删除并保持对齐
顺序多次遗忘累积退化主要在文本能力上累积退化跨模态扩散——从一个模态遗忘可能降级整个 cross-modal projector
遗忘保持评测retain set 文本任务必须同时评估文本任务、视觉任务、cross-modal 任务三轴

核心引用："unlike in unimodal models, MLLM lifelong unlearning is constrained by the need to preserve multimodal alignment. Continually unlearning from one modality could degrade the entire model." 这条结构性发现把过去所有 unimodal unlearning 工作（TOFU / WMDP / MUSE）的方法学显式认定为不充分——它们没有覆盖 MLLM 场景的 cross-modal 退化路径。

方法学贡献二：9 类 / 127 实体 / 顺序请求 的工业级 benchmark 规模。对比之前的 unlearning benchmark：TOFU 200 个虚构作者 / WMDP ~3,668 题 multiple choice / MUSE 哈利波特等具体作品。MLUBench 选了9 类（class）共 127 个实体（entity） 的顺序遗忘请求——这是多模态独有 的设计：每个 entity 在 visual / textual 两个模态都有数据，遗忘时必须同时处理两个模态。9 类的覆盖（论文未在摘要点名，但根据 MLLM 隐私敏感主题可推测涵盖：人物、品牌、敏感地标、版权艺术作品、敏感内容、私人医疗、儿童识别、车牌等）让 benchmark 具备 GDPR / DMCA / 平台隐私政策的真实合规映射。

核心发现：现有方法严重累积退化。论文用 MLUBench 跑现有遗忘方法（GA / NPO / RMU / SCRUB 等），发现"severe, cumulative degradation"——意味着不只是性能下降多少，而是退化累积，每次新的遗忘请求都让模型保留能力变差。这条 finding 把 unlearning 圈子从研究单次遗忘的有效性 推到研究退化曲线的形状——从静态评测升级为动态评测。这与 6-09 MAC-Bench（arXiv 2606.07805）"Beyond Goodhart's Law + Agent-as-a-Benchmark" 在不同问题领域上一致——都是把静态 metric 升级为动态过程。

方法学贡献三：LUMoE 用 Mixture-of-Experts 架构缓解多模态对齐降级。LUMoE（Lifelong Unlearning Mixture-of-Experts）的核心思路：不让一个统一模型承担所有遗忘请求，而是用 MoE 把不同遗忘域路由到不同 expert——这样某一 expert 退化不会扩散到整个模型。这条设计与 5-19 ANNEAL"locate operator + 类型化补丁" 同源——都是用结构隔离 防止局部修改扩散为全局退化。LUMoE 与 5-26 SafeHarbor（arXiv 2605.05704）"分层记忆树自演化" 在精神上也接近——都是用结构而非训练目标来达成 plug-and-play 治理。

哲学定位：unlearning 圈子的"第二代评测"。过去两年 unlearning 研究"第一代评测" 是 TOFU / WMDP / MUSE——对应的是静态、单模态、一次性。MLUBench 立成的"第二代评测" 是动态、多模态、终身。这条范式转变会显著改变未来 unlearning 论文的写作方式：必须报告 (a) 顺序退化曲线；(b) cross-modal 保持能力；(c) 与 LUMoE 类结构隔离方法的对比。这条范式转变也直接呼应 6-10 PreAct-Bench"Prefix Foresight F1 把预测时刻立为 first-class metric" / 6-11 SciConBench"clean-room evaluation harness" 的精神——把 benchmark 设计理念从"测一个静态指标" 升级为"测一个动态过程"。

开源价值：完整的代码 + 数据集开源。GitHub lihe-maxsize/Lifelong_Unlearning_main 提供 MLUBench 数据集与 LUMoE 实现。任何在生产中部署 MLLM（GPT-4V / Claude 3.5 Vision / Gemini Vision Pro）的团队都可以接入做合规性 benchmark。这条工程价值在 GDPR Article 17 / EU AI Act 落地强制实施的 2026 H2 是具体可操作的合规工具。

对你三个研究方向的启发
Harness Engineering（直击毕设）

MLUBench 给你的 KV-cache 毕设的最直接启发是：把 cache 系统的"顺序请求 + 累积退化" 立为评测维度。当前 KV-cache 论文几乎都用静态 trace 评测——给定一个 trace、跑一次、报告 hit rate / latency。MLUBench 的"severe, cumulative degradation" 思想提示一个真实但被忽视的场景：cache 在长程服务的退化曲线——cache 经过几小时 / 几天的运行后是否会出现累积性能下降（碎片化、淘汰策略偏移、热点变化）。毕设建议：在实验部分显式报告 (a) 短程 hit rate；(b) 长程（几小时 / 几天）退化曲线；(c) 不同顺序到达的 workload 模式下的退化对比。这条动态评测 姿态让你的工作具备"系统在真实负载下的可持续性" 视野。同时 LUMoE 的"结构隔离" 思想直接给你的 KV-cache 设计提供借鉴：用分区 / 分块 / hierarchical 结构隔离不同 workload，避免一个 workload 的退化扩散到整个 cache。

Agent Skills Safety

MLUBench 与今日方向 #2 Containment Gap 形成互补—— Containment Gap 揭示memory poisoning 单次写入 → 88.9% 持久污染；MLUBench 揭示合法 unlearning 顺序请求 → 累积退化。把两者合起来：agent 持久记忆面临两类压力——(a) 恶意污染（Containment Gap）；(b) 合法顺序遗忘（MLUBench）。两者都要求结构隔离 + 完整性验证 作为基础设施。这条结合也呼应 5-19 Memory Laundering"SPG" 与 5-25 SafeHarbor"分层记忆树自演化" 的精神——agent 持久记忆必须是可隔离、可验证、可遗忘 三位一体的。具体建议：未来 agent safety 论文应同时报告 (a) memory integrity（抗攻击）；(b) memory unlearning（合规遗忘）；(c) cross-skill alignment preservation（防止结构性退化）。

Safety Benchmark

MLUBench 自身就定义了下一代 benchmark 写作模板——把静态评测 升级为动态过程评测。今日方向 #1 HarnessBridge 把 harness 立为可学习模块，今日方向 #2 Containment Gap 把架构层立为审计对象——这三者合起来构成 2026 H1 后期的三轴姿态变化：(a) harness 可学习化（HarnessBridge）/ (b) framework 可审计化（Containment Gap）/ (c) benchmark 可持续化（MLUBench）。三者都把"静态" 元素动态化。建议：你下一个研究 benchmark 时必须显式声明：(a) 评测是一次性还是顺序的；(b) 是否覆盖累积退化 维度；(c) 多模态系统是否同时评估 cross-modal 保持。这三条姿态直接接续 6-03 What Benchmarks Don't Measure / 6-09 MAC-Bench / 6-10 PreAct-Bench / 6-11 SciConBench 的方法学路线，构成 2026 H2 evaluation methodology 的统一姿态。
