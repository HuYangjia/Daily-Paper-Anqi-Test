摘要（中文翻译，忠实原文）

交互式 agent benchmark 在可扩构造与真实工作流评测之间一直存在张力——人手编写的任务难以扩展和修订，静态 prompt 评测又错过那些只有在 agent 操作"持久状态"时才会暴露的失败。已有交互式 benchmark 大幅推进了 agent 评测，但大多从干净状态初始化任务，并不系统检验 agent 在面对预先存在的部分、陈旧、或冲突 artifact 时如何应对。

我们提出 ClawForge——一个generator-backed 的可执行命令行工作流 benchmark 框架。框架把场景模板、grounded slots、初始化状态、参考轨迹与 validator 编译为可复现的 task 规约，并在持久工作流面上逐步评测 agent，使用归一化末态与可观测副作用而非"完全 trajectory 匹配"作为评分。框架被实例化为 ClawForge-Bench（17 场景，6 能力维度）。

跨 7 个 frontier 模型的结果显示：最强模型仅达 45.3% 严格正确率；所有模型的"错误状态替换"率均低于 17%；模型分布的最大跨度（17% → 90%）由"是否在动手前先检视已有状态"决定。partial-credit 与 step-efficiency 分析进一步揭示：许多失败是"近终点 closure 失败"而非早期崩溃，且不同模型在 state-conflict 下表现出定性不同的失败风格。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去两周 benchmark 方向一直围绕"新评测面"扩展——5-08 XL-SafetyBench 加多语种、5-09 Partial Evidence Bench 加权限完整性、5-12 Agent-ValueBench 加价值系统、5-13 TRIVIA+ 加 desiderata、5-14 BenchJack 测 benchmark 自身。本文则补上了一个长期被默认忽略的维度：真实部署里 agent 拿到的从来不是干净状态——它面对的是上一次执行留下的部分文件、陈旧 cache、冲突配置。ClawForge 是第一个把"state conflict"做成 first-class 评测变量的 benchmark。这一步的重要性等同于把"动作链"接到了"持续部署语境"上，与今日 Harness 选定的 ChromaFlow（强调操作 telemetry）、Agent Safety 选定的 知-行 gap（强调内部表征）共同构成"从内部到外部都不再用干净状态评测"的方法学跃迁。

核心方法（generator-backed + executable + state-conflict-aware）：

  • Generator-backed task spec：场景模板 + grounded slots（具体值）+ 初始化状态 + 参考 trajectory + validator 一同编译——每个具体任务都是从程序生成的、可重新参数化的。这是过去 benchmark"手工编/众包/翻译"路径的根本性升级。
  • 持久工作流面：agent 不在每个 task 中"重生"——它面对的是上一步留下的真实文件系统/进程态/环境变量。这与 OSWorld、TerminalBench 不同：那些 benchmark 也用 sandbox，但每 task 通常是冷启动；本文是跨任务持续状态。
  • 归一化末态 + 可观测副作用评分：不强求 agent 走和 reference trajectory 一模一样的路径——只检查末态等价 + 关键副作用是否被触发。这是与 ChromaFlow telemetry 思想呼应的"过程行为可见"评分。
  • Strict / Partial-credit / Step-efficiency 三档评分：strict 看完美正确；partial 拆功能子模块；step-efficiency 算 token/动作开销。这让"近终点崩坏"不被一票否决——这是 benchmark 评分体系的细分突破。
  • 17 场景 × 6 能力维度：覆盖文件、进程、网络、git、包管理、CLI 工具——是命令行 agent 的"核心 capability surface"。

核心数据亮点 / 关键论点
  • 最强模型 strict 准确率 45.3%：在仅 17 场景 / 6 能力维度的 benchmark 上，frontier 模型也只能完美完成不到一半 的任务。这是当前 frontier agent 仍未触及"持久状态"层面的硬证据。
  • 模型分布 17% → 90%（拉开 73 pp）：单一 benchmark 内 frontier 模型间分布如此之宽，说明这个评测面区分能力强——而它的判别量是"是否在动手前先检视已有状态"，比所谓"原始能力"更具机制层意义。
  • 错误状态替换率 < 17%（所有模型）：这是一个有意思的负面发现——模型很少把对的状态变错（积极避免破坏），更多是没改对（保守不前）。这意味着 alignment 的核心问题不是"激进破坏"而是"保守失能"。
  • 近终点 closure 失败 > 早期崩溃：partial-credit 分析显示多数失败是"差最后一步没收尾"。这与一直以来"agent 早期 plan 错"的直觉相反——更像是近终点的 verifier/commit 缺失。
  • 失败"风格"差异质化：不同模型在 state-conflict 下展现出定性不同的失败画像（有的"重写覆盖"、有的"反复探查"、有的"提前放弃"）。这意味着 agent benchmark 应配套失败模式 taxonomy 而不只是 pass-rate。
  • generator-backed 可扩：场景从 17 个模板能扩到任意数量，每个 slot 可用不同 grounded value 实例化——避免 benchmark 一次性枯竭。这是 BenchJack（昨日）希望但未做的另一面。
对你三个研究方向的启发
Harness Engineering

ClawForge 给了 harness 一个长期忽视的输入维度：初始 state 应作为 harness 配置参数，而非默认"clean"。这与你 KV-cache offload 实验的同型问题——你测 prefetch 命中率时，是否每次都从空 cache 开始？真实生产里 cache pool 永远带"前一步残留"，分布与冷启动差很大。建议毕设论文 v3 加一组实验：warm cache（带前一 batch 残留）vs cold cache 下的 hit-rate / latency 对比——这就是 ClawForge 思想在系统层的翻译。它和今日 Harness 选定的 ChromaFlow（telemetry-driven） 共同支持你写一节"我们不再假设 clean baseline"的实验设计。

Agent Skills Safety

state-conflict 与今日 Agent Safety 选定的 知-行 gap 拼起来格外锋利：模型在面对持久状态时的失败多数是"知道该看 state 但没去看"——这正是知-行 gap 的"动作"端。可以在 ClawForge 上跑一套 hidden-state probe：看模型有没有"该看 state"的 cognition 方向，再看它是否在 last-token 上路由进 action。如果"看 state"的 cognition 强但 action 弱，就是论文 5-15 在 tool-use 之外的第二个验证场。这正是把 agent safety 从"防攻击"扩到"防自身懒惰"的方法学桥梁。

Safety Benchmark（直击主线）

给 benchmark 社区一个明确的设计公式：generator + 持久状态 + 三档评分。每个新 benchmark 都应回答："是否能用模板生成？是否考虑前次残留？是否区分 strict / partial / step-efficiency？" 把这三条作为 next-gen agent benchmark 的最小集。它与昨日的 BenchJack（揭露 reward-hack）、5-13 TRIVIA+（属性 desiderata）、5-12 Agent-ValueBench（价值系统）合在一起，形成 2026-Q2 的benchmark 工程方法学栈——这值得你在 v3 实验章作为"评测协议来源"统一引用。
