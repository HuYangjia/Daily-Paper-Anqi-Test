摘要（中文翻译，忠实原文）

LLM agent 越来越多地访问私有用户数据，并代表用户与第三方系统交互。用户定义"什么可以共享、什么绝不能共享"，agent 必须在第三方系统行为对抗时也能稳健地遵循这一意图。我们提出 POLAR-Bench（Policy-aware adversarial Benchmark）：一个 trusted model（带 privacy policy + 一个任务）与一个第三方 model（对 task-relevant 与 protected attribute 双向探测）对话的对抗式基准。

跨 10 个 domain × 7,852 个 sample，作者用确定性 set-membership 同时打分隐私与效用，沿两条正交轴变化 privacy policy dimension 与 attack strategy，从而生成每个模型一张 5×5 诊断面。

结果呈现一条尖锐分界：当前前沿模型守住 > 99% 的 protected attribute；而1–30B 范围的小开源模型——也是用户最常以 trusted agent 形式 on-device 或私有推理跑的那一档——表现明显更差，最弱者泄漏过半。POLAR-Bench 因此能定位每个模型 intent-following 在哪种 policy/攻击组合下崩溃，给"最关键之处"的隐私对齐提供着力点。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去三周 benchmark 方向一直在"扩攻击面/扩任务面"——5-13 TRIVIA+ desiderata（报告）、5-14 BenchJack 的 reward-hack（219 漏洞）、5-15 ClawForge state conflict、5-18 SaaS-Bench 真业务系统、5-19 TOBench 100 任务 27 MCP。这些工作共同把 benchmark 推向"任务复杂度" 的极限。POLAR-Bench 正交补上一档：当 agent 已经能完成复杂任务后，"它会不会替你说错话"才是真正的下一道关——而且这道关恰好集中在用户最依赖的小模型上。这是一份"反 leaderboard 误导"的 benchmark：前沿模型在它上拿到不痛不痒的 99%+，但用户日常跑的 8B/13B 直接腰斩。

核心方法：trusted/3rd-party 对话 + 5×5 诊断面 + 确定性 set-membership 打分

  • 双角色对抗框架：trusted model 持 privacy policy 和任务，第三方 model 同时探测任务相关属性（要给）和受保护属性（不能给）——这把"信息泄漏"从"不要说脏话" 推到"说与不说哪些受 policy 约束的事实"。
  • 10 domain × 7,852 sample：覆盖足够广的真实场景——医疗、金融、家庭、企业等——每个 sample 带 deterministic 答案标签，避免 LLM-judge 的不稳定。
  • 5×5 diagnostic surface：5 类 privacy policy dimension × 5 类 attack strategy，每个模型得到一张 25 格的诊断面——不是单一 ASR 数，而是哪个组合崩溃的定位图。
  • Deterministic set-membership scoring：用集合包含来判定泄漏——比 LLM-as-Judge 稳定得多，也呼应了 5-15 ClawForge 与 5-18 SaaS-Bench 的"避免 judge 噪声"路线。
  • frontier vs 1–30B 双档对比：把"哪类模型在哪种攻击下漏洞最多"画到具体格子里，给"用户跑在自己设备的小模型"画了一张精度足够的风险地图。

核心数据亮点 / 关键论点
  • 10 domain × 7,852 sample × 5×5 诊断面：评测面规模与诊断颗粒度同时给到——这是 benchmark 设计的高水位线，单一 ASR 数已经不够。
  • 前沿模型 protected attribute 守住 >99%：意味着"用 GPT/Claude 等做 trusted agent"在 POLAR-Bench 上几乎不漏——但代价是 cloud inference 的可用性与隐私本身的悖论。
  • 1–30B 开源模型最弱漏 > 50%：恰好是用户为追求"on-device / private-inference"而选择信任的那一档——该论文等于把"小模型自我托管 = 隐私"这一假设打回原形。
  • Deterministic set-membership scoring：避免 LLM-judge 噪声——这条与 5-15 ClawForge / 5-18 SaaS-Bench / 5-19 TOBench 的"per-task grounded evaluator" 同向，是 benchmark 圈正在收敛的评分方法学。
  • 5×5 诊断面：每个模型一张二维表，告诉你它在哪种 policy + 哪种攻击下崩溃——等于把 leaderboard 升级为一份 model card 风格的"风险定位图"。
  • 双正交轴：policy dimension 与 attack strategy 显式正交——避免把 policy 复杂度的 confound 算到攻击鲁棒性上去，呼应 5-13 TRIVIA+ 的 desiderata 思路。
对你三个研究方向的启发
Harness Engineering

POLAR-Bench 的 5×5 诊断面是给所有 harness 评测的方法学样板："单一 ASR/任务通过率"已不足以诊断系统，必须用"两条正交轴 × 多格"展开。把这套思路嫁接到你的 KV-cache offload 仿真：横轴是负载特性（短上下文 / 长上下文 / 多 batch / 大 prefetch_k），纵轴是策略（LRU / FIFO / Random / Watermark / HU），把 throughput 改成"相对 ideal 的延迟增加百分比"——立刻得到一张策略×负载的 5×5 诊断面。这比单条曲线更适合毕设展示，也方便审稿人看到"哪种组合崩溃"。

Agent Skills Safety

POLAR-Bench 把"小开源模型才是真战场"这一事实推到正面——这跟 agent skill safety 的供应链威胁是同一种"民主化失守"图景：5-13 SKILL.md 86%/77.6%/100% 攻击（报告）的对象正是 ClawHub 上自由发行的第三方 skill；今日 ECA（报告）的 typed certificate 也最适合无法信任的 self-hosted skill 生态。把 POLAR-Bench 的 5×5 诊断面套到 SKILL.md 安全：横轴是 policy 维度（"哪些 capability 被允许 / 哪些必须 sandbox"），纵轴是攻击策略——会立刻给 self-hosted agent 的 skill 安全画出第一张诊断学风险地图。

Safety Benchmark（直接对位）

对 benchmark 方向的启示最直接：把"诊断面"设为下一代 safety benchmark 的 first-order 输出格式。所有 leaderboard——HarmBench / TrustLLM / XL-SafetyBench / SaaS-Bench / TOBench / DTap / BenchJack——都该放弃 "一个数"，改报每个模型一张诊断面（policy/任务维度 × 攻击/复杂度维度）。再叠加：deterministic set-membership scoring（避免 judge 漂移）+ 1–30B 开源模型必报（避免只看 frontier 的乐观偏差）+ Wilson 上界（呼应 ECA 的统计严谨度），就是"benchmark 2.0"的最小约束。
