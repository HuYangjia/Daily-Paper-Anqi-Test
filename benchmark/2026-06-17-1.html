摘要（中文翻译，忠实原文）

随着 LLM-based shopping agent 进入生产，现有 benchmark 无法捕获购物者需求到达方式的复杂性：有些隐含地陈述于查询、有些记录在画像中、还有些仅在被对的问题问到时才显露。把全部意图前置呈现、仅按最终选择评分的 benchmark 既无法构造这种长程挑战，也无法解释 agent 漏掉了哪条需求。

为弥合这一空缺，我们提出 EComAgentBench，一个基于真实 Amazon 商品与评论的 662 任务基准。每条任务把这些需求分散到 visible query、tool-gated profile、scripted clarification 三处；agent 必须揭示隐藏意图、对照属性与评论证据验证候选项、并在 100 次工具调用内承诺一个商品。此外，类型化、来源标记的 rubric 对每条任务评分，把每个失败归因到具体需求与其来源。构建过程自动但可靠——所有答案在任何文本生成前已固定在代码中，每条样本经过验证。我们对 7 个模型的评估显示：最强模型整体精度仅 57.1%，rubric 满足率从可见来源到隐藏来源退化。我们相信 EComAgentBench 将成为把 shopping agent 从单查询搜索推向长程可靠协助的可复现基础。

核心内容解读（三源分布意图 / 类型化来源 rubric / 七模型对比 / 单调退化模式）

立论：shopping agent 评测必须区分"检索意图" vs "使用意图"。过去半年（2026 H1）的 agent benchmark 主线把"评测面下沉" 推到深处：5-25 #3 Boiling the Frog 把多轮持久状态立为 first-class；6-09 #3 SciAgentArena 把跨 agent 共享失败模式立为诊断；6-15 #3 WorkBench Revisited 用纵向 21 模型重测证明 capability 与 safety 同向提升。EComAgentBench 是这条线的意图维度新切面——它问的是：当一个真实购物者的需求不全在 query 里，agent 是否还能可靠服务？这条问题在 WebShop / Mind2Web / WebArena / AssistantBench 中被规避（它们把全部意图前置呈现）。EComAgentBench 的关键设计是分散意图到三个来源：

来源访问方式隐藏程度Rubric 数量占比
Visible Query直接可读部分需求隐含陈述60.1%（3,993 条）
Tool-Gated Profile需调 get_user_profile 工具隐藏在用户档案 product_requirements 字段19.9%（1,326 条）
Scripted Clarification需调 ask_user + 触发关键词仅在 agent 提对问题时显露19.9%（1,326 条）

系统设计一：persona 与 demographics 分离防止信息泄漏。persona 约束作为陈述偏好存储在专用 product_requirements 字段；其他不计分上下文（demographics / lifestyle / shopping_habits）独立存放——保证 persona 必须被查询而非推理。这条设计与 6-15 #2 MINIM 把 sensitive 信息分桶的 CI 思想同构：MINIM 用敏感度分数控制哪些信息被发往云端 LLM；EComAgentBench 用 profile 字段隔离控制哪些信息能被打分使用。两者都把"访问模式"立为评测对象。

系统设计二：脚本化 clarification 防止用户随机性归因混淆。每个 clarification slot 包含 slot_id / linked_rubric_ids / hidden_info / trigger_keywords / user_response / default_response，最多 10 轮澄清。Clarification 是脚本化而非 LLM 模拟——遗漏 clarification 的责任可明确归因到 agent 而非 user simulator 的随机性。这条设计是 6-08 #3 AARRI 二轴评测之后又一次"把 user simulator 失真度从评测中剔除" 的方法学进步。

系统设计三：6 类 × 3 源类型化来源标记 rubric。每条 rubric 同时携带 type（attribute_match / numeric_range / entity_match / negative_attribute / budget_match / review_opinion）与 source（query / persona / clarification）。失败时同时定位"哪条需求未达成 + 该需求来自何处"——这是 WebShop / Mind2Web / WebArena 等先前购物 benchmark 无法做到的。论文构建管道代码先行：rubric 的 id / type / field / expected_value / source 在任何文本生成前已锁定，LLM 仅控制表面表达，无法改变正确性标准。共 6,645 rubric / 平均 10.04 rubric/任务 / 8 类 intent / 10 工具。

Rubric 类型检查内容典型示例
attribute_match精确属性值Color = Black
numeric_range数值范围average_rating ≥ 3.5
entity_match标题中的实体短语标题含 "Wireless Charger"
negative_attribute排除属性值非 waterproof
budget_match优惠后总价 ≤ 预算后券价格匹配
review_opinion基于评论的观点"standing up or lying down"

有趣的策略分化：Opus-4.6 用最多工具调用（33.9）但完成率最低（84.6%）——"深搜索易耗尽预算"；GPT 系列工具调用少（15.5–25.1）+ 完成率高——"早问早答"；Qwen3 完成率 100% 但精度仅 19.5%——"快速完成但准确性差"。这条策略空间区分能力是简单准确率指标 无法提供的。

失败率从 query 来源（12–31%）→ persona（27–57%）→ clarification（29–55%）大致翻倍。论文在 §6.3 给出关键诊断："几乎每个模型都读取了 profile，因此差距在于使用隐藏信息而非检索它"——这条结论把 agent 评测从"测能力检索" 推向"测能力使用"，是 2026 H2 后期 benchmark 设计的重要新维度。

核心实证三：隐式需求驱动难度（非 rubric 数量）。Table 7：隐式需求 0–1 个 → 平均精度 51.8%；3+ 个 → 27.2%（几乎减半）。原始 rubric 数量增加并不带来同等难度提升。这把"任务难度" 重新定义为意图分布的隐式程度而非rubric 总数。review_opinion 是最难子技能（满足率 33.7–67.4%）——从非结构化评论中提取与聚合证据是当前 agent 普遍弱点。

方法学姿态：意图维度 benchmark = WorkBench Revisited 的横向对偶。EComAgentBench 与 6-15 #3 WorkBench Revisited 形成"纵向 + 横向" 双姿态：WorkBench Revisited 是同一基准跨 21 模型纵向重测；EComAgentBench 是单一时刻跨 7 模型 × 3 信息源横向评测。两者都把"agent 进步" 重新定义——WorkBench 证明 capability 与 safety 同向提升；EComAgentBench 证明能力差距集中在"使用 hidden 信息" 而非 "检索"。2026 H1 后期 benchmark 设计姿态变化（与 6-12 MLUBench"动态化" / 6-09 SciConBench"clean-room 评测" / 6-08 AARRI"二轴评测" 一脉相承）：(1) 评测对象从能力 → 使用；(2) 评测面从单时刻 → 纵向 + 横向；(3) 评分粒度从整体准确率 → 类型化 + 来源标记 rubric。EComAgentBench 是这条姿态变化的代表作。

对你三个研究方向的启发
Harness Engineering（直击毕设）

EComAgentBench 给你的 KV-cache 系统层毕设最直接的启发是：把"工作负载需求" 也立为分布式来源。当前 KV-cache 论文常见的失败姿态是"给一个完整 trace、看 hit-rate"——但真实部署中 cache 系统的"需求" 同样分散在多个来源（应用层 hint / 系统层指标 / 用户层 SLA / 历史 access pattern）。毕设方法论建议：(1) 借鉴三源分布——为你的 cache 评测设计 "显式 trace（user query 类比）/ 持久 hint（profile 类比）/ 运行时反馈（clarification 类比）" 三档需求来源；(2) 借鉴类型化 + 来源标记 rubric——cache 系统的失败也应被分类为"哪类失败（cold miss / capacity miss / coherence miss）" + "因何来源信息没用上（应用 hint / 历史 / 实时反馈）"；(3) 借鉴"使用 vs 检索" 区分——很多 cache 论文报告的"读到了 hint 但没用上" 等同于 EComAgentBench 的 persona 退化模式，应当被独立测量；(4) 借鉴 EComAgentBench 强调"意图被分散到多源" 是评测设计选择，把这条姿态写进你的"cache evaluation methodology" 章节。论文写作姿态：把 EComAgentBench 列为"分布式来源 + 类型化 rubric" 评测范式的方法学先例——你的 cache 系统改进必须在多来源 + 多类型失败诊断 维度都站得住。

Agent Skills Safety

EComAgentBench 与今日方向 #2 Rift"欺骗 vs 错误对照范式" 在评测哲学上同构——两者都把"看似单一的失败"分解为多个独立可归因维度。safety 方向的关键启发：(1) 未来 agent skills safety 评测必须借鉴三源分布——一条 jailbreak 任务的 trigger 可拆为"visible query trigger / persona trigger / clarification trigger" 三源，分别测量 agent 的拒绝鲁棒性；(2) 借鉴来源退化模式——很多模型在 visible jailbreak 下拒绝率 90%+，但在 persona-embedded jailbreak（"用户档案显示是医生"）或 clarification-embedded jailbreak（"反复追问让用户确认有需求"）上拒绝率会显著退化；这条退化曲线应被立为safety benchmark 标准维度；(3) EComAgentBench 的persona 与 demographics 字段分离 设计可直接套用 agent safety——把"有意泄露的 sensitive 信息" 与"背景信息" 分开存储防止评测污染；(4) 与 6-15 #2 MINIM 形成对偶——MINIM 在客户端控制 agent 不读不必要 sensitive 信息，EComAgentBench 在 benchmark 中考核 agent 是否会从 profile 中误用 sensitive 信息——两者是 agent privacy safety 的"防御 + 评测" 双轴。

Safety Benchmark

EComAgentBench 与今日方向 #1 SEAGym"5 视图评测" 形成"纵向 + 横向" 双姿态——SEAGym 是纵向（同一基准跨 epoch / snapshot 评测演化过程），EComAgentBench 是横向（同一时刻跨 7 模型 × 3 信息源评测隐藏意图恢复）。统一启发：未来 safety benchmark 论文应同时具备 (a) 纵向视图（snapshot 时序 / replay 退化）；(b) 横向视图（多模型 / 多 backend / 多基准）；(c) 来源视图（信息可见 / 信息隐藏）。EComAgentBench 揭示的"差距在使用而非检索" 现象在 safety 上对应"差距在执行 alignment 决策而非记住 alignment 训练"——这条诊断维度应被纳入 safety benchmark 标准。EComAgentBench 把类型化 + 来源标记 rubric 立为失败可归因 的工程基础——这条范式与 6-09 #3 SciConBench"原子事实分解" 一起，把 benchmark 的失败诊断粒度从"整体 ASR / 准确率" 推向"哪条具体需求 + 哪个来源 + 哪类失败" 三维归因。建议：未来 safety benchmark（如 HarmBench / TrustLLM 续集）也应采纳 6 类 × 3 源类型化 rubric——让"这个 agent 拒绝率 60%" 升级为"在 persona-leaked 来源下拒绝 review_opinion 类需求 X 个百分点"。
