摘要（中文翻译，忠实原文）

Benchmark 污染——评测样本出现在模型训练数据中——威胁着 LLM 评估的有效性。用于检测训练数据成员关系（training-data membership）的统计工具是存在的，但它们几乎全部只在受控的学术环境中得到了验证：大型同构预训练语料 + 透明的单阶段训练管道。在更现实的审计场景下，这些方法是否仍可靠，目前并不清楚。

我们识别出两种未被充分研究的失效模式：分布偏移（distribution shift）——可疑集合与验证集合违反 IID 假设；规模约束（scale constraints）——benchmark 比预训练语料小数个数量级。我们系统评估了三种主流范式：LLM Dataset Inference、Post-Hoc Dataset Inference、CoDeC，覆盖 27 个来自不同家族（含 Pythia、OLMo 2、专门文化与医学 LLM）与不同规模（最大 27B）的模型；并把分析延伸到前沿工业模型。

在 335 次评估中，只有 199 次得到了正确结果。LLM Dataset Inference 在分布偏移下产生 false positive；Post-Hoc Dataset Inference 在 benchmark 规模下能力不足；CoDeC 仅给出粗粒度的 provenance 信号，不足以验证单个 benchmark 切片。我们的结果揭示了"受控验证"与"实战 benchmark 审计"之间一个系统性的可靠性差距，并表明统计检测目前无法取代透明的数据来源追踪（transparent data provenance）。我们开源了基准供后续研究。

核心内容解读（污染检测三范式 / 两大失效模式 / 实证结果 / provenance 立论）

问题背景：benchmark 信任危机的元危机。2024-2026 期间，LLM benchmark 出现了"泄漏猜想"——GSM8K / MMLU / SWE-bench 等热门 benchmark 据信至少部分被前沿模型见过。这刺激出三套统计检测武器：(a) LLM Dataset Inference（基于成员推断攻击）、(b) Post-Hoc Dataset Inference（基于困惑度差与统计假设检验）、(c) CoDeC（基于压缩率与编码差异）。它们在 Pythia / GPT-2 这样的受控研究模型上都给出过亮眼报告。但 Zarzecki 等指出：这些工具未在审计真实场景下被验证——也就是说"检测器自己有没有问题"是一个被忽视的元问题。

实证规模与失效证据。论文跑了 335 次评估，覆盖 27 个模型（Pythia 全系列 / OLMo 2 / 专门 cultural & medical LLM）从小到 27B。正确结果只占 199 次（59.4%）——这意味着如果你随便选一个污染检测方法去审计一个新 benchmark，约 4 成的概率得到错误结论。这是定量地把"受控 vs 实战的可靠性差距"立出来——之前的工作只在受控环境验证检测器，缺少这种跨家族、跨规模、跨场景的系统性扫描。

结论的力度：从工具到立场。论文不止指出工具问题，还给出明确立场——统计检测无法取代透明 data provenance。换言之：审计 benchmark 污染的根本路径，不是"训练后逆向猜测"，而是"训练前公开来源"。这与去年 EleutherAI / OLMo 阵营推动的"open data + open weights"路线在精神上同构——但 Zarzecki 等用检测器在实战下失效把这个立场定量化了。

Harness Engineering

本文给"评测的评测"提供了直接范式——你毕设的 KV-Cache 预取 benchmark（GAIA / SWE-bench / 长上下文 LongBench-v2）应当回答："命中率提升真的是预取贡献的，还是 benchmark prompt 中存在已被记忆的前缀？"建议：在毕设实验章节加一个provenance audit 小节，明确公开 prompt 来源 / 训练数据切分 / 评测 split 的可见性——这样能对今日方向 #2 abstention 与本文 contamination 的双重质疑提前设防。同时今日方向 #1 SkillDAG 的 propose-then-commit 协议给"评测断言要有 evidence-back"提供了直接 IR——你毕设的每个性能数据点都可以被解释为"commit by execution-backed evidence"。

Agent Skills Safety

本文揭示的失效模式分布偏移 + 规模约束，与 5-25 #2 TTT Undermines Safety 揭示的攻击面测试时分布迁移是同一物理现象——TTT 攻击之所以能 jailbreak，正是因为 safety alignment 在受控分布上"看起来稳定"，但在分布偏移下崩坏。这暗示safety guardrail 的"实战可靠性"也需要类似 335 次系统扫描——把 SafeHarbor / TTT / Memory Laundering 放在跨家族跨规模 audit harness 下复检，可能会暴露同样的 "受控验证 vs 实战" 差距。

Safety Benchmark（直击）

这是 2026 上半年 benchmark 方向最重要的"元方法论"工作之一。它与今日方向 #2 abstention competence 形成镜像——前者揭示 benchmark 未测的维度，后者揭示 benchmark 测错的维度。两者合起来构成"测什么不全 + 测的不准"双重危机。建议：未来 SafetyBench / HarmBench / TrustLLM 系列升级时，应同时引入：(a) 三-gap × Safety/Usability/Informed Refusal 的 abstention 矩阵，(b) data provenance manifest 替代 statistical contamination detection。这能把 leaderboard 从"谁分高"升级为"分高在 IID 假设下站得住吗"。同时本文开源的 audit benchmark 是任何 benchmark 设计者都应当提前跑一遍的"体检表"。
