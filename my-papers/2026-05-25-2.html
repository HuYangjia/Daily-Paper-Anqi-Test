[Caroline Trippel · Formal Arch/Chip] 19 篇论文逐篇深度分析（学习提纲 + 方法要点 + 复现实操）

这篇是对 #1 的补全版本：每篇都按“问题-方法-结论-学习重点-复现建议”展开，便于系统学习。

覆盖 19/19 strict arch/chip formal 逐篇学习导向
学习顺序建议
  • 基础线：ArMOR → TriCheck → ILA-MCM → TransForm（先打牢 memory-model 形式化）
  • 安全线：CheckMate → MeltdownPrime/SpectrePrime → Contracts → RTL2MµPATH
  • 工程线：RTL-to-model synthesis、QED、NL-to-SVA（关注“如何让 formal 真正跑起来”）
逐篇学习卡片（19 篇）

1) ArMOR (ISCA 2015)

检索入口
  • 问题：异构系统中不同 MCM 语义不一致，导致跨设备行为不可预测。
  • 方法：形式化描述 + 自动化推理/转换，显式建模跨模型语义桥接。
  • 学习重点：理解“语义对齐”比“单点优化”更关键。
  • 复现建议：先复做最小 litmus 集，观察不同 MCM 下可观测差异。

2) Counterexamples and Proof Loophole (arXiv 2016)

arXiv
  • 问题：编译映射证明可能存在“看似正确但不可机证”的漏洞。
  • 方法：构造可执行反例击穿 proof loophole。
  • 学习重点：“证明文本”与“模型可证”是两回事。
  • 复现建议：对同一映射实现反例驱动检查。

3) TriCheck (ASPLOS 2017)

arXiv
  • 问题：语言、编译器、ISA、硬件的 MCM 可能分层正确但整体错误。
  • 方法：全栈三岔口验证，统一比较允许行为集合。
  • 学习重点：形式化对象从“单层”升级到“端到端契约”。
  • 复现建议：先跑经典 RISC-V/C11 litmus 用例。

4) CheckMate (MICRO 2018)

PDF
  • 问题：硬件安全验证依赖手工构造攻击，覆盖不系统。
  • 方法：利用 µhb/µspec 形式化攻击模式，自动合成 exploit 和安全 litmus。
  • 学习重点：把“攻击经验”转成“可判定约束”。
  • 复现建议：先复现论文内两类攻击模式再扩展自定义 pattern。

5) ILA-MCM (FMCAD 2018)

检索入口
  • 问题：异构 SoC 的组件级正确性难以合成为系统级正确性。
  • 方法：ILA + axiomatic MCM 联合建模。
  • 学习重点：组合验证（compositional verification）的价值。
  • 复现建议：选双核/异构最小 SoC 场景先打通。

6) MeltdownPrime/SpectrePrime (arXiv 2018)

arXiv
  • 问题：一致性协议中的失效传播可形成新的旁路攻击面。
  • 方法：自动合成攻击程序验证可利用性。
  • 学习重点：coherence protocol 也是安全验证核心层。
  • 复现建议：对比 classic Spectre 与 Prime 变体的触发条件。

7) TransForm (ISCA 2020)

检索入口
  • 问题：传统 litmus 对新型 transistency 行为诊断力不足。
  • 方法：形式化 transistency 词汇并自动生成增强测试。
  • 学习重点：测试生成与模型规格应协同演化。
  • 复现建议：基于同一模型比较原始/增强 litmus 检出率。

8) Synthesizing Formal Models from RTL (MICRO 2021)

PDF
  • 问题：手写 formal model 成本高、易偏离 RTL 实现。
  • 方法：从 RTL 自动抽取可用于 Check 系列的形式模型。
  • 学习重点：formal adoption 的瓶颈在“建模工程化”。
  • 复现建议：先做小流水线核，验证模型抽取正确性。

9) Relational Models of Microarchitectures (arXiv 2021)

arXiv
  • 问题：微架构安全属性难以统一表达与比较。
  • 方法：关系模型化（relational modeling）以统一描述执行差异。
  • 学习重点：关系视角可同时表达正确性与泄漏行为。
  • 复现建议：从两种攻击情形构建 relation check baseline。

10) Axiomatic HW-SW Contracts for Security (ISCA 2022)

检索入口
  • 问题：软件防御常依赖隐含硬件假设，难验证。
  • 方法：把假设写成可验证契约并进行形式化检查。
  • 学习重点：从“文档假设”转向“可机证约束”。
  • 复现建议：把一个常见防御策略拆成 contract clauses。

11) Scalable Assurance via Verifiable Contracts (OSCAR 2022)

CV 来源
  • 问题：契约验证如何规模化落到真实项目。
  • 方法：在 workshop 中给出扩展实践路径。
  • 学习重点：关注验证流水线与团队协作接口。
  • 复现建议：给自己的项目先定义 3-5 条最关键 contract。

12) G-QED (DAC 2023)

检索入口
  • 问题：传统 QED 在复杂加速器交互下覆盖有限。
  • 方法：Generalized QED + 分解策略提升预硅验证能力。
  • 学习重点：QED 思想可与 formal flow 深度耦合。
  • 复现建议：从 non-interfering 到 interfering 场景逐步扩展。

13) NL to SVA for Verification Productivity (DAV 2023)

CV 来源
  • 问题：属性编写是 formal 工程中高门槛环节。
  • 方法：结合电路上下文把自然语言转为 SVA。
  • 学习重点：大模型可作为“spec 生成助手”而非替代验证器。
  • 复现建议：先挑稳定模块，做 NL→SVA→model check 闭环。

14) Design for Hardware MCM Verification (PLARCH 2023)

CV 来源
  • 问题：设计阶段缺少面向验证的 memory-model 指导。
  • 方法：提出 design-for-verifiability 的实践方向。
  • 学习重点：验证前置比事后修 bug 成本低得多。
  • 复现建议：在架构评审中加入 MCM 可验证性检查项。

15) RTL2MµPATH (MICRO 2024)

Lab Publications
  • 问题：RTL 级侧信道路径复杂，手工穷举困难。
  • 方法：Multi-µPATH synthesis 自动揭示可利用路径。
  • 学习重点：路径合成是把 formal 与安全挖掘结合的关键桥梁。
  • 复现建议：对单核实验设计先做 path ranking 与验证。

16) MCM-Aware Cache Coherence (FMCAD 2024)

Lab Publications
  • 问题：cache coherence 与 MCM 分离设计导致全局语义漏洞。
  • 方法：使 coherence 机制显式受 MCM 约束。
  • 学习重点：协议正确性要在语义层闭环验证。
  • 复现建议：先构造最小反例，验证模型对违规行为的捕获能力。

17) Automatically Uncovering Hardware Side-Channels ... (OSCAR 2025)

CV 来源
  • 问题：如何把 RTL2MµPATH 推广到更开放、可迭代场景。
  • 方法：以 workshop 形式给出扩展验证案例。
  • 学习重点：方法迁移能力与可重复评估。
  • 复现建议：记录每轮发现路径与 false-positive 变化。

18) Formal Foundations of MemGlue (FMSD 2025)

CV 来源
  • 问题：conference 结果需要更完整理论化证明。
  • 方法：给出 MemGlue 协议形式化基础与性质论证。
  • 学习重点：从工程实证到理论完备性的过渡。
  • 复现建议：按论文给出的性质列表逐条写检查条件。

19) Spec and Formal Verification of HW-SW Contracts (IEEE Computer 2025)

Stanford Profile
  • 问题：如何把零散成果组织成可实践的高保障架构路线。
  • 方法：用契约中心化叙事统一 correctness/security/reliability。
  • 学习重点：把“方法”变成“体系”（research program）。
  • 复现建议：结合 #10 构建你自己的 contract 模板库。
