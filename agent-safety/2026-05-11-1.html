摘要（中文翻译，忠实原文）

Agent skills——一种结构化的指令、脚本与参考资料的封装包，可以在不修改大语言模型本身的前提下增强其能力——已经从一种便利特性，升格为头等部署 artifact。承载它们的 runtime 因此继承了包管理器和操作系统长久以来的同一个问题：一段内容声称自己具备某种行为；runtime 必须决定是否相信它。

本文给出的核心论点是：一个 skill 在被验证之前是"不受信任的代码"；加载它的运行时必须强制执行这一默认设置，而不是从签名、许可或来源注册表推断信任。这种"默认不可信"的姿态，必须在每一层（manifest、capabilities、policy、runtime profile）都被结构性地强制执行。

论文围绕这一论点提出了四件互相支撑的工程产物：

  • (1) Trust schema：在每一份 skill manifest 上挂一个显式的 verification level 字段；该字段不可被 skill 自身伪造。
  • (2) Capability gate：HITL（human-in-the-loop）策略是 verification level 的函数；高敏感能力（文件写、网络出向、shell exec）只有达到对应级别的 skill 才能 unlock。
  • (3) Biconditional correctness criterion：正确性是双条件的——即 "verification level 为 L ⇔ runtime 给出 capability set ≤ Cap(L)"；任一方向违反即为 runtime bug，可机械检查。
  • (4) Portable runtime profile：包含 10 条规范性指南（normative guidelines），可被任意 agent harness/runtime 实现。

该工作明确为 harness- and model-agnostic，无需重新训练或微调；目的是把"信任"从"假设它可信"翻成"在 runtime 上结构性地约束它"。

核心内容解读（背景·方法·差异）

解决了什么问题：过去半年 agent skill 生态的爆发——从 Anthropic 的 Skills、各大厂的 MCP server、社区维护的 skill 包仓库——把"把第三方 skill 包加载进 agent runtime"这一动作变成了部署默认行为。但这条路径在工程文献里没有被严格定义信任语义：大多数 runtime 默认信任 manifest 上的描述、信任来源域名、信任手工 review；这种"看起来 OK 就放行"的姿态，在 OS/包管理器的几十年血泪史里早就被证明会让攻击者反复打穿。本文把整个 agent skill 生态对齐到 OS/包管理的成熟视角——"未验证的代码 = 不可信"——并给出可在 runtime 上 mechanically check 的双条件标准。

核心方法（4 件互锁的产物）：

  • Trust schema：每个 skill manifest 必须声明 verification level（典型分级：unverified / community-vetted / org-signed / formally-verified），该字段由 runtime（而非 skill）写入，可由审计追溯。
  • Capability gate：HITL policy 是 verification level → capability set 的映射函数。低验证级 skill 只能调用 read-only / pure 计算原语；高敏感能力（fs.write、net.connect、process.spawn）严格要求高 verification 级别。
  • Biconditional correctness criterion：把 "正确性"定义为 双条件——"L ⇔ Cap ≤ Cap(L)"。这意味着：runtime 不仅不能给低级 skill 高权限（前向），也不能在 skill 已被验证时拒绝其应得的权限（反向）；两侧都可以被 fuzz 出违例。
  • 10 条 portable runtime profile：规范性的 runtime 实现指南，跨 harness 和 model 可移植；明确避免对模型权重做改动，因此既不需要重新训练，也不需要 PEFT 微调。
  • "unverified ≡ untrusted code"这一类比贯穿全文——把 OS/包管理器、MCM 的成熟工程经验直接迁移过来，是本文的概念骨架。
  • 明确反对 "trust by inference"：作者反复强调，runtime 不应从签名、许可或注册表来源（domain 名、author 名）推断信任；信任只能来自显式 verification 级别。

核心数据亮点 / 关键论点
  • "Unverified ≡ untrusted"——本文最尖锐的一句话：在 verification level 显式提供之前，所有 skill 都应被视为未受信任的代码。这一类比把整个 agent skill 生态对齐到包管理器的成熟视角。
  • Trust schema 字段不可被 skill 伪造：verification level 必须由 runtime / 第三方注册表写入，skill 自身的 manifest 不得自命验证级别——这是本文区别于"meta-data 即信任"路线的关键。
  • Capability gate = HITL policy 函数：把"哪些能力需要人审"的决策建模成 verification level → capability set 映射，让 HITL 强度自动随信任级别 scale，而不是固定一刀切。
  • Biconditional correctness："L ⇔ Cap ≤ Cap(L)"——双向都可机械 check，包括"已验证 skill 不应被错误降权"这种通常被忽视的反向 bug。
  • 10 条 portable runtime profile：跨 harness/model 可移植的规范性指南，意味着这个方案可直接附加到现有 LangChain / LangGraph / AutoGen / OpenAgents 之上，无需替换 runtime。
  • 明确 harness- and model-agnostic：不要求重新训练或微调，只在 runtime 配置层落地——这是本文工程上最有吸引力的一点。
  • 方法论站位：把 OS/包管理器的几十年安全工程经验完整迁移到 LLM agent 生态，理论与实践之间的桥比单纯的"加 guardrail"硬得多。
对你三个研究方向的启发
Harness Engineering

把 trust schema + capability gate 视为新的 harness 维度：未来 evaluation harness 应在每个测试用例上同时报告"该 skill 在多少 verification level 下还能完成任务"——这给 AgentFloor 那样的能力 ladder 加上"权限轴"，能力 vs 权限二维 frontier 比单一能力 frontier 信息密度高得多。对你 ZipperQuant 项目的 ablation 也有外推：把 W4A4 kernel 的"启用域"建成类似 verification level（如 "fp32 baseline / W4A4 dense / W4A4 + sparse residual / fused kernel"），让 ablation table 自然 capture 哪些下游任务允许激进量化、哪些必须保守。

Agent Skills Safety

本文是当前 agent safety 文献里最干净的"把工程经验迁移过来的"贡献之一。它的双条件正确性可以直接和你的研究方向对接：把 agent runtime 想象成 LLM 推理 stack 的"加载器"，每个 skill 包就像一个动态库；biconditional 让"权限错误降级"也能被 catch，这一反向检查在 LangChain 类 runtime 里是缺失的。结合今日 Harness 报告 AgentFloor 的 model-routing 视角，可以做一个"模型能力 × skill 信任级别"的二维 admission policy——只有 capability tier ≥ k 的模型，才能 invoke verification level ≥ L 的 skill，让 routing 同时是能力路由 + 安全准入。

Safety Benchmark

本文最直接的 benchmark 落地：构造一组"manifest 谎报 vs 真实能力不一致"的对抗 skill，用 biconditional 标准做评测。今日 Benchmark 报告 ML-Bench&Guard 已经在做政策对齐评测——下一步可以加一个"skill manifest 真实性"测试集：随机翻转 verification level 字段或描述字段，看 runtime 是否拒绝。这把 safety benchmark 从"模型是否拒绝有害请求"扩展到"runtime 是否拒绝错误声明的能力包"，对接的是更接近真实部署的攻击面。
