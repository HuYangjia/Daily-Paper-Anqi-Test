一句话核心贡献
把 LLM agent 的"UI 状态隐私防护" 从"云端依赖 + 二元 Allow/Deny" 升级为客户端可信本地代理 + 三元 K/A/R 策略——基于 Contextual Integrity 理论，为每个 UI 元素学习任务无关敏感度 s + 任务条件必要性 n 双分数；用 CI-aware 加权损失（高 s + 低 n 节点惩罚加大）防御最有害披露；GATv2 模型轻量本地部署；WebArena 衍生 662 测试集上把任务无关敏感泄露压低到 Full Observation 的 10.1%（降 89.9%），TCNP-I 99.31%，2FA / 密码 / Slack 通知 >99.9% 抑制，节点保留率 12%（Llama-3.3-70B 基线 34.5%）；ICML 2026 接收。
摘要（中文翻译，忠实原文）

现代基于 LLM 的自主 agent 越来越依赖丰富的 UI 状态观察，以在复杂数字环境中实现可靠的 action grounding。然而，许多部署即便大多数元素与当前任务无关，仍把完整 UI 状态传输至远程推理服务器，可能泄露敏感但不必要的上下文（认证码、私人通知、后台应用状态等）。

我们提出 MINIM——一个可信本地 broker，在任何观察离开设备前于客户端进行隐私感知最小化。基于 Contextual Integrity (CI)，MINIM 为每个 UI 元素学习双分数表示，预测任务无关的固有敏感度 sᵢ 和任务条件下的必要性 nᵢ,ₜ。两分数驱动一个三元披露策略：保留必要元素、按需抽象化敏感属性、移除任务无关内容。我们优化一个 CI-aware 目标，对高风险内容的必要性误差施加更强惩罚，从而在保留任务关键信息的同时实现激进剪枝。基于 WebArena 衍生的真实 UI 观察实验显示，MINIM 在保留任务关键语义上下文与可靠 agent 操作所需的交互可用性 的同时，大幅降低任务无关敏感泄露。

核心内容解读（CI 落地 / 双分数 / 三元 K-A-R 策略 / CI-aware 加权损失 / 部署形态）

立论：把"contextual integrity" 从理论推到部署形态。过去 20 年，Helen Nissenbaum 的 contextual integrity（CI，2004）是隐私理论的基石——"信息是否合规取决于角色 + 目的 + 转移规则"。但 CI 在 LLM 时代的实际部署落地一直缺位：5-19 Memory Laundering 揭示有毒上下文被压缩进 summary 后监测器看不见；6-12 #2 Containment Gap 审计 LangChain / AutoGPT / OpenAI Agents SDK 三大 framework 全部缺乏 memory integrity；6-13 #1 ELK 不可能定理证明仅靠行为反馈不能产生诚实 agent。MINIM 的姿态是：当 model 层和架构层都做不出 honesty / safety 时，把防护推到客户端，在数据离开设备前用 CI 强约束做privacy-preserving sanitization。这条姿态与今日方向 #3 WorkBench Revisited"capability 与 safety 同向提升" 形成精神同构——agent 防护必须从"事后过滤"前移到"数据源头"。

系统设计一：双分数表示——任务无关敏感度 + 任务条件必要性。MINIM 的核心创新是为每个 UI 元素 eᵢ 学习两个独立分数：

分数符号范围语义例：2FA 验证码
敏感度sᵢ[0, 10]任务无关的固有披露风险——一旦泄露后果有多严重sᵢ = 10（高）
必要性nᵢ,ₜ[0, 10]任务 T 条件下完成意图的必要程度购物任务 nᵢ,ₜ = 0（无关）
登录任务 nᵢ,ₜ = 10

这种正交分解非常关键——它解决了过去 LLM-based 隐私过滤的"规模并非银弹"问题（实验中 Llama-3.3-70B 的 TISL 0.312 反而高于 Llama-3-8B 的 0.206）：单个分数无法同时刻画"这条信息有多敏感" 和"这条信息对当前任务有多必要"。MINIM 用 GATv2（3 层 / 256 隐藏维 / 4 注意力头）+ 双 MLP head 同时输出 ŝ 与 n̂，输入特征 512 维（384 维 MiniLM 文本嵌入 + UI 属性 + 结构特征），保持模型轻量可本地部署。

系统设计二：三元披露策略 K / A / R——抽象化是关键中间动作。MINIM 不像 prompt-based LLM 过滤器只做 binary Allow/Deny，而是引入第三个动作：

动作触发条件输出
K (Keep)n̂ ≥ τ_nec ∧ ŝ < τ_sens完整保留（必要且低风险）
A (Abstract)n̂ ≥ τ_nec ∧ ŝ ≥ τ_sens占位符抽象化——保留结构角色，屏蔽敏感值
R (Remove)n̂ < τ_nec移除（任务无关）

部署阈值 τ_nec = 1.0 / τ_sens = 5.0。消融实验证明 A 动作的关键价值（Table 4）——把 A 退化为 R（保守 binary）使 TCNP 从 0.9491 降到 0.9050（−4.41pp），TCNP-I 从 0.9931 降到 0.9650（−2.81pp），代价仅 0.0021 TISL 改善。这条 finding 直接对位 6-12 HarnessBridge"trajectory-grounded rejection" / 6-12 Containment Gap"policy gate"——三者共享一个洞察：纯 binary 控制不足，必须有第三档 graded action（rejection / abstraction / approval）。

系统设计三：CI-aware 加权损失——惩罚最有害的披露。普通 L1 损失对所有节点平权，但"高敏感 + 低必要"节点的误判（如把 2FA 码标记为 keep）后果远比"低敏感 + 低必要" 节点的误判（如把装饰元素标记为 keep）严重得多。MINIM 的 CI-aware 必要性损失：

𝓛_nec = Σᵢ ( 1 + λ · (sᵢ/10) · (1 − nᵢ,ₜ/10) ) · |nᵢ,ₜ − n̂ᵢ,ₜ|

当 sᵢ 高（敏感）+ nᵢ,ₜ 低（不必要）时，乘法因子接近 1+λ，惩罚加大，迫使模型对"最危险的披露场景" 给出最准确的预测。配合敏感度损失 𝓛_sens（普通 L1）和 总损失 𝓛 = 𝓛_sens + α·𝓛_nec（论文中 λ = α = 1.0），形成端到端可训练目标。这条思想是把 CI 理论里"信息流的合规性取决于角色 × 目的 × 转移" 的 nuance 翻译成可微目标——是 5 月以来对位 SafeHarbor 信息熵驱动节点分裂、SkillJuror 语义受控变体的第三种方法学路径。

三个核心 finding：(a) 泄露压低 89.9%（TISL 0.101 = Full 的 10.1%）；(b) 规模并非银弹——Llama-3.3-70B 的 TISL（0.312）反而高于 Llama-3-8B（0.206），证明仅靠模型规模无法实施 CI 约束；(c) 节点稀疏性洞察——现代网页 1100+ HTML 元素中绝大多数是结构容器/装饰，真正必要元素稀疏，因此 12% 保留率合理且最有效。MINIM 的 TISL 是最强 LLM 基线 Llama-3.3-70B 的约 1/3。安全性验证：在 2FA/OTP 验证码（s=10）/ 密码提示（s=10）/ Slack 通知（s=8-9）/ 邮件预览（s=7-9）四类注入敏感内容上，>99.9% 被有效抑制——这是当前 prompt-based LLM 过滤器普遍做不到的水平。

方法学姿态：把 agent safety 从 cloud 推到 edge。MINIM 与今日方向 #1 HarnessX 在系统姿态上互补——HarnessX 把"能力提升" 推到 harness 层，MINIM 把"安全防护" 推到客户端层；二者共同构成 2026 H1 后期"层次化 agent 系统" 的方法论：cloud model（Anthropic / OpenAI / 等）+ 客户端 trusted local broker（MINIM）+ middleware harness（HarnessX）+ runtime monitor（Containment Gap memory integrity validator）四层栈。这种分层防护思路是过去半年 agent safety 论文（5-21 #2 CUGA / 5-19 #2 Memory Laundering / 5-23 #2 Latent-space Refusal Evasion / 6-11 #2 ELK 不可能定理 / 6-12 #2 Containment Gap）逐渐汇聚出的统一姿态：不要指望模型本身 honest，必须在系统的每一层都做可验证的约束。

核心数据亮点 / 关键论点
  • WebArena 衍生评测床：5,403 (tree, task) 变体（150 唯一可访问性树 / 27 任务类型 / 3 域：Shopping + Reddit + Gmail）→ 训练 4,741 / 测试 662。
  • 89.9% 泄露抑制：TISL 0.1010 = Full Observation 的 10.1%——任务无关敏感泄露压低近 10 倍。
  • 99.31% TCNP-I：交互元素保留率几乎不损失——agent 可执行性不受影响。
  • >99.9% 抑制：注入的 2FA / 密码 / Slack 通知 / 邮件预览四类敏感内容（按 s=7-10 分类）几乎完全不被传输。
  • 12% 节点保留率：远低于最强 LLM 基线 Llama-3.3-70B 的 34.5%——说明现代网页结构性元素稀疏、绝大多数对当前任务无关。
  • 规模并非银弹：Llama-3.3-70B 的 TISL（0.312）反而高于 Llama-3-8B（0.206）和 Gemma-3N-E4B（0.194）——证明仅靠模型规模无法实施 CI 约束。
  • 抽象化 (A) 不可或缺：消融实验显示去除 A 动作（A→R）使 TCNP 降 4.41pp / TCNP-I 降 2.81pp，仅 +0.0021 TISL 改善——graded action 是关键。
  • 任务语义鲁棒性：Sensitive/Admin 任务上 TCNP 与 TCNP-I 双 100%（K=15.84% + A=9.84%），证明在敏感任务上既未泄露也未拒绝执行。
  • 域级表现：Gmail 域 TCNP 96.61% / TISL 1.88%；Shopping 域 TCNP 93.60% / TISL 0.85%；Reddit 域 TCNP 95.38% / TISL 13.50%（社交内容更难判断必要性）。
  • 本地部署可行：GATv2 仅 3 层 / 256 隐藏维 / 4 注意力头 + 2 个 MLP head——轻量可本地推理，无需云端 LLM 调用。
  • 核心论断：当前 agent 部署中"把完整 UI 状态发到云" 是不可接受的隐私默认；MINIM 提供首个可部署的 CI-grounded 客户端 sanitizer，与 ICML 2026 会议背书。
对你三个研究方向的启发
Harness Engineering

MINIM 给你的 KV-cache 毕设最直接启发："必要性 + 敏感度" 双分数模型可类比"cache value + cache cost" 双分数模型。当前 KV-cache 系统层论文绝大多数用单一启发式（如 attention score 决定 keep/evict），但 MINIM 证明单分数不足——必要性与敏感度（或 cache value 与 cache cost）正交，需要分别预测。毕设方法论建议：(a) 借鉴双分数 + 三元策略——你的 cache 系统决策不应只是 keep/evict 二元，而应有 keep / compress / evict 三档（compress 对位 MINIM 的 abstract，保留结构但减小占用）；(b) 借鉴 CI-aware 加权损失——你的 cache 训练目标应在"高 value + 低 cost" 节点（最不该 evict 的）误判处加大惩罚；(c) 借鉴 客户端可信代理架构——MINIM 把决策推到 edge，你的 cache 系统也可借鉴"分层决策"思路（local fast cache + cloud verifiable cache）；(d) MINIM 的 GATv2 模型轻量本地部署经验直接对位"cache 控制器轻量推理" 工程问题。论文写作姿态：把 MINIM 列为"多目标 + 分层决策 + 轻量本地推理" 的方法学先例。

Agent Skills Safety（直击方向）

MINIM 是 agent skills safety 方向 2026 H1 后期最重要的系统论文之一——它让 CI 理论从"顶会论文中的 desideratum" 第一次变成"可部署的客户端 broker"。与 6-11 #1 ELK 不可能定理（DeepMind 形式化）/ 6-12 #2 Containment Gap（架构层审计）/ 6-14 #1 Did You Lie?（测谎器审计）形成"理论 + 系统 + 检测 + 防御" 四档完整证据矩阵。关键启发：(a) 分层防护已是共识——cloud model + edge broker + middleware harness + runtime monitor 四层；(b) 三元策略 (K/A/R) 是 binary control 的关键升级——这条姿态已在 6-12 HarnessBridge"trajectory-grounded rejection" / 6-12 Containment Gap"policy gate" / MINIM 三处独立印证；(c) "规模并非银弹" 现象需写入 safety 论文标准 caveat——即将更大模型用于 sanitization 不一定更安全；(d) CI-aware 加权损失是把"不对称风险" 写入训练目标的方法学样板，可推广到任意 safety 训练。

Safety Benchmark

MINIM 与今日方向 #3 WorkBench Revisited 在评测姿态上互补——MINIM 用 TCNP / TCNP-I / TISL 三档指标量化"隐私-效用 tradeoff"；WorkBench 用任务完成率 / 副作用率二档指标量化"capability-safety tradeoff"。两者都把"tradeoff 可量化" 立为 benchmark 设计的第一性原则。建议：未来 agent safety benchmark 论文应同时报告 (a) 主指标（任务成功率）、(b) 副作用指标（敏感泄露 / 副作用率 / off-target loss）、(c) graded action 比例（K / A / R 三档分布）。这三档在 MINIM Table 3（任务语义鲁棒性）和 Table 10（域级表现）已经完整呈现，可直接借鉴为 benchmark 报告范式。这条姿态在 6-09 #3 MAC-Bench"CSR + MG 双指标" / 6-10 #3 PreAct-Bench"Prefix Foresight F1" 之后构成 2026 H2 多指标 benchmark 的统一立场。
