摘要（中文翻译，忠实原文）

虽然基于大语言模型的多 agent 系统（LLM-MAS）通过编排专门化 agent 与外部工具，在解决复杂任务上展现了显著能力，但它们对工具输出的隐式信任构成了一个关键的攻击面。现有的工具攻击受限于领域专一或固定与静态的模板。

为应对这些挑战，我们提出 Evo-Attacker，把工具攻击形式化为一个自演化、记忆增强的强化学习过程。Evo-Attacker 构建一个动态攻击记忆（dynamic attack memory），并采用审议式推理（deliberative reasoning）来检索对抗模式，在关键时刻（critical moments）策略化地施加修改性干预。

此外，我们引入 Attack-Flow GRPO，通过终末结果（terminal outcomes）来优化中间推理步骤，解决长程信用分配（long-horizon credit assignment）的难题。综合实验证明 Evo-Attacker 一致地优于基线，凸显其泛化与演化能力，以及建设防御型工具防护的紧迫性。

核心内容解读（动态攻击记忆 · Attack-Flow GRPO · 关键时刻干预）

问题：为什么"静态模板攻击"已经过时？过去一年的 LLM-agent 工具攻击大多分两类：(a) 固定 prompt 注入（"忽略上面所有指令…"）和 (b) 领域专一脚本（针对某 MCP / 某 tool API）。这两类在 frontier LLM-MAS（如 OpenClaw、Code Agent、计算机使用 agent）面前会出现两个困境：单点 prompt 触发会被前置 guardrail 截断；领域专一脚本随版本更新而失效。Evo-Attacker 的根本观察是——工具攻击的真正脆弱面是"多步工具调用之间的隐式信任链"，而长程攻击需要有记忆、有 RL 信用分配、有关键时刻调度的攻击体而非脚本。

三大支柱：

支柱解决的问题关键机制
Dynamic Attack Memory 静态模板覆盖不足、不可迁移 把成功 / 失败的攻击轨迹索引化，按上下文 / 工具类型 / 模型族检索复用
Deliberative Reasoning + Critical Moment 每一步都注入会被立即识破 RL 主动选择"最有杠杆"的步骤施加修改，避免无意义噪声
Attack-Flow GRPO 长程奖励稀疏、无法把成功归因到中间步骤 用 terminal outcome 反传到中间推理步骤，类 RLHF GRPO 的"流式"变体

为什么这套设计像 attacker 版本的 ReAct？常规 ReAct 是 thought→action→observation 的明确多步循环，但安全研究里的 attacker 常常退化为"一次性 prompt"。Evo-Attacker 正是把 attacker 也升级为 ReAct 形态——读到 agent 中间状态后再决定要不要篡改，这与昨日的 Latent-space Refusal Evasion 一脉相承（都把"多步上下文敏感"作为一等设计）。但 Evo-Attacker 更强的是它把记忆和 RL 显式拼上，而不是单步表征级的 ablation。

"工具输出的隐式信任"为何严重？当前 LLM-MAS 的设计假设——子 agent 返回的 tool result 是 ground truth，主 agent 把它直接拼入下一步 prompt。Evo-Attacker 揭示这一假设在长程任务里几乎总会被攻破：只要中间某个工具被污染，污染会无监督地传到所有后续 agent。这条链路与 5-25 #2 PoisonForge "1% 中毒预算可击穿 11/12 模型"、5-23 Latent-space Evasion "ablation = 攻击" 共同说明 2026 年攻击研究范式的整体转向——从单点输入扰动迁移到长程系统级注入。

对你三个研究方向的启发
Harness Engineering

Evo-Attacker 反向证明了"verification layer + governance layer 不能省"——这与今天 #1 Harness 报告的 6 大评测维度完全互锁。它意味着 harness 设计里"每一步工具输出都必须配 deterministic 校验或可信元数据"，否则长程攻击会直接攻穿 memory substrate。可以把它视作 harness 评测维度"通信保真度（communication fidelity）"的极端反例样本。

Agent Skills Safety（直击主题）

这是 Agent Skills Safety 研究的新一代攻击范式标杆。它提出三个被动防御者无法回避的事实：(1) 工具调用链是隐式信任面，prompt-only guardrail 不够；(2) 攻击者已经有 RL + memory；(3) 关键时刻干预比均匀扰动更难检出。对应防御方向至少有三条：①给 tool result 加 verifiable provenance（参考 5-19 Provenance Position Paper）；②在 governance layer 部署"step-aware" detector（比 prompt detector 多一维步骤上下文）；③在 memory write 时强制脱毒。可以直接把这套 Evo-Attacker 当成红队基线，去测自己设计的 agent 安全方案。

Safety Benchmark

当前 agent safety benchmark 普遍只测"单点 prompt 注入"——Evo-Attacker 揭示这远远不够。新一代 benchmark 必须包含"带攻击者记忆的 多步 sandbox 评测"，把模型放在一个能演化的攻击者面前跑 N 步，记录何时被攻穿、被攻穿在第几步、被攻穿后的扩散范围。这是今天 #1 Benchmark 报告 AgentHijack（环境 corruption）之外的另一极。
