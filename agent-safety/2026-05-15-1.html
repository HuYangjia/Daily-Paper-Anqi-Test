摘要（中文翻译，忠实原文）

大型语言模型越来越多地作为自主 agent 工作，必须决定何时直接回答、何时调用外部工具。已有自适应工具调用研究大多把"是否需要工具"当作 model-agnostic 属性——由人或 LLM 评判，并多覆盖那些"显然需要"的极端例（如取天气 vs 改写文本）。然而真实部署中工具必要性远比这复杂：同一道题，强模型可以独立解、弱模型却必须借助工具——必要性随能力边界发散。

本文提出 model-adaptive 的工具必要性定义——把它锚定在"每个模型自己的实证表现"。在该定义下，作者比较 4 个模型在算术 QA 与事实 QA 上的"必要性 vs 实际调用行为"，发现显著 mismatch：26.5–54.0%（算术）与30.8–41.8%（事实）。

为定位失败原因，作者把 tool use 拆为两阶段——内部 cognition（模型是否相信工具是必要的）与 execution（模型是否真去 call）。对隐藏态做线性探针，发现两个信号都线性可解码，但在驱动 next-token 动作的"晚期层、最后一个 token"位置上，两个 probe 方向几乎正交。继续追踪样本在两阶段的轨迹，作者发现 mismatch 主要集中在 cognition→action 的转换，而非 cognition 阶段本身。

结论：LLM 工具调用存在知-行 gap（knowing-doing gap）——提升可靠性不仅需要更好地"识别工具需要性"，更需要更好地"把识别翻译成行动"。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去 12 天 agent-safety 主线一直围绕"外部加 guard"——5-08 SafeHarbor、5-09 Chain of Risk、5-12 PRISM、5-13 SKILL.md 攻击、5-14 HAM³。所有这些工作都把 agent 当 black box，要么注入扰动，要么外挂 guardrail。本文是第一篇沿"内部表征"打开 tool-use 安全的论文——它把"模型为什么不该 call 时却 call、该 call 时却不 call"定位到模型内部的两条独立方向。这对 agent safety 的意义远超 tool-use 本身：很多 agent 安全失效（refusal 不一致、policy 漂移）都可能是同一种 cognition→action 断裂的实例。

核心方法（model-adaptive 必要性 + 两阶段拆解 + 探针方向几何）：

  • Model-adaptive necessity：把工具必要性 N(x; M) 定义为模型 M 在题 x 上独立作答的实证准确率低于阈值 → 必要。这把 evaluator 的"主观必要"换为模型自身的可观测表现，是把 tool-use 评测从 ground-truth 一致改成 capability-conditioned。
  • 两阶段分解（cognition / execution）：cognition 阶段——"模型相信此题需 tool 吗"；execution 阶段——"模型实际是否生成 tool-call token"。两者既可来自同一前向、也可分别度量。
  • 隐藏态线性探针 + 方向几何：在每一层、每一个 token 位置训一个 logistic probe 解码两个信号。关键发现：两 probe 在大多数层方向对齐，但在晚期层 × 最后一个 token（即决定下一步 action 的位置）方向近正交——意味着模型的"认知信号"在最后被路由到一个不参与 next-token decoding 的子空间。
  • 轨迹追踪：把 mismatch 样本分成 4 类（认对+做对、认错+做错、认对+做错、认错+做对）；最终发现"认对+做错"占多数——即模型知道该 call 但没去 call。这就是"知行 gap"。

核心数据亮点 / 关键论点
  • 算术 QA 26.5–54.0% mismatch / 事实 QA 30.8–41.8% mismatch：跨 4 个模型，必要性与实际行为之间的 gap 都不小、且因模型而异——直接证伪"tool-use 是 model-agnostic 行为"这一长期默认假设。
  • 两个信号都线性可解：cognition probe 与 execution probe 在大多层准确率都很高——意味着模型不是不知道。
  • 晚期层 × last-token：probe 方向近正交：这是论文最具洞察的几何发现——cognition 信号在决定 next-token 那一刻不在 action 子空间。这解释了"认对但做错"这种最常见的失败。
  • 失败集中在 cognition→action 转换：不是 cognition 错、不是 execution 不会，而是从 cognition 到 action 的路由断了。这把研究焦点从"提升判断力"或"提升执行力"转到"提升路由对齐"。
  • 方法学含义：把 model-adaptive 这个定义引进 agent safety 后，过去所有 tool-use 安全 benchmark（含 Gorilla、ToolBench 等）都需要被重新审视——它们的 ground-truth"必要性"标签都是 model-agnostic 的。
  • 对齐的可干预面：既然 cognition 与 action 在 last-layer 是近正交的两个方向——理论上可以用一个低秩 rotation 把 cognition 方向与 action 方向对齐。这给了一个比 RLHF/RL-fine-tune 更便宜的干预面。

Harness Engineering

本文给 harness 工程一个清晰任务：必须把"模型自身实证准确率"作为评测的一阶变量——不能继续用 model-agnostic ground-truth 给 tool-use / refusal / hedging 这类能力相关的行为打分。这与今日 Harness 选定的 ChromaFlow（负向消融）互为镜像：ChromaFlow 揭露 harness 评测外部观测面的盲区，本文揭露 harness 评测内部表征面的盲区。建议你 KV-cache offload 论文的 v3 在第 5 章度量定义那里加一句：评测 prefetch 命中率时应锚到"模型自身在该层的注意力 sparsity 实证"，而非用统一阈值——这就是 model-adaptive necessity 的 KV-cache 翻译。

Agent Skills Safety（直击主线）

这是过去 12 天 agent-safety 路线最直接可用的"架构级干预" 论据。建议你的下一步研究：把"知-行 gap"从 tool-use 推广到 refusal-execution gap——模型在 cognition 阶段判断"该 refuse"的方向 vs 它实际生成 refusal token 的方向，是否同样在 last-layer 近正交？如果是，那么很多"模型说会 refuse 却最终满足了 jailbreak"的现象就有了机制层解释，且可用一个 lightweight rotation 修复——这比 RLHF 便宜两个数量级。它也直接契合你逐渐转向"具身智能基础设施/系统"的研究主线：embodied agent 的 perception → action 路径同样存在这种几何断裂，是系统级 alignment infra 的具体抓手。

Safety Benchmark

给 benchmark 社区的硬告诫：所有 tool-use / refusal benchmark 的 oracle 都应改成 model-adaptive。否则强模型会被无端罚（"它本可独立解但 oracle 说该 call tool"），弱模型则会被误判（"它根本解不了 oracle 却说不需 tool"）。今日 Benchmark 选定的 ClawForge 用 reference trajectory 而非 exact-match 评分——是同向但不同位置的修补；本文则把修补推到 oracle 定义本身。两者结合可设计 next-gen agent benchmark：oracle = model-adaptive 必要性 + 评测 = telemetry-driven 行为路径。
