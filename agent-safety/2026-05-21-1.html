摘要（中文翻译，忠实原文）

企业 agent 越来越被期待跨工具与界面自主操作，但生产部署要求"按构造治理"（governance by construction）：系统必须明确哪些动作允许、何时需要人类监督、哪些信息可被暴露，并且不为每个域重建 agent。本 demo 介绍 CUGA 的策略系统——一个模块化的 policy-as-code 层，与一个 generalist LLM agent 组合，能在组合工作流中提供可预测、可审计、合规感知的行为，且无需模型微调。

我们提出一种运行时治理架构——策略不是被动约束，而是在执行的每个关键阶段主动介入。具体来说，策略在 5 个结构化检查点 拦截 agent：

  • Intent Guard：上游 — planning 之前；阻止恶意/事故性的有害意图。
  • Playbook：system prompt 内 — 引导推理的结构化导调。
  • Tool Guide：tool-call 边界 — 强制工具的正确使用方式。
  • Tool Approvals：reasoning loop 之外 — 高风险动作的 Human-in-the-Loop 阀门。
  • Output Formatter：输出阶段 — 过滤并结构化最终响应。

这 5 个阶段把治理连续嵌进 agent 执行管线，而不是事后兜底。论文用一个医疗场景多层执行干预 demo 展示：动态 playbook 注入做结构化 tool-序列强制、intent guard 阻断恶意/有害请求、HIL 工具审批守住具有破坏性的动作。该 artifact 说明类型化治理基元如何让企业级 agentic 系统的部署"更快、更安全"，同时改进策略遵循度与执行一致性。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去一周的 agent safety 主线刻画了一条惊人清晰的"从模型层下沉到管线层"曲线——5-13 SKILL.md 揭示 markdown 即攻击面、5-15 "知-行 gap" 把工具调用刻画成 cognition / execution 不一致、5-19 memory laundering 提出 SPG，5-20 ECA 直接把幻觉重写成 authorization-failure 并用 deterministic gate 把 unsafe rate 打到 0%。CUGA 在此线上做了一次架构层综合：所有这些攻击面都需要一个统一的、跨阶段的 policy 干预层——单点防御不够，必须在 5 个点同时收紧。这就是"by construction"——治理不是 fine-tune、不是 prompt、不是事后审计，而是嵌进执行管线的拓扑结构。

核心方法：5 检查点 × policy-as-code × 组合工作流

  • 检查点 1 · Intent Guard（上游）：在 planning 启动之前对 user intent 做策略评估——阻止恶意/事故性有害请求进入 agent 执行域。类比 firewall 的"入站规则"。
  • 检查点 2 · Playbook（system prompt 内）：策略以"动态 playbook"注入到 system prompt——结构化引导 agent 在特定场景使用哪些 tool 序列。不是"自由发挥 + 事后管"，而是"上 rails"。
  • 检查点 3 · Tool Guide（tool-call 边界）：每次工具调用前的强制策略校验——tool 必须按 policy 定义的方式被调用，类型与参数都要满足合约。
  • 检查点 4 · Tool Approvals（reasoning loop 之外）：高风险动作进入 Human-in-the-Loop 队列——破坏性写入、跨账号操作、超阈值消费等。这是 ECA "authorization gate" 的工程化版本。
  • 检查点 5 · Output Formatter（输出阶段）：最后一道——过滤敏感信息泄露、结构化最终响应、保证可审计 trace 完整。

核心数据亮点 / 关键论点
  • 5 个结构化检查点覆盖 agent 执行管线的全部关键交叉口——Intent Guard / Playbook / Tool Guide / Tool Approvals / Output Formatter。
  • 不需要模型微调：所有 policy 都用 policy-as-code 写——把"治理"和"通用 agent 能力"解耦。
  • "动态 playbook 注入"：tool 序列的强制以 prompt-time 注入实现，相比静态 system prompt 更具上下文敏感性。
  • HIL 阀门用于破坏性动作：把人类反馈从训练阶段下移到关键决策时刻——这跟 ECA 的"authorization gate"在精神上同向。
  • "治理跨阶段连续" 是 CUGA 的核心论断——policies are not afterthoughts。
  • 医疗 demo 案例展示了：1) playbook 强制 tool 序列、2) intent guard 阻断恶意请求、3) HIL 守住破坏性动作的三重叠加。
  • 类型化治理基元意味着 policy 可被组合、可被复用、可被独立审计——对企业合规非常关键。
对你三个研究方向的启发
Harness Engineering（直击毕设）

CUGA 的"5 检查点"思路对你 KV-cache 仿真是一份精确的工程化模板：把当前的 cache 决策也改成"5 个结构化拦截点"——Pre-attention（哪些 K 行该被预热）/ Inside-attention（attention 阶段的实时驱逐 hint）/ Post-attention（淘汰决策）/ Spill-to-CPU（H2D 决策）/ Recall-from-CPU（D2H 决策）。每个点用 policy-as-code 风格的 declarative rule 描述，而不是把所有逻辑塞进同一个 evict 函数。这能直接让你毕设第三章的"组合策略"评测拥有 CUGA 同款的结构化可解释性——读者一眼能看到"Top-K prefetch 是哪一层介入、HU 驱逐又是哪一层介入"。

Agent Skills Safety（核心）

CUGA 把过去 7 天 agent safety 的所有片段（5-13 SKILL.md / 5-15 知-行 gap / 5-19 memory laundering / 5-20 ECA authorization-failure）缝合成一个工程架构——每个攻击面都被映射到 5 个检查点中的某一个：SKILL.md 攻击 → Intent Guard + Tool Guide；知-行 gap → Tool Approvals；memory laundering → Output Formatter；ECA → Tool Guide + Tool Approvals。这就是 agent safety 的"操作系统拓扑"。后续 RA/PhD 团队若要部署生产级 agent，可以直接对照 CUGA 的 5 点做 audit checklist——这比逐 paper 列攻击面工程化得多。

Safety Benchmark

对 benchmark 方向的启发是"按检查点切片报告"：未来的 agent safety benchmark 应该不只报告"整体 ASR"，而是按 5 个检查点分别报告——攻击在哪个点没被挡住？这会让 HarmBench / BenchJack / POLAR-Bench 等 benchmark 拥有更精细的攻击-防御映射。同样，AgentAtlas（5-21 harness）的 9 类 trajectory failure 可以直接对位 CUGA 的 5 检查点做 cross-reference——哪种 failure 来自哪一层缺乏 policy。这对你帮 PhD/RA 做 safety benchmark 是一个直接可用的报告骨架。
