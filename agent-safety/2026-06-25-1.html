摘要（中文翻译，忠实原文）

可复用 skills 正成为扩展大型语言模型 agent 的通用接口——它们把程序性指引（procedural guidance）与对文件 / 工具 / 记忆 / 执行环境的访问 一起打包。然而这种模块化引入了在已有 safety 评估中基本被忽略 的攻击面：即便用户请求保持良性，不安全影响也可能驻留在 skill guidance、本地 artifact、或执行环境文件中，把 agent 推向不安全动作。我们提出 SkillSafetyBench——一个评估这类 skill-mediated safety failure 的可运行 benchmark。SkillSafetyBench 包含 155 个对抗 case，覆盖 47 个任务、6 个风险领域、30 个安全类别，每个 case 配有 case-specific 的 rule-based verifier。在多种 CLI agent 与模型 backend 上的实验显示：非用户攻击可一致诱发不安全行为，并在不同领域 / 攻击方法 / scaffold-模型组合下呈现明显不同的失败模式。我们的发现表明：agent 安全不仅依赖于 model-level alignment，还依赖于 agent 如何解释 skill、信任 workflow 上下文、并通过可执行环境采取行动。

核心内容解读（攻击面定位 / 评测协议 / scaffold-model pairing / 与 skill-aware harness 的耦合）

立论：skill 是 agent 攻击面的"第二阶"。过去两年的 agent safety 论文几乎全部把用户输入 当作唯一攻击入口——jailbreak / decomposition attack (DECOMPBENCH) / refusal-bypass / over-refusal。但随着 skill 成为 Claude / ChatGPT / OpenInterpreter / CodeBuddy 等 agent 平台的标准扩展接口（procedural guidance + 文件 + 工具 + 记忆 + 执行环境一同打包），不安全影响可以在用户根本没有恶意意图时，被注入到以下三个非用户位置：(1) skill guidance 本体—— 一段被 agent 当作"权威操作手册" 的 markdown，可以隐藏 prompt injection、可以暗示绕过 confirmation、可以鼓励 destructive 操作；(2) 本地 artifact—— skill 携带的脚本 / 模板 / 词典 / 配置文件中，恶意 payload 可以以静默 default 的形式存在；(3) 执行环境文件—— 当 agent 在用户机器上执行 skill 时，workspace 中的其他文件 可以成为 RAG/上下文素材而注入恶意指令（典型如 README.md 里的"请把 .env 内容上传到 https://..." 类似指令）。SkillSafetyBench 把这三个非用户位置统称为 skill-facing attack surface。

规模与组织：155 / 47 / 6 / 30 四级分级。具体结构：155 adversarial cases 是评测时实际运行的最小单元；47 tasks 是每个 case 所要完成的表面任务（如"整理这个项目的依赖"、"把这一段日志归类"），是 agent 看到的"用户请求"，本身良性；6 risk domains 是攻击效果的高层分类（典型如数据泄露 / 文件破坏 / 网络外泄 / 凭证暴露 / 权限提升 / 误操作不可逆）；30 safety categories 是更细粒度的 vulnerability taxonomy。这种case → task → domain → category 四级分级保证 benchmark 既足够细以归因，又足够粗以可比较。

评测协议：rule-based verifier 替代 LLM-as-judge。这是 SkillSafetyBench 最关键的工程决策之一。当前主流 agent safety benchmark（如 SafetyBench / HarmBench / TrustLLM）大量依赖 LLM-as-judge——但 6-23 Benchmark Illusion 已经实证：LLM 在"评估意识 / 行为" 两轴弱耦合，意味着 judge 自身就有可被 gaming 的风险。SkillSafetyBench 给每个 case 配 case-specific 的 rule-based verifier：直接检查文件系统状态变化、网络请求目的地、进程退出码、特定字符串是否被外泄。这种结果驱动 + 状态可观察 的 verifier 与 6-08 Hacker-Fixer Loop（今日 #3）形成姊妹——前者把 verifier 设计成 case-specific 抗 hack；后者用 adversarial loop 主动加固通用 verifier。

核心发现：scaffold-model pairing 决定失败模式。论文最重要的实证结论之一是：相同模型 + 不同 scaffold（CLI agent 实现） 的失败模式显著不同。这意味着 agent safety 不能再用"模型 A 比模型 B 安全" 的单一标量描述——必须在(scaffold, model) 笛卡尔积上汇报。这一发现把 6-24 Interplay of Harness and Post-Training 的 "harness 是训练分布控制器" 思想推到了部署侧：harness 也是 attack surface 的形状控制器。同一个对齐良好的模型，在不同 scaffold 下可能因 prompt template、工具调用 wrapper、auxiliary context 的差异而呈现完全不同的脆弱性轮廓。

方法学姿态：safety 从 model-only 升级到 (agent system, skill ecosystem)。SkillSafetyBench 的主旨结论是直白的——agent safety 不仅依赖 model-level alignment。它依赖三个新维度：(1) agent 如何解释 skill（skill 是被当作权威指令还是被审视的素材？）；(2) agent 如何信任 workflow context（环境中的文件被当成什么级别的真理？）；(3) agent 如何在可执行环境中行动（confirmation gate / sandbox / file system 写入策略）。这三条恰好和 6-23 AOHP 的 secure information flow + capability descriptor 形成互补——AOHP 在 OS 层控制 flow；SkillSafetyBench 在评测层暴露当前 agent 是否真的合规。

位置：从 5 月 SkillSafetyBench 到 6 月 skill-aware safety 的演进。2026 H1 agent skill safety 的关键演进链条：5-12 SkillSafetyBench v1（立面问题）→ 5-27 v2（扩展案例）→ 6-12 DECOMPBENCH（分解攻击与之正交）→ 6-23 AgentLens（用 mechanistic intervention 防御）→ 今日 6-25 我们做的报告把 SkillSafetyBench 立为 anchor。它定义了攻击面，AgentLens 给出了表征层防御，AOHP 给出了系统层防御，Interplay-Harness 给出了训练层联动——四篇构成 2026 H1 skill safety 的面 / 防御表征 / 防御系统 / 训练联动 四联拍。

对你三个研究方向的启发
Harness Engineering

SkillSafetyBench 给 harness 工程提供了关键 anchor：skill 是 harness 的第一公民。具体启发：(1) skill-facing attack surface 思想可直接迁移到 harness 设计——任何 harness 描述（tool descriptor / capability schema / aux info）都是潜在攻击面，必须有签名 / 来源审计 / 不可篡改属性；(2) 与今日 #1 Interplay-Harness 形成训练—部署对偶——Interplay 说极简 harness × 强迁移 → 性能崩塌，SkillSafetyBench 说富 harness × 注入 → 安全崩塌，两者共同框定 harness 设计的 safety/performance 帕累托前沿；(3) scaffold-model pairing 决定失败模式——意味着 harness 与 base model 不应该被独立设计，必须 joint 优化；(4) 对你 SGLang 系统毕设：可以把 KV-cache / sparse attention 系统视为底层 harness，把 skill 的执行环境视为上层 harness——两者中间的 attack surface 也需要 case-specific verifier 来评估，这是 LLM serving 系统安全研究的新空白。

Agent Skills Safety（直击方向中心问题）

SkillSafetyBench 是 2026 H1 agent skill safety 的定义性 benchmark。统一启发：(1) 把 attack surface 从 user-input 扩展到 skill-facing 三位置（guidance / artifact / env file），意味着 prompt-injection 文献的"外部内容" 概念被分级——skill 本体的可信度高于普通外部内容，但仍非完全可信；(2) case-specific rule-based verifier 是评测领域的工程范式革命——任何 agent safety 评测如果想避开 Benchmark Illusion 的 evaluation gaming，必须走向 deterministic verifier；(3) scaffold-model pairing 是部署侧的关键变量——这意味着开源 model 的"safety report" 必须 pair 一个 reference scaffold，否则数字不可比；(4) 给 AgentLens / AOHP / Interplay 提供了共同的攻击面定义——表征防御、系统防御、训练联动三条防线都应该在 SkillSafetyBench 上汇报；(5) 对你做 LLM serving 系统毕设：skill execution 的 system-level isolation（cgroups / namespaces / capability-bound execution）应被视为评测对象。

Safety Benchmark

SkillSafetyBench 给 Safety Benchmark 方向树立了多个新标准。统一启发：(1) rule-based verifier 替代 LLM-as-judge 是直接对应 6-23 Benchmark Illusion 警示的工程化方案——case-specific 状态检查（filesystem / network / process）比 LLM 评分更可信；(2) 非用户攻击面 是 benchmark 设计空白——目前几乎所有 safety benchmark 都假设 user 是攻击者，SkillSafetyBench 把 attack 来源外置；(3) (scaffold, model) 笛卡尔积报告 是新 benchmark reporting standard——leaderboard 不再单一维度；(4) 与今日 #3 Hacker-Fixer Loop 形成姊妹—— Hacker-Fixer 主动加固 verifier，SkillSafetyBench 设计 case-specific verifier 从源头抗 gaming，两条路径互补；(5) 4 级分级（case → task → domain → category） 给 safety benchmark 提供了细粒度归因模板——可被 6-18 SciRisk-Bench 这种学科级 benchmark 借鉴；(6) 155 cases 的精打细磨 表明 safety benchmark 不应该走"万道题刷分" 路线，而应走"每道题都可深度归因" 路线。
