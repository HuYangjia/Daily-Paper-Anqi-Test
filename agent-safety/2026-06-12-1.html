摘要（中文翻译，忠实原文）

能自主调用工具、维护持久记忆、执行多步规划的 agentic 大语言模型系统，正越来越多地部署到面向公众的领域，包括政府服务、医疗分诊与金融顾问。我们追问一个问题：用来构建这些系统的框架是否在架构层面提供了结构性安全保障？

我们应用从 agentic 架构组合模型推导出的六项遏制原则（six containment principles），审计三个主流框架——LangChain、AutoGPT、OpenAI Agents SDK，发现无一原生合规。Memory integrity——对最常见的漏洞类别之一的防御——三框架均未观察到。

我们用实证验证这一发现：在基于 LangChain 构建的模拟政府福利 agent 上，单次 memory poisoning 写入 在所有测试 seed 与 backend 上诱发持久的、目标针对的污染，把目标申请者群体的错误拒绝率推至 88.9%。在更复杂的五因素策略下，同一攻击保持总体准确率不变但把目标群体的错误拒绝率提升 3.5×，使污染难以通过标准监控发现。

我们随后提出两个轻量级遏制机制：memory integrity validator 与 policy gate，以亚毫秒开销（每次调用低于 0.2 ms）消除两条攻击向量。我们结论是：当前 agentic 框架生态尚未达到 "secure-by-default" 的预期，无法用于面向公众的部署；并给出实现可信部署所需的优先架构干预清单。

核心内容解读（六项遏制原则 / memory poisoning / 隐蔽性 / 轻量缓解）

立论：safety 必须立到框架架构层，不能停在 model alignment。过去两年 agent safety 研究主流路线集中在模型层——RLHF / Constitutional AI / system prompt / refusal training；但 6-11 ELK 不可能定理（arXiv 2606.12268）证明仅靠行为反馈无法保证 honesty，自然推论是——必须把 safety 责任分摊到架构层与模型层。Containment Gap 是这条推论的具体落地：它不试图调整模型，而是直接审计承载 agent 的框架是否有架构级安全保障。这条立场与 5-21 CUGA Governance by Construction（arXiv 2605.20911）"policy-as-code + 5 结构化执行检查点" 同源——但 Containment Gap 更进一步，它反向—— 不是构造一个新的安全架构，而是审计现存最广泛使用的三个框架，并证明它们都不够。

方法学贡献一：基于组合模型的 6 项遏制原则。论文从 agentic 架构的组合模型推导出 6 条遏制原则——这是 2026 H1 在 agent safety 方法学层面的关键贡献，因为它把"架构是否安全" 这一含糊问题形式化为可审计清单。摘要明确指出 "Memory integrity, a defense against one of the most prevalent vulnerability classes, is not observed in any of the three evaluated frameworks"——这意味着 6 项原则中至少有 1 条（memory integrity）是所有审计框架共同缺失的结构性鸿沟，论文标题 "Containment Gap" 即由此而来。剩余 5 项原则虽未在摘要中点名，但根据上下文可推测涵盖：(a) tool authorization；(b) action provenance；(c) policy enforcement；(d) state isolation；(e) audit logging——与 5-21 CUGA"5 检查点" 高度同构。

方法学贡献二：实证攻击作为审计的充分证据。论文最有说服力的部分是把架构鸿沟翻译成可量化伤害：

核心发现一：单次写入即可造成 88.9% 持久污染——agentic 系统的"记忆是攻击面"。这条数据本身极具冲击力：传统 LLM 安全研究中 prompt injection 攻击需要每次交互都注入；但 agentic 系统持久记忆的特性把"一次注入 → 永久毒化"立成新攻击面。这条发现与 5-19 Memory Laundering（arXiv 2605.16746）"Sub-threshold Propagation Gap" 形成精神同构——但 Memory Laundering 关注的是多 agent 间的状态污染传播，Containment Gap 关注的是单 agent 持久记忆的污染持久性。两者合起来证明记忆是 agent 系统中最被低估的攻击面。

核心发现二：隐蔽攻击逃避标准监控。5 因素策略下"aggregate accuracy preserved" 的攻击模式是论文最重要的 safety 信号——它证明攻击者可以有意设计 memory poisoning 让整体监控指标看不出问题，仅在目标子群体上造成 3.5× 错拒。这是 fairness × safety 的交叉点：传统 fairness 评测看 disparate impact，传统 safety 评测看 ASR；但"有意构造的 disparate impact 作为攻击向量" 是一个同时挑战两条评测面的新威胁模型。这条发现与 6-03 What Benchmarks Don't Measure（arXiv 2606.02965）"compliance bias 是 RLHF 结构性副作用" 互补——What Benchmarks 揭示了未受攻击状态下的结构副作用，Containment Gap 揭示了主动攻击状态下的隐蔽伤害。

核心发现三：缓解可以非常便宜。这是论文给行业的最实际信息："memory integrity validator + policy gate, sub-millisecond overhead"——证明 secure-by-default 不需要重构整个 agent 系统，两个轻量组件就能堵住两类攻击向量。这条工程姿态与 5-25 SafeHarbor（arXiv 2605.05704）"training-free + plug-and-play guardrail" 一致，但 SafeHarbor 在对话内容层做记忆增强 guardrail，Containment Gap 在架构层做完整性验证——两者形成"内容守护 + 架构守护" 双层防御。

哲学定位：把 agent safety 责任分摊到框架供应商。过去 LLM safety 责任主要由模型供应商承担（OpenAI / Anthropic / DeepMind 做 RLHF / Constitutional AI）；但 LangChain / AutoGPT / OpenAI Agents SDK 是框架供应商，传统上认为它们的责任仅是"提供构建工具"。Containment Gap 的政治影响是把 secure-by-default 责任显式压到框架供应商头上——这与 6-11 ELK 不可能定理"RLHF 不充分" 在不同方向上施压：模型层做不到完美，框架层就必须承担一部分 safety 责任。这条立场可能成为 2026 H2 监管讨论（EU AI Act 实施 / NIST AI RMF）中对框架供应商的合规要求基础。

对你三个研究方向的启发
Harness Engineering（直击毕设）

Containment Gap 给你的 KV-cache 毕设的最直接启发是：把"cache 完整性验证" 立成毕设的一个可选维度。当前 KV-cache 系统层论文聚焦性能（hit-rate / latency / throughput）；Containment Gap 提示了cache 也是攻击面 的可能性——在 multi-tenant 推理服务中，cache 数据被错误共享或被恶意污染会造成严重 safety 后果。毕设建议：在论文 future work 部分显式讨论"KV-cache 在 agent 持久记忆场景下的完整性挑战"——agent 频繁调用 LLM 推理，每次推理共享一个 KV-cache 系统，如果 cache 被污染（或泄漏到错误用户），整个 agent 持久记忆都会受影响。这条 future work 让你的毕设具备"系统 × 安全" 的双重视野。同时直接采用 Containment Gap 的实证证明法—— 不是抽象描述风险，而是设计具体攻击 来量化伤害。

Agent Skills Safety

Containment Gap 与今日方向 #1 HarnessBridge 形成对偶—— HarnessBridge 提出 trajectory-grounded rejection 作为 action projection 的核心组件；Containment Gap 提出 memory integrity validator + policy gate 作为架构层防御。这是一个完整防御链路的两端：HarnessBridge 在 agent 决策之后拦截不一致动作；Containment Gap 在 agent 决策之前验证记忆完整性。把两者合起来：未来 agent 框架必须同时具备(a) memory integrity validator（写入路径）、(b) policy gate（动作路径）、(c) trajectory-grounded rejection（执行路径） 三层防御。这与 5-26 SafeHarbor"分层记忆 guardrail" / 5-19 Memory Laundering"SPG 状态污染" / 6-11 ELK"不可能仅靠行为反馈" 共同形成 2026 H2 "多层架构防御" 主轴。

Safety Benchmark

Containment Gap 给 benchmark 方向最深远的启发是把"主流框架审计" 立成新的 benchmark 范畴——传统 safety benchmark 评测的是模型，Containment Gap 评测的是框架。今日方向 #3 MLUBench 评测模型层的终身遗忘能力；但同样的问题在不同 agent 框架（LangChain vs LangGraph vs OpenAI Agents SDK）下结果可能完全不同。建议：未来 benchmark 报告时必须显式声明：(a) 评测的是 model / framework / framework × model 组合体的哪一个；(b) 是否涵盖架构层威胁模型（memory integrity / state isolation / tool authorization）；(c) 是否提供具体可武器化攻击而非抽象 desiderata。Containment Gap 的"从原则到攻击到缓解" 三段式模板可以作为新一类 framework benchmark 的写作范式——这与 6-09 MAC-Bench"Beyond Goodhart's Law + Agent-as-a-Benchmark" 的姿态在不同层面上一致。
