摘要（中文翻译，忠实原文）

当前 LLM agent 在调用孤立的 API 上已经熟练，但在商业软件自动化的"最后一公里"上仍然挣扎。真实场景里的工具不是相互独立的——它们是原子的、相互依赖的、对环境噪声敏感的。

我们提出 ComplexMCP，一个面向上述严苛条件的评测 benchmark。它建立在 Model Context Protocol (MCP) 之上，提供 300+ 经过精细测试的工具，来源是 7 个有状态沙箱（stateful sandboxes），覆盖从办公套件到金融系统的真实业务环境。与已有数据集不同，ComplexMCP 采用 seed-driven 架构来模拟动态环境状态与不可预测的 API 失败，从而保证评测既确定性又多样化。

我们在 full-context 与 RAG 两种范式下评测了多个 LLM，结果揭示出鲜明的性能差距：即便顶级模型，成功率也无法突破 60%，远落后于人类的 90%。基于细粒度轨迹分析，我们识别出三条根本性瓶颈：(1) 工具检索饱和（tool retrieval saturation）——随着动作空间扩大，agent 无法稳定定位正确工具；(2) 过度自信（over-confidence）——agent 跳过必要的环境校验；(3) 策略性认输（strategic defeatism）——倾向于把失败合理化，而不是继续追求恢复。

这些发现表明：当前 agent 不足以应对相互依赖的工作流。ComplexMCP 因此被定位为下一代韧性自治系统（resilient autonomous systems）的关键测试床。

核心内容解读（背景·方法·差异）

解决了什么问题：过去十天的 harness 工作正在做一件事——把 agent 系统的隐含决策面外化为可评测变量：Token Arena 把 (provider, model, SKU) endpoint 当评测单元；Uno-Orchestra 学 (model, primitive) 决策面；More Is Not Always Better 把 scaffolding 子集做 2⁵ 全因子；AgentFloor 把 model-size routing 建成 capability ladder。但这些工作几乎都把"工具"当作孤立 API 调用。ComplexMCP 攻的是另一根硬骨头：当工具有状态、相互依赖、还会以不可预测方式失败时，agent 是否仍然可靠？这是从"toy tool-calling"到"production agent"的真实门槛。

核心方法（MCP 原生 + seed-driven 故障 + 7 个有状态沙箱）：

  • 建于 MCP 协议之上：所有 300+ 工具按 Model Context Protocol 描述与暴露，agent 调用接口与 production 完全一致——没有"toy adapter"开销，是当下最贴近真实生产 agent 的工具评测层。
  • 7 个有状态沙箱：从 office 套件到金融系统跨域覆盖。"有状态（stateful）"是关键修饰词——意味着工具调用的成功不仅取决于"参数对不对"，还取决于"环境处于什么状态"。这是把"工具相互依赖"做成第一原则。
  • seed-driven 评测架构：用同一个 seed 可以重放整个环境状态序列与 API 失败序列。确定性（deterministic）+多样化（diverse）第一次同时成立——和单次随机 fuzz 完全不同，这给评测体系装上了"压力测试可重放"。
  • 双范式评测：同时跑 full-context（把所有工具描述塞进 context）和 RAG（按需检索工具描述），直接量化"工具检索饱和"作为 agent 失败的主要因子。
  • 失败模式细粒度归因：基于轨迹分析将失败拆为三类——retrieval saturation / over-confidence / strategic defeatism。这给 harness 工程提供了"按失败模式切片"的诊断面。

核心数据亮点 / 关键论点
  • 300+ MCP 工具 × 7 有状态沙箱——首个跨办公套件、财务系统等真实业务的 MCP-native 工具评测层；规模上是 τ-Bench 子域工具数的 5–10×。
  • 顶级模型成功率 < 60%（人类 ≈ 90%）：在动态环境与故障注入下，前沿模型与人类间存在 30+ pp 鸿沟，这是当下 agent 系统在 production 部署中的核心可靠性 gap 量化。
  • seed-driven 可重放：同一 seed 重放完整状态序列与失败序列——这是 deterministic + diverse 的关键设计，使"压力测试"第一次能进入 CI pipeline。
  • 三大 root cause：(1) tool retrieval saturation 在 RAG 范式下随动作空间扩大显著恶化；(2) over-confidence 在状态变更类任务上表现为"跳过 verify 直接 act"；(3) strategic defeatism 在多跳失败后表现为放弃 recovery。
  • full-context vs RAG 范式对比：揭示出 RAG 在大工具空间下的可扩展性优势，但同时暴露 retrieval saturation 这条新瓶颈——这是 production router/scaffolding 设计的一手依据。
  • 方法学重要性：把"工具相互依赖 + 环境噪声"做成 harness 第一原则，是从 toy tool-calling benchmark 走向 production-grade agent harness 的标志性一步。
对你三个研究方向的启发
Harness Engineering

ComplexMCP 给 harness 引入两条新原则：(1) 工具拓扑作为评测变量、(2) 动态环境状态可重放。和 AgentFloor 的 capability ladder、Token Arena 的 endpoint SKU 拼起来，可以构成一个四轴坐标系 (model size, scaffolding subset, endpoint, tool topology+seed)。对你正在做的 ZipperQuant W4A4 e2e benchmark：可以借 seed-driven 思路把"输入序列分布"也固定下来——不只是 prefill_len/decode_len 长度变量，连具体 token 流也由 seed 决定，这样 W4A4 vs BF16 的对比会变成完全可重放的、按 seed 拆分的 distribution-level 对比，比单纯报均值更有说服力，也能直接定位某些 seed 上的 quality regression。NeurIPS 投稿的 ablation 也可以学它"按失败模式切片"，把性能 drop 拆为 retrieval / over-confidence / defeatism 三类。

Agent Skills Safety

ComplexMCP 暴露的"over-confidence + skip environment verification"几乎是 agent skill safety 的核心攻击面：当 skill 在错误的环境状态下被调用，agent 不仅会失败，还可能造成不可逆副作用（误删/误转账）。这给 Skills as Verifiable Artifacts 提出的 capability gate 提供了一个直接的设计原则：每个高敏感 skill 调用前必须 enforce 环境状态前置条件（pre-condition check）。把 ComplexMCP 的 sandbox + seed 直接复用为 agent safety harness，可以做"特定 seed 下故意触发 over-confidence，观察是否被 capability gate 拦截"——这是一个非常自然的红队 benchmark。

Safety Benchmark

seed-driven 故障注入直接给 safety benchmark 提供了缺失的"压力维度（stress dimension）"——以前的 safety benchmark 大多在 happy path 下做单次评测，ComplexMCP 让 fail injection 成为评测变量。今日 Benchmark 方向 Agent-ValueBench 已经在做"价值潮汐"跨 harness 的现象观察——下一步自然是把 ComplexMCP 的故障注入嵌进 Agent-ValueBench 的 4 个主流 harness，看 agent 在 stress 下是否仍能保持价值一致性。这是更接近真实部署的"safety under stress"评测形态。
