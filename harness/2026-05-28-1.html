一句话核心贡献
把 agentic LLM serving 的"框架↔引擎接缝"重写为第三层 runtime——以 agent identity 为共享坐标暴露 observe / score / predict / act 四原语，9 类跨层策略统一接入。
摘要（中文翻译，忠实原文）

多智能体 LLM 系统已成为主流生产工作负载，但底层 serving 栈并非为它们而设计。上层的 agent 框架知道 agent 身份、角色、模式和调度结构，却看不见引擎级事件；下层的 serving 引擎看到每一条事件，却对 agent 一无所知。许多跨层策略——前缀缓存、批处理形状、推测解码、公平性、工具结果记忆化、安全执行——同时依赖两侧信息，目前只能靠对其中一边打补丁来"临时缝合"。

我们主张这条接缝应当通过架构变更而非点修复来解决：在框架与引擎之间插入第三层——agent runtime layer，对外暴露四个原语 observe（观察）、score（评分）、predict（预测）、act（行动），任何 agent 感知策略都可以接入，以 agent identity 作为共享坐标。我们把 9 种具体策略映射进该层，并对其中"直接服务成本杠杆最大"的一个——跨会话 KV 缓存——做深入验证：实例化为 CacheSage，在线学习每个工作负载的 agent 转移矩阵，用于基于"生存概率"的驱逐与"步间预取"。

在 5 个真实 multi-agent workload（MMLU / MT-Bench / GAIA / GSM8K + 12-agent 合成链）上，相对未修改的 serving 栈：缓存命中率 +13~37 个百分点，平均 TTFT 降低 12%~29%，吞吐 +6%~14%。

核心内容解读（接缝问题 · 四原语 · 9 策略映射 · CacheSage 单点验证）

问题背景：2024-2026 上半年的 LLM serving 栈通常是双层结构——上面是 AutoGen / MetaGPT / SWE-Agent 这类 agent 框架，知道 agent role、tool schema、planner 状态；下面是 vLLM / SGLang / LMCache 这类引擎，知道 KV block、batch shape、prefix。而所有"跨层策略"都被夹在中间：前缀缓存只看 token 哈希、看不见 agent 身份；公平性只看 request 来源、看不见 agent role；安全执行只看 tool schema、看不见请求级 KV 状态。每加一个策略，就要在两层各打一个补丁，且要同步重写两套 IR——这是当前 serving 栈在 multi-agent 场景下扩展性最大的瓶颈。

四原语 IR：论文把这条接缝形式化为一个独立的中间层，提出统一接口：

原语语义典型用法
observe(event)引擎事件流入运行时block touch、tool call return、agent step start
score(state)运行时对块/请求/agent 打分survival probability、cache priority、fairness slot
predict(agent)运行时预测下一步转移矩阵下一个 agent / 下一个 prefix
act(decision)把决策回写引擎预取 prefix、驱逐块、调整 batch、阻断不安全 tool

这一层以 agent identity 作为共享坐标——每个块、每个请求、每个 step 都被打上 (agent_id, role, session_id) 标签，使得"只看 token 哈希就丢失语义"的问题被一次性解决。

9 类跨层策略统一映射：论文把当前 serving 文献里散落的 9 类策略——前缀缓存、批处理形状、推测解码、公平性、工具结果记忆化、安全执行、KV 卸载、调度路由、跨 agent 复用——全部重写为四原语调用，证明该 IR 的表达完备性。

CacheSage 单点深入：论文不止于框架级讨论，而是挑出"最有直接服务成本杠杆"的一个策略——跨会话 KV 缓存——做完整原型验证。CacheSage 由三个组件构成：

  • 转移学习器（Transition Learner）：在线学习每个工作负载的 agent 转移矩阵 P(at+1 | at, ctx)，处理"同一 framework 在不同任务源下路由模式不同"——论文实测个别边的转移概率在不同任务源下可变化 27~47 个百分点。
  • 生存概率评分器（Survival Probability Scorer）：把转移矩阵的下游期望命中率，转换为单个 KV block 的"未来仍会被命中"的概率，作为驱逐优先级——比 LRU 直接看时间衰减更精确。
  • 工作负载门控的步间预取（Workload-Gated Inter-Step Prefetch）：在 agent step 切换时主动把"下一个 agent 高概率引用的 prefix"预热回 GPU KV，比被动 prefix cache 提前 1 个 step。

CacheSage 的关键观察（论文 §3）：(i) 跨会话表面（system prompt + tool schema + few-shot）占首轮 prompt 的 34%~52%——这是 KV 卸载方向"常驻热块"假设的实证支撑；(ii) 下一个 agent 可预测性的相对条件熵减少 R ∈ [0.40, 0.48]——意味着 agent 转移并非纯随机，预取有信号；(iii) 同一 framework 在不同任务源（如 MMLU vs MT-Bench）的转移概率大幅漂移——说明"全局静态预取"在 multi-agent 下注定失效，必须 per-workload 在线学习。

开销分析：论文给出热路径成本——observe ≈ 1 μs/块触摸、BFS scorer 单位数 μs、协调器 24 agent 状态 ≤ 25KB。这意味着该 runtime 层不引入显著调度开销，可在生产负载下直接打开。

与你毕设最直接的联系：本文的 CacheSage 与你毕设里"稀疏注意力 KV-cache 卸载 + 双水位线 GPU 热缓存 + CNN/Linear prefetcher"的设计在 IR 层面高度同构——你毕设的 prefetcher 可以直接被解释为 CacheSage 四原语中的 predict 实现，而双水位线则是 score 的具体形式。Q.md 评审里关于"为什么不直接看 latency"的质疑，可以反引这篇论文的"agent identity is the shared coordinate"作为 first principle 答辩弹药——agent 上下文如果不进入 KV 决策，命中率/带宽都是有偏估计。

Agent Skills Safety

论文的 9 策略表里明确有"safe execution"——这意味着 safety guardrail 也可以接入到同一 runtime 层，与 KV 调度共享 agent identity 坐标。结合昨日 #1 SafeHarbor（分层 guardrail）和今日 #2 Agent Skill Ecosystem Threats（恶意 skill 泛滥），可以构想一个"policy-driven safety runtime"：每个 skill 调用先经过 score(safety)，再经过 act(allow/deny/audit)。这给出了"agent runtime 即 safety substrate"的统一架构。
