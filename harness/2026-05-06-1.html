一句话核心贡献
把"实验室单次跑分"换成"亿级事件流连续监测"——首次系统性证明 ROUGE/BERTScore/AgentBench 完全测不到 7 大生产失败模式中的 4 个。
摘要（中文翻译，忠实原文）

现有 LLM 评估框架——包括 HELM、MT-Bench、AgentBench、BIG-bench——都是面向受控、单会话、实验室规模的设定设计的。它们无法应对 Agentic AI 系统在生产环境持续运行时所暴露出的评估挑战：决策错误的复合、工具失效的级联、非确定性输出漂移，以及长时间跨度任务下"无 ground truth"的窘境。本文给出三项贡献：

(1) 提出一个由生产观察归纳而成的 7 类 Agentic 失败模式分类法，每一类都有亿级事件量的真实样本支撑；
(2) 实证表明 ROUGE / BERTScore / accuracy / AUC 以及上述 Agent benchmark 无法识别其中 4 类失败模式，对另外 3 类则要延迟若干评估周期才能察觉；
(3) 提出 PAEF（Production Agentic Evaluation Framework）——一个面向生产流量、五维度、可持续运行的评估框架，并提供开源参考实现，专为取代而非补充过去那种"季度性 benchmark 跑一次"的工作流。

核心内容解读（背景·方法·差异）

解决了什么问题：Harness 主线最近一年逐步把"自动化"做深——从 AHE 的观测驱动自演化、Last Harness 的 Meta-Evolution、VeRO 的 Edit-time 评测，到昨日 AgentPulse 把外部社区信号纳入连续评估——大家都在把 harness 范围向外推。但所有这些工作仍然把 Agent 当作"系统外的被测对象"，它们要么使用人造工作负载，要么使用社区的二级信号。PAEF 第一次直接走进了正在跑生产流量的 Agent 内部，把"线上事故"作为评估的一等公民。

核心论点：实验室 metric 测不到 4 类失败模式。下表是论文给出的失败模式清单与传统指标的覆盖情况：

失败模式（论文归纳）典型表现传统 metric 是否能识别？
Compounding decision errors每步小偏差，几十步后跑偏到完全错误目标❌ 单步打分被平均掉
Tool failure cascades工具 A 给了脏数据，B 拿来生成了"自信但错"的结论❌ 工具出错本身被遮蔽
Non-deterministic output drift同一输入，今天明天答案不同；trajectory 分布漂移⚠️ 多周期延迟才能看出
Missing ground-truth long-horizon"写一份季度市场报告"没有标准答案❌ ROUGE/BERTScore 失效
Unsafe action emergence合规步骤 ×N → 复合违规⚠️ 单步合规率掩盖整体风险
Context-window erosion长会话后忘了用户初始约束❌ Benchmark 默认短任务
Adoption-quality decoupling能力提升但用户流失（或反之）❌ Benchmark 不感知采用

PAEF 的五维度（论文表 3）：

  • Outcome Quality——含 LLM-as-judge、用户反馈、规则校验，用以替代单一 ROUGE 类指标。
  • Trajectory Coherence——衡量 Agent 多步决策路径是否前后逻辑自洽（覆盖"compounding errors"和"context erosion"）。
  • Tool Reliability——单独把工具调用错误与上游脏数据建模成可观测信号（覆盖"tool cascades"）。
  • Drift & Variance——对相同输入的输出分布做时间序列监控（覆盖"output drift"）。
  • Operational Safety——以"trajectory-level"违规率代替"step-level"违规率，补齐 Owner-Harm、Symbolic Guardrails 在线上的盲区。

与现有工作的核心差异：

  • 对比 HELM / MT-Bench / BIG-bench：它们仍是"prompt → response"的单回合静态打分；PAEF 接的是"trajectory"，每条都包含工具调用、状态变化与时间戳。
  • 对比 AgentBench：AgentBench 把任务搬到沙盒，PAEF 直接挂在已有线上系统的可观测面（OpenTelemetry trace + LLM judge），无需把 Agent 重新塞进 sandbox。
  • 对比 AgentPulse（昨日 Harness 报告）：AgentPulse 用 GitHub/社区/榜单等"外部生命体征"评估 Agent；PAEF 用"内部失败模式"评估 Agent。两者构成互补：外部生命体征 + 内部失败诊断 = 完整的 production harness。

对你三个研究方向的启发
Harness Engineering

PAEF 把 Harness 研究从"实验台"正式扩展到"生产监控管"。结合昨日 AgentPulse 的"宏观社区信号"，可形成"内观 PAEF + 外观 AgentPulse" 的双层 production harness。建议下一步 Harness 工程把"trajectory schema"标准化（OpenTelemetry-Agent 提案值得跟进），让不同 Agent 框架都能直接接入同一套五维度评分器。

Agent Skills Safety

论文把 Operational Safety 从"step-level"升到"trajectory-level"，与 SafetyDrift（"几个安全步骤复合成数据外泄"）和 Owner-Harm 形成自然对接。建议把 Symbolic Guardrails 等运行时防护策略的评估指标，从"被拦截率"改为"trajectory-level 违规漂移率"——这正是 PAEF Drift & Variance 维度提供的新接口。

Safety Benchmark

"4/7 失败模式 benchmark 完全测不到"是对当下 Safety Benchmark 设计哲学的有力警示——纯静态题面只能覆盖显性风险。Safety Benchmark 设计者应该把生产侧的失败模式（compounding error、tool cascades、context erosion）作为新的题型注入，使其与今日 Benchmark 报告 Dreadnode 红队 Agent 的"自动化高吞吐红队"结果形成内外闭环。
