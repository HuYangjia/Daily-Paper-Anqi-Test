摘要（中文翻译，忠实原文）

自主语言模型 agent 越来越多地把规划、工具调用、文档处理、浏览、代码执行、验证循环组合在一起。这些能力让 agent 系统更有用，但同时也引入了一类只看最终准确率看不见的操作性失败模式。本报告提出 ChromaFlow——一个工具增强自主推理框架，围绕规划者主导的执行、专门化工具调用、以及 telemetry-driven evaluation三条原则建造。

我们在严格干净评测约束下、于 GAIA 2023 Level-1 validation 53 题上分析 ChromaFlow。一个冻结的 Level-1 完整 baseline 拿到 29/53（54.72%）正确。一个加了"扩展 orchestration"的恢复配置则只拿到 27/53（50.94%）正确，同时 traceback、timeout、tool-failure 出现次数、token-line 调用数和 campaign-log 成本估计全部上升。两次随机 20-题的 smoke 评测得到 12/20 与 11/20——表明小尺度上看起来的"诊断收益"在样本间并不稳定。

因此中心结果是一份负向消融：更激进的 orchestration 没有提升整体性能、反而抬升了操作噪声。报告主张：有界的 planner escalation、确定性 extraction、evidence reconciliation 与显式 run gate 应被视为可靠 autonomous agent 评测的一等要求（first-order requirements），而不仅是工程便利。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去两周 harness 主线已经把外化决策面逐项摆出来——5-08 endpoint（Token Arena）、5-09 scaffold（More Is Not Always Better）、5-12 工具拓扑（ComplexMCP）、5-13 声明语义（Evaluation Differential）、5-14 前瞻 metacognition（TRIAGE）。所有这些工作都在"加结构"的方向推进。ChromaFlow 是第一份明确报告"加结构反而变差"的硬证据——并把责任归到"没有把操作性遥测当 first-order 评测变量"上。这是 harness 工程从"加东西能不能更好"转向"加东西在评测面上要付什么代价"的关键拐点。

核心方法（telemetry-driven evaluation 的三条柱）：

  • Planner-directed execution + 专门化工具：所有调用走显式 planner，不允许"匿名 retry"——使每次 escalation 在 telemetry 上有可追溯的信号。这是把 agent 系统从"behavioral"放到"structural"评测面的前提。
  • Telemetry-driven evaluation：报告把 6 个量明确列入评测面——准确率、traceback 次数、timeout 次数、tool-failure mention、token-line call、campaign-log cost。关键论点：这 6 个量必须 joint report，否则会把"看起来涨了 1 题、其实多花一倍 token 还多 4 个 traceback"当作改进。
  • Run gate + bounded planner escalation：每次 planner 升级前需经过显式 run gate，不允许"无约束追加 orchestration step"——这是对昨日 TRIAGE 的"前瞻 commit"思想的执行端镜像：TRIAGE 强迫模型在执行前 commit 计划，ChromaFlow 强迫 harness 在执行中限制 planner 的迭代深度。

负向消融的实验设计（其严谨性正是论文价值的核心）：

  • Frozen baseline (Level-1 full)：29/53 = 54.72%；这是"什么都不加"的对照。
  • Recovery 配置（扩展 orchestration）：27/53 = 50.94%——下降 ~3.78 pp（绝对）。但 traceback、timeout、tool-failure mention、token-line call、cost 估计全部上升。
  • 2 次随机 20-题 smoke：12/20、11/20——若拿这两次 smoke 当"局部胜出"则会得出反向结论；这正是论文要警告的"小样本 diagnostic gain 不稳定"陷阱。
  • 结论结构：作者明确把这命名为 negative ablation——这是 harness 文献里少见的"先承认 baseline 更好"的论文。

核心数据亮点 / 关键论点
  • 54.72% → 50.94%（绝对 −3.78 pp）：扩展 orchestration 不仅没收益，反而把整体准确率吃掉将近 4 pp。GAIA-L1 53 题量级里这是"一题以上"的差距，远高于偶发噪声。
  • 5 个操作量同时上涨：traceback、timeout、tool-failure mention、token-line call、campaign-log cost——这些没有任何一个是预期会涨的"功能性副产物"，因此只能解释为"orchestration 自身在抢资源"。
  • 2 次 smoke = 12/20 + 11/20：方差大到一次"看起来涨"会被另一次抵消。这是论文最实用的方法学警告——20 题 smoke 不能用作 ablation 的最终证据。
  • 负向 ablation 作为 first-class 论证：报告自我标定为"the central result is therefore a negative ablation"——这在 harness 文献里仍是稀缺品类，对你写 KV-cache offload 论文里的"prefetch ablation"具有报告范式参考价值。
  • 方法学主张：bounded planner escalation / deterministic extraction / evidence reconciliation / explicit run gate 应作为可靠 agent 评测的 first-order requirements——这把 harness 的 contract 从"能跑"升到"能 audit"。
  • 样本规模透明：53 题 + 两次 20 题——作者明示样本规模上限，不掩饰。这是 negative result 论文成立的前提。
对你三个研究方向的启发
Harness Engineering（直击毕设）

ChromaFlow 给了你毕设论文里最缺的一份范式参考：negative ablation 也是结果。具体落到 DeepSeek-V3.2 KV-cache offload：现在你只有"prefetch + LRU 命中率 / per-step miss"这一条线，但从未明确报"prefetch off vs prefetch on 时 PP overlap 的失效率、async copy contention 比例、HBM 抢占次数"——这些就是 ChromaFlow 的 traceback / timeout / tool-failure 的 KV-cache 同型物。建议在 v3 实验章加一节 Operational Telemetry：除 hit-rate 外，同时报告 prefetch 引起的 stall、async copy 相对 PP 的 overlap-loss、Per-step 跨 stream 抢占——并在某些层呈现 negative ablation（即"加 prefetch 反而拖慢"）。这会让审稿人对"为什么只在某些层有效"的质疑有结构化答案，而不是 hit-rate 一笔带过。

Agent Skills Safety

把 negative ablation 思想搬到 agent safety：今日选定的 Tool Necessity / 知-行 gap 给出"模型知道要 call tool 但不去 call"的内在 gap；ChromaFlow 给出"orchestration 加多了把执行搞砸"的外在 gap。两者合起来描述同一个问题：safety 干预（如 guard 链路）也是一种 orchestration overhead，加得越多越可能在 telemetry 上表现为 retry、timeout、abort——这些信号本身就是被攻击者利用的指纹。设计 safety harness 时应同样汇报"加 guard 之后的 traceback / timeout / abort 频次"，避免把 safety 等价于"看上去拒得多"。

Safety Benchmark

给 benchmark 社区的明确启示：每个 agent benchmark 都应该自带 telemetry 字段。现在 GAIA / OSWorld / WebArena 的 leaderboard 只报 accuracy；如果要求 leaderboard 同时报 5 个操作性量（traceback / timeout / tool-failure / token-line / cost），就能把"靠 orchestration 蛮力刷 0.5 pp"的工作筛掉。今日选定的 ClawForge 在评测面上加了"是否检视 state"维度——这与 ChromaFlow 的"是否限制 planner 升级"是镜像设计：都是在 accuracy 之外把"过程行为"做成 first-class 评测量。
