一句话核心贡献
把 LLM agent 的 harness 本身从手工工程升级为可端到端学习的 plug-in 模块——双向投影（observation projection 蒸馏轨迹为决策相关状态 / action projection 把动作转译为可执行 transition 或基于轨迹的拒绝），在 Terminal-Bench 2.0 / SWE-bench Verified 上匹敌或超越专门 harness，同时显著降低 token 消耗与轨迹长度。
摘要（中文翻译，忠实原文）

大语言模型作为 agent 越来越多地被部署到长程任务中，但其表现不仅由模型能力与环境设计决定，同样取决于负责调解 agent–环境交互的 harness。现有 harness 大多是手工工程的，随着轨迹变长、交互变复杂，难以扩展。

本文提出问题：harness 是否可以由一个可端到端训练的可学习 plug-in 模块来生成？ 我们引入 HarnessBridge——一个轻量、可学习的 harness 控制器，把 agent–环境接口参数化为双向投影。HarnessBridge 学习两条双向投影：observation projection，把原始轨迹蒸馏为紧凑的、决策相关的状态；action projection，把提议的动作转换为可执行 transition 或基于轨迹的拒绝（trajectory-grounded rejections）。我们通过统一指令微调（unified instruction tuning）在一个 harness 监督数据集上训练 HarnessBridge。

在 Terminal-Bench 2.0 与 SWE-bench Verified 上，HarnessBridge 匹敌或超越强专门 harness，同时显著降低 token 消耗与轨迹长度，并能从较小的 generator 模型泛化到更大的商用模型。

核心内容解读（双向投影 / 拒绝信号 / 监督微调 / 跨规模泛化）

立论：harness 是 agent 性能的"第三轴"，但它一直在第三轴上的手工时代。过去两年 LLM agent 的提升路线被压缩为两轴：(a) 模型能力（更大、更强、更长 context）；(b) 环境设计（OSWorld / WebArena / SWE-bench / Terminal-Bench 越来越接近真实部署）。但这两轴中间还藏着一根决定性变量——harness，即"把环境的 raw observation 变成模型能理解的 prompt / 把模型输出变成可执行 action"的胶水层。Anthropic 的 Claude Code、Cognition 的 Devin、OpenAI 的 SWE Agent 各自用了不同 harness，同样的 base model 在不同 harness 上分数差距常达 10–20 pp——这是 6-08 AARR/AARRI-Bench（arXiv 2606.07462）"harness 与模型解耦评测" 的核心论据。但 AARRI 只是呼吁分离评测；HarnessBridge 更进一步——把 harness 本身从手工工程升级为可学习模块。这是 2026 H1 在 harness 工程方向上"研究方法学 → 训练范式"的关键跃迁。

方法学贡献：把 harness 拆成两条对偶的投影。HarnessBridge 的工程设计哲学非常清晰：

投影方向作用对应解决的问题
Observation Projectionenvironment → agent把原始 raw 轨迹蒸馏为紧凑、决策相关的状态——只保留对当前决策必要的信息解决"context 越来越长但有效信号被稀释"——直接对位 6-09 SKILL.nb"选择性形式化"思想
Action Projectionagent → environment把提议的动作转译为可执行 transition，或返回基于轨迹的拒绝（trajectory-grounded rejection）解决"模型生成的动作不可执行 / 触发未审计的副作用"——把 ECA / SafeHarbor 的"error-handling 是攻击面"拉回 harness 层

双向投影的本质：HarnessBridge 不试图替代 agent 的决策能力，而是夹在 agent 与环境之间充当"翻译器 + gate"。这与传统 harness 设计有三个关键差别：(a) 传统 harness 是规则驱动（如 ReAct 的 "Thought / Action / Observation" 模板），HarnessBridge 是学习驱动；(b) 传统 harness 是单向（只把 observation 变 prompt 或只把 action 变命令），HarnessBridge 是对称双向；(c) 传统 harness 无拒绝信号，HarnessBridge 引入 trajectory-grounded rejection——当一个 action 与当前轨迹不一致时，可以拒绝执行并返回理由。这第三点尤其关键，它把 5-15 ChromaFlow（arXiv 2605.14102）的"bounded planner escalation / explicit run gate"工程姿态内化到 harness 训练目标中。

核心发现一：harness 学习的是"跨模型可迁移"的接口知识。论文最重要的发现是跨规模泛化：HarnessBridge 用较小的 generator训练，可以直接迁移到更大的商用模型——这意味着 harness 学到的是"environment-side 接口规范"而非"model-side 偏好补丁"。这条性质让 HarnessBridge 与 5-23 Life-Harness（arXiv 2605.22166）形成精神同构——Life-Harness 用一条 Qwen3-4B 轨迹演化的 harness 零成本转移到 17 个其他模型 / 平均提升 88.5%；HarnessBridge 把这种"harness 的可移植性"显式立为训练目标而不是事后观察。

核心发现二：harness 学习同时降低 token 消耗与提高任务成功率。这是反直觉的——传统理解是"更复杂的 harness 通常会带来更多 prompt overhead"，因此 harness 与 token 消耗是权衡。HarnessBridge 证明这是可以同时优化的两个目标：observation projection 通过"压缩到决策相关状态"自然降低输入 token；action projection 通过"拒绝错误动作"避免无效轨迹延长。这条 finding 直接呼应 5-15 ChromaFlow 的"加 orchestration 反而变差"负面 ablation——HarnessBridge 用学习驱动的 harness 把 ChromaFlow 担心的"token 爆炸 + 轨迹失控"问题结构化解决。

哲学定位：harness 是 agent 论文的"第三作者"。论文最深远的影响可能是政治层面——过去所有 agent 论文都把 harness 当作"不可见的脚手架"，少有论文显式描述 harness 设计；HarnessBridge 把 harness 提升为可被训练、可被引用、可被对比的第三作者。这与 6-08 AARR/AARRI-Bench"LLM × harness 二轴评测" 共同推动 2026 H2 agent 论文模板的根本变化：未来 agent 论文必须显式声明 harness、评测必须 harness 与模型解耦、harness 自身必须可被训练或对比。

开源价值：可作为生产 agent harness 的 drop-in 替换。论文摘要明确支持从小生成器泛化到商用大模型——这条性质让 HarnessBridge 在工业落地上有实际意义：训练一次，部署到 GPT-5 / Claude Opus 5 / Gemini 3 等不同后端 agent 上，无需为每个商用 backend 重新工程化 harness。这条工程价值与 5-26 "From Model Scaling to System Scaling"（arXiv 2605.26112）的"系统级研究议程" 的动态 skill routing 部分高度耦合——HarnessBridge 可以作为 skill routing 层的具体实现底座。

对你三个研究方向的启发
Harness Engineering（直击毕设）

HarnessBridge 给你的 KV-cache 毕设的最直接启发是：把 cache controller 立成可学习模块的可能性。当前 KV-cache 系统层论文绝大多数是规则驱动——预取策略、驱逐策略、双水位线、hierarchical 组织都是手工工程；HarnessBridge 的双向投影设计提示了一种训练驱动 cache controller 的研究方向：(a) access projection—— 把原始 KV 访问序列蒸馏为决策相关的统计特征（局部性 / 重用距离 / 块级关联）；(b) policy projection—— 把"是否预取 / 是否驱逐 / 块在哪一级"立为可学习决策。毕设方法论建议：不必直接做学习 cache 系统（工程量过大），但在论文里显式指出"当前 cache 系统是 manually engineered，HarnessBridge 提示了 learnable cache controller 的研究方向" 作为 future work，可让你的工作具备 system × learning 的视野厚度。同时直接采用 HarnessBridge 的跨规模泛化实验设计——你的 cache 设计应该用小模型上的 trace 训练 / 调参，但在大模型上不重新调参就能受益。

Agent Skills Safety

HarnessBridge 与今日方向 #2 Containment Gap 形成对位——Containment Gap 证明 LangChain / AutoGPT / OpenAI Agents SDK 三大主流 harness 缺乏 memory integrity；HarnessBridge 提出 trajectory-grounded rejection 作为 action projection 的核心组件——这正是 Containment Gap 缺位的"policy gate"的可学习版。把两者合起来：未来 agent harness 必须同时具备 (a) trajectory-grounded rejection（学习驱动 / HarnessBridge）和 (b) memory integrity validator（规则驱动 / Containment Gap）——双层防御。这条结合也直接呼应 5-26 SafeHarbor"分层记忆 guardrail + 信息熵自演化" 与 6-08 Lean4Agent"形式化 IR for workflow" 的精神同构——三者共同把"safety 在 harness 层"立成 2026 H2 主流共识。

Safety Benchmark

HarnessBridge 给 benchmark 方向最深远的影响是评测设计的根本变化：未来 agent benchmark 必须显式声明 harness 是固定还是可学习。今日方向 #3 MLUBench"终身遗忘" 评测是这一变化的反例——MLUBench 关注的是模型权重层的能力变化；但同样的遗忘任务在不同 harness 下结果可能差距巨大（比如 observation projection 是否泄漏被遗忘内容）。建议：未来 safety benchmark 报告时必须显式声明：(a) harness 是 frozen 还是 learnable；(b) 评测是否 harness 与模型同时评估；(c) 跨 harness 鲁棒性是否被测试。这条姿态在 6-08 AARRI"LLM × harness 二轴"、6-11 SciConHarness"clean-room evaluation" 之后构成 2026 H2 评测方法学的统一姿态。HarnessBridge 动摇了"benchmark 评测的是模型能力"这一核心假设——评测的是(模型 + harness) 这一组合体的能力。
