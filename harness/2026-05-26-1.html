摘要（中文翻译，忠实原文）

本文研究 agentic AI 中的下一个主要瓶颈，即 系统扩展（system scaling），而非单一的模型扩展：围绕 foundation model 设计可审计、持久、模块化、可验证的体系结构。我们把这一转向称为 "scaling the harness"——把环绕 foundation model 的结构化执行层视为一等设计、评测和优化对象。

尽管最近的大语言模型已经让 agent 可以使用工具、检索信息、维护记忆和执行长程工作流，但当前的评测仍然以模型为中心，往往把 agent 简化为 final-task success，而把记忆、检索、工具使用、编排、验证、治理这些都当作次要实现细节。这种框架越来越不够用，因为 agent 的性能涌现自 foundation model、记忆基质、上下文构造器、技能路由层、编排循环、验证与治理层 之间的交互。它们共同构成 agent harness，把模型能力翻译成长程 agent 行为。

我们通过三个核心瓶颈研究 scaling the harness：上下文治理（context governance）、可信记忆（trustworthy memory）和动态技能路由（dynamic skill routing），以及协调和约束它们的编排和治理机制。我们进一步勾勒了 harness-level benchmark 的研究议程，超越 one-shot 任务成功，去衡量轨迹质量、记忆卫生、上下文效率、通信保真度、验证成本以及随时间的安全演化。

为使讨论具体化，我们开发了 CheetahClaws —— 一个 Python-native 参考 harness，并把它与 Claude Code、OpenClaw 进行对比。我们的核心主张是：未来 agentic AI 的进步将同样依赖系统设计，而不只是更强的 foundation model。

核心内容解读（七大组件 · 三大瓶颈 · 六大评测维度）

问题背景：2025-2026 上半年的 agent 评测有一个隐性的"模型独裁"——所有数字（SWE-bench resolution rate、AgentBench score、WebArena success）都只看 final-task success，把记忆错配、上下文越界、技能路由失败、工具调用泄露、轨迹被污染等问题统统压扁到一个标量。这种模型为中心的评测在 GPT-5.4、Claude Opus 4.6、Gemini 3.1 这种 frontier model 之间几乎已经测不出差距，性能差距实际来自 harness 而非模型——这与今日候选 #2（Y. Zhang et al. 的 "Binding Constraint Thesis"）一致。本文是对这一现象的系统性回应。

七大组件视角：论文把 agent harness 分解为 7 个一等组件——

组件角色scaling 维度

三大核心瓶颈：论文在 7 个组件中圈出三个最受瓶颈牵制的——context governance（上下文治理）、trustworthy memory（可信记忆）、dynamic skill routing（动态技能路由）。这三者的共同点是：它们都不是模型权重决定的，而是由 harness 设计者直接控制。

六大评测维度：这是论文最有 actionable 价值的部分——把 harness benchmark 从"final-task success"展开为：

  • 轨迹质量（trajectory quality）：每一步是不是无浪费、可解释、可追踪。
  • 记忆卫生（memory hygiene）：写入是否最小化、是否及时遗忘、是否抗污染。
  • 上下文效率（context efficiency）：token 预算下信息密度、prompt 重复率、冷启动成本。
  • 通信保真（communication fidelity）：跨 agent / 跨工具传递信息是否丢失、是否被歪曲。
  • 验证成本（verification cost）：每步断言所付出的额外 inference / 工具调用 / 时间。
  • 安全演化（safe evolution over time）：harness 自我演化时不退化、不引入新缺口。

CheetahClaws 参考实现：论文不止于 position-paper 层面——它给出了一个 Python-native reference harness CheetahClaws，并和两个工业 harness（Claude Code、OpenClaw）做横向对比，把抽象的 scaling 主张落到可对照的代码骨架上。这种"论文 + 参考实现 + 工业 baseline"三件套是当前 agent 系统研究的最高完整度形态，也是它能在六大维度全部高于今日其他候选的关键。

与"Stop Comparing LLM Agents"（候选 #2）的关系：Y. Zhang 等的 Binding Constraint Thesis 是这篇论文的实证基础——它通过 control-theoretic 形式化和 variance decomposition 论证 harness 引致的方差经常超过模型引致的方差，甚至会发生 ranking reversal。本文则是把那个观察做成了完整的 research agenda。两篇可以视作姊妹论文。

Harness Engineering（直击毕设）

这是迄今最完整的 harness 研究议程。它把"harness"明确分成 memory substrate / context constructor / skill-routing layer 三个最 KV-cache 友好的子系统——而你毕设的 "稀疏注意力 KV-cache 卸载 + 双水位线 GPU 热缓存 + CNN/Linear prefetcher" 工作恰好可以贴在 memory substrate 这一层。论文用"memory hygiene"作为评测维度，给了你一个绝佳的论文框架——把 LongBench-v2 上的 hit rate / bandwidth / 取胜窗口都重新包装成"harness-level memory hygiene 指标"，比在毕设答辩里只讲"系统优化"高一个抽象层。Q.md 评审里对"为什么不直接看 latency"的质疑，也可以反引这篇论文的"final-task success 不再够用"作为 first principle 答辩弹药。

Agent Skills Safety

论文的"safe evolution over time"维度直接命中 Agent Skills Safety 的核心问题——技能、记忆、context 的演化往往悄悄越过 guardrail，且单步指标看不出来。把昨日 #1 的 SafeHarbor（分层 guardrail）+ 昨日 #2 的 TTT-Undermines-Safety 放进 CheetahClaws 的 governance layer 来评估，可以构成一个 self-contained "safety-aware harness"原型——这是当前安全方向最缺的产物形态。

Safety Benchmark

"6 大评测维度" 已经构成完整 benchmark 论文的骨架。把每一维度展开为 1～2 个具体测试集（trajectory quality 用 SWE-bench 的轨迹标注、memory hygiene 用注入式污染、context efficiency 用 token utility curve 等等），即可造出一个"harness-level safety benchmark"。这恰好补上 5-25 #2 PoisonForge 这种"单维 attack benchmark"之外的系统级评测空白。
