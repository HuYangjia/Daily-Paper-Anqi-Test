摘要（中文翻译，忠实原文）

基于 LLM 的 self-evolving agent 主要通过改变 agent harness（围绕基础模型的结构化执行层，包括 prompts、memory、tools、middleware、runtime state 与模型-工具交互循环）来获得提升。然而现有评估常将这一过程简化为孤立的任务分数或单一序列曲线，掩盖了一次更新到底是产生可复用的提升、过拟合最近任务、增加成本，还是损害较旧行为。

我们提出 SEAGym，一个跨 training / validation / test / replay / cost 记录测量 agent harness 更新的评估环境。SEAGym 把 Harbor-compatible 基准转化为带有 train batches、frozen update-validation、held-out ID 与 OOD transfer 视图、replay 诊断以及 snapshot 与 metric 记录的动态自进化任务源。在 Terminal-Bench 2.0 与 HLE 上，我们以共享 epoch/batch 协议比较 ACE、TF-GRPO 与 AHE。结果显示，这些评估视图提供了关于演化过程的互补信号：频繁更新可能无法提升 held-out 性能，有用中间快照可能后期崩溃，source diversity 与 model backend 也会影响 harness 可靠性。

核心内容解读（5 视图评测面 / Harbor 兼容性 / 三方法对比 / 三个反直觉发现）

立论：self-evolving harness 必须按"过程"而非"终点"评估。过去半年（2026 H1）连续多篇论文已经把 harness 立为 first-class research object：5-23 Life-Harness 用 runtime 接口替代训练；6-12 HarnessBridge 把 harness 立为可学习模块；6-14 ToolSense 把 harness 立为审计工具；6-15 HarnessX 把 harness 立为可演化的铸造厂。SEAGym 是这条线的评测方法学闭环——它问的是：当我们说一个 harness 在演化中"变好了"，到底变好的是什么？仅看 final task score 既无法区分"真泛化" vs "过拟合最近任务"，也无法发现"提升 A 任务但回退 B 任务"的 seesaw，更看不到"中间快照很好但最终崩溃"的过程级失败。论文给出的回答是5 视图正交评测面 + 成本记录。

系统设计一：5 视图正交评测面（Table 1）。SEAGym 将基础数据集切分（D_train / D_val / D_test）物化为 5 个评估视图，每个视图测量 self-evolving 过程的不同方面：

#视图名池/来源衡量内容
1Update Evidence（训练证据）B_t ⊂ D_train哪些轨迹与可见反馈驱动下一次 harness 更新
2Process Assessment（过程评估 / Update-Validation）V_update-val ⊂ D_val冻结的中间快照是否在不把验证用作更新证据下持续提升
3ID Transfer（同分布迁移）V_ID ⊂ D_test更新是否能迁移到来自源分布的未见任务
4OOD Transfer（分布外迁移）V_OOD ⊂ D_test更新是否能迁移到偏移或目标域任务
5Replay / Retention（回放/保留）V_replay更新后的 harness 是否保留或恢复早期行为（衡量遗忘与回归）

另外配套成本记录（token 用量 / 工具调用次数 / wall-clock 时间），与主指标分离呈现。关键派生指标：UVG（Update-Validation Gain）= 当前快照 − 上一/初始快照在 V_update-val 上的性能差；IDG / OODG = 最终快照相对初始快照在 ID / OOD 上的提升；FR（Forgetting Rate）= max(0, Perf(A₀, D_R) − Perf(A_T, D_R))。这套 5 视图把"是否真在进步"分解为5 个互不可替代的子问题——任何单一视图都不充分。

系统设计二：Harbor 兼容性 = 任务执行与评估调度分离。SEAGym 不重写基准定义，而是复用 Harbor Framework（容器化任务执行 + verifier + 并行作业）作为底层运行器。两者职责正交：Harbor 管"任务怎么跑"，SEAGym 管"什么时候跑、怎么记录、怎么验证"。集成新基准 → 写 Harbor task/adapter 模板；集成新方法 → 用 thin wrapper 实现 rollout/update 接口，保留方法的 native update 语义。任意 Harbor-compatible 基准都可被无侵入地转化为动态自进化任务源——这种"评测调度层 / 任务执行层" 分离是 5-15 #1 ChromaFlow"operational 量与 accuracy 联合报告" / 6-08 #3 AARRI"LLM × harness 二轴评测" / 6-15 #1 HarnessX"9 维分类法" 之后又一次方法学升级。

系统设计三：ACE / TF-GRPO / AHE 三方法在共享协议下首次公平比较。论文实例化于 Terminal-Bench 2.0 + HLE：source train 80 任务（Terminal-Bench 2.0 + HLE Math/Physics）/ source validation 35 / source test ID 55 / OOD test 80（HLE CS/AI/Engineering）；5 epoch × batch 20，主对比模型 DeepSeek-V4-Flash。三方法的更新对象与机制差异：

方法更新对象UVG（验证）IDG（同分布）OODG（分布外）Rollout token/任务
AHE（Agentic Harness Engineering）完整 harness（prompts/tools/middleware/project files）+17.1+9.1+6.31.46M
ACE（Agentic Context Engineering）持久 skill/context 库（13 个活跃技能）+2.9+3.6+2.51.93M
TF-GRPO（Training-Free GRPO）任务族经验存储 / grouped rollout 证据+17.1+3.6−2.52.33M

核心实证一：AHE 三视图同时为正，TF-GRPO 暴露源分布过拟合。AHE 是唯一同时在 validation / ID / OOD 三视图都为正的方法（编辑范围最大，主结果最佳但有过程级风险）；TF-GRPO 在源分布表现强（UVG +17.1 与 AHE 持平）但 OOD 反而 −2.5——典型"看似在进步、实则过拟合最近任务" 的失败模式。这种区分能力是任何单一 final score 视图都无法提供的。

核心实证二：中间快照可能后期 collapse（AHE 训练遗忘案例）。AHE 最终 agent 解决 43/80 train-replay 任务（初始 34/80），但第 4 epoch 后回放性能崩溃至 6/80，最后一个 epoch 才恢复；修复 13 个、遗忘 4 个，净增 9 个任务。如果只看 final score，会错过这个中间灾难性回归——而 SEAGym 的 process assessment + replay 视图能直接看见。

核心实证三：batch size 非单调（甜点效应）。同样 AHE 设置下，batch=10 → 验证 −14.3 / ID −14.5；batch=20 → 验证 +17.1 / ID +9.1（最佳）；batch=40 → +2.9 / +1.8；batch=80 → 验证 −17.1 / ID −16.4。这种非单调甜点效应在"scaling = 越大越好" 的常识下是反直觉的——SEAGym 把这个发现明示出来。

核心实证四：Source diversity 与 backend 显著影响可靠性。HLE-only 训练在最终快照 collapse 至 0%，混合源（Terminal-Bench + HLE）能恢复；DeepSeek-evolved harness 改善 GLM-5.1（+7.3）但损害 GPT-5.4（−3.6）；GPT-5.4 自演化 harness 反而 OOD −7.5——ID gain 不蕴含 OOD gain。这条结论与 6-12 HarnessBridge"从小生成器泛化到大商用" / 6-15 HarnessX"cross-backend 协同进化" 形成对偶——前两篇推 harness×model 协同进化，SEAGym 警示协同进化的可迁移性边界。

方法学姿态：harness 演化与 RL 训练共享评测面。SEAGym 把持续学习经典框架（adaptation / transfer / retention / forgetting）从参数学习扩展到agent-system learning，与 6-12 HarnessBridge / 6-15 HarnessX 把 harness 演化形式化为符号空间 MDP 形成"形式化 + 评测化"双姿态。2026 H1 后期 harness 工程的统一姿态已经清晰：(1) harness 是类型化代数对象（HarnessX D1-D9）；(2) harness 演化是 RL 同构（HarnessX AEGIS）；(3) harness 必须可被审计（ToolSense）；(4) harness 演化必须多视图评测（SEAGym）；(5) harness 与 model 必须协同进化（HarnessX co-evolution）。SEAGym 补齐了这套姿态中"怎么知道演化真的好了" 这一块。

对你三个研究方向的启发
Harness Engineering（直击毕设）

SEAGym 给你的 KV-cache 系统层毕设最直接的启发是：把 cache 系统的"策略改进" 也立成 5 视图评测对象。当前 KV-cache 论文常见的失败姿态是"报一个 final hit-rate / latency 数字"——但 SEAGym 的方法论告诉你：cache 策略改进必须分别报告 (a) 训练 workload 上的提升、(b) 验证 workload 上的提升、(c) ID 类似 workload 的迁移、(d) OOD 不同 workload 的迁移、(e) 旧 workload 的回归。毕设方法论建议：(1) 借鉴 5 视图——为你的 cache 系统设计 train/val/test_ID/test_OOD/replay 5 视图评测协议；(2) 借鉴 cost record 与主指标分离——hit-rate / latency 单独报，token / memory / energy 成本单独报；(3) 借鉴非单调甜点效应 警示——cache 参数（batch / cache-size / eviction-threshold）也必须做4 档对比 而非线性扫描；(4) 借鉴 backend 不对称迁移——cache 策略在不同 model size 上效果可能反向，必须分别报告。论文写作姿态：把 SEAGym 列为"多视图评估范式" 的方法学先例——你的 cache 系统改进必须经得住"过程 + 终点 + 迁移 + 保留 + 成本" 五重审视。

Agent Skills Safety

SEAGym 的 5 视图评测面与今日方向 #2 Rift（conflict signature for deception）在方法论上同构——两者都把"看似 OK 的最终行为"分解为多个独立可测维度。SEAGym 用 process / ID / OOD / replay 区分"真泛化 vs 过拟合最近任务"；Rift 用 residual rank 区分"真知识缺失 vs 知识冲突"。safety 方向的关键启发：未来 agent skills safety 评测必须借鉴 SEAGym 的 5 视图——(a) 训练时 alignment 任务上的拒绝率；(b) 冻结快照的验证集拒绝率（防止训练时漂移）；(c) 同分布 ID transfer（同类风险新样本）；(d) OOD transfer（新风险类别）；(e) replay（早期 alignment 行为是否被 catastrophically 遗忘）。SEAGym 揭示的"中间快照崩溃" 现象在 safety 上对应"training 中 alignment tax 突然飙升" 的隐患——必须有 process record 才能发现。

Safety Benchmark

SEAGym 与今日方向 #3 EComAgentBench 在 benchmark 设计姿态上形成"纵向 + 横向" 双轴——SEAGym 是"纵向"评测（同一基准上跨 epoch / snapshot 评估演化过程），EComAgentBench 是"横向"评测（同一时刻跨 7 模型 × 3 信息源评估隐藏意图恢复）。统一启发：未来 safety benchmark 论文应同时具备 (a) 纵向视图（snapshot 时序 / replay 退化）；(b) 横向视图（多模型 / 多 backend / 多基准）；(c) 来源视图（信息可见 / 信息隐藏）。SEAGym 揭示的"frequent updates may fail to improve held-out" 现象与 6-15 #3 WorkBench Revisited"capability 与 safety 同向提升" 的反共识一起，把 benchmark 设计从"单时刻一次评测" 推向"纵向多时刻评测 + 横向多源评测" 时代。SEAGym 提议"任意 Harbor-compatible 基准都可被转化" 这条 plumbing 路径，是 safety benchmark 工程化推广的一个关键 enabler——safety benchmark 也应能直接借此挂载到 SEAGym 框架做纵向追踪。
