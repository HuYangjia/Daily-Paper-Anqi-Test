摘要（中文翻译，忠实原文）

工具集成的 LLM agent 通常被一个 harness 包裹：harness 决定哪些工具被暴露、它们如何被描述、每一步观察附带什么辅助信息。虽然 agent 本身已经被常规地 post-training，但这一层 scaffolding 一直被视为固定的工程细节，其设计努力被局限在训练-free 的设定中。此外，现有 post-training 算法都默认环境是静态 的——而部署时的工具环境与任务往往会发生迁移。为弥合此差距，我们扩展了 ALFWorld：(i) 把 harness 视为可控设计维度；(ii) 支持任务与工具环境迁移 下的评测。在此之上，我们系统分析 harness 设计如何在 ID 与 OOD 两种设定下影响 post-training。我们实证显示：harness-aware 的 post-training 不仅提升 ID 性能，也让 agent稳健地适应 OOD。在 minimal-design 的 harness 下，post-training 在更强的工具环境迁移下会经历剧烈的性能下降，进一步突显 harness-aware post-training 在此类迁移下的必要性。

核心内容解读（harness 的训练化 / ALFWorld-Shift / 三层论证 / 与既有工作的根本区别）

立论：harness ≠ 工程附属物 — harness 是 post-training 算法看到的"事实分布"。2024–2026 H1 的 harness 文献链条已经把 harness 描述为"感知—决策—行动—验证" 四元运行时的工程化承载（4-28 综述 / 5-25 Code as Agent Harness / 6-12 HarnessBridge / 6-15 HarnessX / 6-23 AOHP）；但所有这些工作都隐含一个前提：训练一个 agent 模型，然后把它放进某个 harness 里跑。POSTECH 团队的关键观察是：这个前提在工具环境与任务都会迁移的真实部署里是错的。因为 harness 决定了 agent 在训练阶段实际看到的观察分布、工具描述分布、辅助信息密度——这意味着 harness 是训练时的 covariate shift 控制器。如果训练时 harness 把工具描述写得很全、把观察压得很干净，那训练得到的 agent 在稀疏 harness 下就会失去其推理依据。反之，极简 harness 训练出的 agent 虽然在 ID 上看起来还行，但一旦工具环境迁移，它没有任何 harness 提供的语义脚手架 可以挽救它。

实验平台：ALFWorld-Shift——同时控制 harness × shift 两个维度。论文对经典 text-game agent benchmark ALFWorld（Shridhar et al. 2021）做了两项扩展：(1) 把 harness 暴露为可控设计变量——具体来说，可以独立切换 tools-exposed（哪些 primitive 暴露）、tool-description-style（natural language vs 结构化）、per-step auxiliary info（是否提供 inventory / room-state / hint）等子维度；(2) 支持两种 OOD 评测：task shift（训练所未见的目标类别 / 物体组合）与 tool environment shift（同一任务但工具命名、组合、调用语法发生迁移）。这是 ALFWorld 第一次被赋予训练—评测的环境位移控制能力——它把"environment is static" 这一现有 post-training 算法的标准假设直接暴露在显微镜下。

核心实证 ① harness-aware post-training 在 ID 与 OOD 同时占优。论文在多种 harness 配置上跑 post-training，把 harness 设计与 post-training 视为联合优化对象，并与"固定 harness + 单独 post-training" 的传统范式对比。结果是清晰的：当 post-training 把 harness 的差异 explicit 地纳入训练目标（如 harness-conditioned 的 reward / sample sampling），同一个底层 LLM 在 ID 上仍然占优，而在 task shift 与 tool shift 两种 OOD 上的优势更显著。换言之，"harness 不参与训练" 的传统范式付出了双重代价：训练里没看到的 harness 变体，部署时无法泛化；部署里的环境迁移，训练里没建模过。

核心实证 ② 极简 harness × 强工具迁移：断崖式性能崩塌。论文最尖锐的发现，是极简 harness（minimal-design：少工具暴露、简短工具描述、稀疏辅助信息）下的 post-training 在弱工具迁移上还勉强保留性能，但在强工具迁移（工具命名/调用语法/组合方式大幅改变）下出现剧烈的性能下降。这是 harness 工程谱系第一次拿到的"反例数据"——它直接反驳了"训练一个足够强的 model，harness 随便配" 的乐观叙事。这个发现对 6-15 HarnessX 与 6-23 AOHP 的工程线条都构成补充约束：HarnessX 的 9 维分类 + Substitution Algebra 不仅是组合自由度，更是训练—部署 covariate；AOHP 的 capability descriptor 不仅是OS 抽象，更是训练时的标签维度。

方法学姿态：harness 工程进入"training-aware" 阶段。从 4-28 综述把 harness 立为统一研究对象，到 6-15 HarnessX 把 harness 算子化，到 6-23 AOHP 把 harness 下沉到 OS——所有这些工作都假设训练算法是给定的、不变的。Interplay 是第一篇明确把训练算法 与 harness 设计 视为耦合优化问题 的论文。它意味着 2026 H2 的 harness 工程线条会出现新的子方向：harness-conditioned post-training（如把 harness 描述作为额外 context；把 harness 配置作为 RL 中的 sampling 维度；把 harness 变体当作 distributional augmentation）。从 KV-cache / 推理系统的视角，这正是系统层设计变量进入训练循环 的早期信号——也是你做 SGLang / vLLM 系统毕设时需要关注的训练化协同 视角。

位置：把 harness 工程的"三联拍" 升级为"四联拍"。06-23 三篇报告组合成"OS 降维（AOHP）+ 表征下沉（AgentLens）+ 评测升维（Benchmark Illusion）" 的方法学三联拍；今日的 Interplay 给出了第四拍——训练循环联动。意味着 harness 概念第一次同时跨过 4 个层次：OS / runtime / 评测 / 训练，构成完整的"harness 工程立方体"。

Interplay 是 harness 工程链条上第一篇正面打入 training-aware harness 的论文。对你 KV-cache / 推理系统毕设的启发：(1) harness × post-training 耦合 思想可直接迁移到 KV-cache—— cache 策略（eviction / placement / prefetch）不应该被独立训练后塞进系统，而应该作为训练时的 covariate 暴露给上层 reasoning loop；(2) covariate shift 控制器 思想可套到 KV-cache 系统层—— 不同 sparse pattern / DSA index granularity 是训练分布的关键变量，必须在训练时显式标注；(3) 极简 harness × 强迁移崩塌 反例提示你 KV-cache 系统的"极简 API" 倾向（如只暴露 read/write 两个原语）在 deploy-time 工具/算子迁移下会有同款断崖；(4) ALFWorld-Shift 的"同时控制设计变量 + 评测迁移" 方法论可作为系统层 KV-cache benchmark 的设计模板—— sparse pattern × workload shift；(5) 对 SGLang / vLLM 开源贡献，这一线索意味着harness-aware kernel（让 attention kernel 看到 harness 的描述维度）是新的研究空白。

Agent Skills Safety

Interplay 间接揭示了 agent safety 的训练化 视角：如果 harness 决定了 post-training 看到的事实分布，那"safety property" 也必须 harness-aware。统一启发：(1) safety 不再只是 post-training 的对齐目标，而是harness × post-training 联合优化的输出；(2) 极简 harness × 强工具迁移 → 崩塌这一反例同样适用于 agent safety—— 当工具环境迁移时，safety 行为会出现"分布外失败"（与今日 #2 SkillSafetyBench 的 skill-mediated 失败模式 mutually reinforcing）；(3) tool-description-style 作为可控 harness 子维度——意味着 prompt-injection / RAG poisoning 等攻击面也可以视为"对 harness 维度的恶意输入"，这给攻防研究提供了形式化抓手；(4) auxiliary info density 子维度对应 safety alignment 中的"有多少 contextual constraint 被显式 surfaced"——可视为 contextual integrity（6-15 MINIM / 6-23 AOHP）的训练对偶。

Safety Benchmark

Interplay 给 Safety Benchmark 直接示范了"设计变量 × 迁移变量" 的二维评测矩阵——这正是 Safety Benchmark 缺失已久的对照实验设计。统一启发：(1) 把 ALFWorld-Shift 范式平移到 safety 评测——同一条 jailbreak / refusal-bypass 样本在不同 harness 配置 × 不同 task/tool shift 下的 ASR 矩阵；(2) 断崖式崩塌反例 给 safety benchmark 提供了一种"压力测试" 范式——不只是测 ID safety rate，而是要测harness 极简 + 强迁移 时的安全行为持久性；(3) 与今日 #3 Hacker-Fixer Loop 形成姊妹—— Interplay 表明 harness 是训练分布的控制器，Hacker-Fixer 表明 verifier 也是训练分布的控制器，两篇共同构成"benchmark 既是评测也是训练分布" 的方法学双门；(4) Safety benchmark 不应该只 report 单一 ASR，而应 report"harness × shift" 双轴热力图—— 这与 6-23 Benchmark Illusion"评测意识是多变量"的姊妹论述 mutually reinforcing。
