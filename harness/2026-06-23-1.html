摘要（中文翻译，忠实原文）

AI agent 正在驱动一种新的软件范式：它可以自主调用工具、提取信息、管理记忆，并完成横跨应用与数据源 的任务。然而现有的终端用户操作系统主要为application-centric 工作流 设计，对 AI agent 几乎没有原生支持。这种不匹配限制了 agent 的更广泛采用，并在传统系统上运行 agent 时带来执行开销和安全风险。虽然 agent-native 操作系统 的概念正在浮现，研究社区仍然缺乏一个开放的 testbed 来探索 agent-mediated 交互所需要的架构原语。我们提出 AOHP（Android Open Harness Project）——一个构建在 Android Open Source Project（AOSP）之上的 OS 级 agent harness。AOHP 的核心设计原则是把 agent 视为 OS 一等 actor，从而支持自适应用户界面与 agent 友好的运行时环境。AOHP 在保留成熟 Android 软硬件生态的前提下，引入三种 agent-oriented 系统机制：个性化服务编排（personalized service composition）、高效 agent 接口（efficient agent interfaces）、安全信息流（secure information flow）。在覆盖 OS agent 关键能力的若干挑战性任务上，AOHP 展示出明确的优势：任务完成率 +21.12%、执行成本 −51.55% token cost、安全策略合规性提升。

核心内容解读（OS 一等 actor / 三大系统机制 / 与 application-layer harness 的本质区别）

立论：agent runtime ≠ application runtime——必须重新设计 OS 抽象。2024–2026 H1 的 harness 工程链条已经证明，agent 的"感知—决策—行动—验证"四元运行时 与传统 application 进程的本质区别越来越大：application 是事件驱动 / 单用户 / 持有 UI 焦点 / 资源边界由 sandbox 静态划定；而 agent 是任务驱动 / 跨应用 / 无 UI 焦点也能行动 / 资源边界由意图与上下文动态决定。AOHP 团队的关键观察是：把 agent 强行装进 application sandbox（如把 Android Accessibility Service 当成 agent 接口），无论 prompt 工程做得多好，token 消耗、延迟、安全风险都是 OS 抽象错配的结构性代价。他们的解决路径不是再造一个 OS，而是沿用 AOSP 的 Linux + Binder + Activity Manager + System Server 全栈，但在其上"开 agent 通道"——把 agent 升格为 OS 一等 actor，与 application 进程平级，享受 OS 级调度、IPC 与权限托管。

机制一：个性化服务编排（Personalized Service Composition）。传统 Android 是"app 提供 service，user 通过 UI 主动调用"。AOHP 把这个关系反转为"OS 知道 user 的目标和上下文，动态从 N 个 app 暴露的服务里编排一个最匹配的子集"——核心是把 service 抽象从 Manifest 静态注册 升级为语义化、可发现、可组合。具体设计上：(1) 服务清单从 Manifest XML 扩展为 capability descriptor（含语义标签、输入输出类型、副作用集合、SLA）；(2) System Server 新增 capability registry + 编排器，按 user profile 与当前任务上下文产出 service composition；(3) UI Framework 暴露 adaptive surface，让 agent 可以不打开 app 也能调用其暴露的 capability，必要时再唤起 UI。这条机制等价于在 OS 层落地了 MCP（Model Context Protocol）——但与单纯 MCP 不同的是，它绑定了 Android 的进程模型、权限模型与 lifecycle，使 capability 调用具有 OS 级真实性。

机制二：高效 agent 接口（Efficient Agent Interfaces）。当前 mobile agent（如 AppAgent / Mobile-Agent / OS-Atlas）几乎全部走 Accessibility Service + 屏幕截图 + UI tree 解析路线，token 消耗主因有三个：(1) UI tree 的冗余（一帧屏幕往往 100+ 节点）；(2) 多帧重复（agent 每一步都要"看一次屏幕"）；(3) 多模态视觉 token 价格。AOHP 在 OS 层提供三类"agent 友好接口"，对应消除这三类冗余：(1) 语义压缩 UI 流——OS 主动剔除不可交互节点、合并重复区域、按 agent 当前焦点 prune；(2) 事件订阅式接口——agent 不再每步全量拉取，而是订阅 view hierarchy diff；(3) 结构化 capability 调用回执——一旦走机制一的 capability 通道，agent 完全绕过 UI 渲染。这三条接口合起来直接对应了 −51.55% 的 token 消耗下降——这是 application-layer harness 永远做不到的，因为 UI tree 冗余是 application sandbox 模型的固有特性。

机制三：安全信息流（Secure Information Flow）。application-layer agent 的安全模型本质上是"把整个 user 都暴露给一个进程"——agent 一旦拿到 Accessibility 权限，几乎可以观察并操纵全屏；一旦拿到 storage 权限，可以扫描全部文件。AOHP 引入 OS 级 information flow 控制：(1) capability descriptor 标注每个 service 的敏感度类别（PII / 财务 / 健康 / 通信）；(2) 引入意图—容量绑定——agent 必须在 capability 调用时声明意图，OS 校验意图与 capability 的敏感度兼容性；(3) 跨应用流监督——OS 持续追踪敏感字段在 agent 调用图上的流向，禁止"从 health app 流向 finance app 再流向网络" 这类违反上下文完整性（contextual integrity）的链路；(4) 与 Android 现有 permission model 双层叠加（不替换，而是叠在上层）。这条机制天然吸收了 6-15 MINIM（client-side CI）与 6-22 AgentCIBench 的姿态——但 AOHP 把它从 application 端的挡门员 升级为 OS 端的河道与堤坝。

实证：OS 关键能力评测同时拿到 completion / cost / safety 三角增益。论文在覆盖 OS agent 关键能力的多项挑战性任务上初步实验。相比 application-layer 基线，AOHP 实现：(1) 任务完成率 +21.12%（直接来自机制一让 agent 不必"找 app—开 app—点 UI" 而是直接 capability 调用）；(2) token cost −51.55%（来自机制二让 UI 流变窄）；(3) 安全策略合规 提升（来自机制三让违反 CI 的信息流被 OS 阻断）。三角同时增益意味着 AOHP 没有走"用更多 token 换 completion" 的常见 trade-off，而是在系统抽象错配处释放了真正的"免费午餐"。

方法学姿态：harness 概念的"降维" 完成。从 4-28 综述把 harness 立为可统一研究对象，到 5-25 综述把 code 视为 harness 基质，到 6-12 HarnessBridge / 6-15 HarnessX 把 harness 算子化、铸造化，到 6-17 SEAGym 把 harness 加上 5 视图评测面，再到 6-18 Xcientist 把 harness 升维到科研生命周期——AOHP 在系统层 做了反方向的"降维"：harness 也可以是操作系统的子系统。这意味着 2026 H2 的 harness 工程将出现"降维（OS）—平面（agent runtime）—升维（科研生命周期）" 三段式产业链。开源 testbed 属性 + AOSP 基础：意味着任何研究者都可以在 AOHP 之上做 OS-level agent 实验（如 agent IPC 调度、agent 文件系统、agent 网络栈），社区门槛被显著降低。

Harness Engineering（直击毕设）

AOHP 是 2026 年 harness 工程链条上罕见的"系统层论文"——把 harness 从语言/agent 抽象下沉到 OS 子系统。对你 KV-cache 系统层毕设的启发：(1) capability descriptor 思想可直接迁移到 KV-cache—— eviction / placement / prefetcher policy 不再是隐藏的 hyperparameter，而是显式 capability，registry 化、可发现、可组合；(2) efficient agent interfaces 思想可直接套到推理引擎—— 让 attention 内核与 cache 系统之间走"语义压缩 + diff 订阅" 接口（如 DSA 索引器只暴露稀疏注意力差分 region 而非完整 KV 维度）；(3) secure information flow 思想给 multi-tenant LLM serving 提供 OS 级隔离的方法学样板—— 不同 tenant 的 KV-cache 在 information-flow 层被显式分隔；(4) AOHP 的三角增益（completion / cost / safety 同时正）是 harness 论文里最理想的系统层免费午餐 形态——你可以在 cache 论文里直接复用这种"三轴非 trade-off" 的叙事框架；(5) cs.OS 分类 首次出现 harness 关键词——意味着 harness 论文未来可以从 cs.AI / cs.LG 投往 OSDI / SOSP / EuroSys / ATC，这对你"系统层 KV-cache" 毕设是非常重要的会议路径 信号。

Agent Skills Safety

AOHP 的secure information flow 是 application-layer agent safety（如今日 #2 AgentLens、6-15 MINIM）的OS 化——前两者把 CI 校验放在 agent 内部或 application proxy，AOHP 把它升到 OS。统一启发：未来 agent skills safety 形成三层防御纵深—— (a) 模型内 安全（today #2 AgentLens 的 mechanistic subspace + 6-18 SRP 的反思预训练）；(b) application 端 CI（6-15 MINIM + 6-22 AgentCIBench）；(c) OS 端 信息流（AOHP）。AOHP 的capability 敏感度标注 思想可推广到 skill 层：skill 自身标注其 PII / 健康 / 金融敏感度类别，运行时 OS 校验意图—skill 兼容性。AOHP 的意图-容量绑定 在 skill 场景下变成"意图—skill 绑定"——agent 在调用 skill 时必须声明 user-facing 意图，OS（或 harness）校验意图与 skill 副作用集合的兼容性。这与 6-21 Confidence Laundering"潜在不确定性需要 carrier" 形成互补——AOHP 给的是 carrier 的 OS 形态：information flow tag。

Safety Benchmark

AOHP 论文目前只做了"OS agent 关键能力评测"（preliminary experiments），未给出公开 benchmark——这给 Safety Benchmark 方向留下两个清晰机会：(1) OS-level agent benchmark——把 6-17 EComAgentBench / 6-22 EnterpriseClawBench / 6-22 MacAgentBench / AgentCIBench 这一支应用层任务族同 capability descriptor 重新组织，让 benchmark 可以同时测试 application-layer 与 OS-layer harness 上的同一任务；(2) 三角增益 benchmark——专门为 completion / cost / safety 三轴同时 设计的评测面（参考 6-17 SEAGym 的 5 视图扩展为 6 视图：train / val / ID / OOD / replay / cost-safety frontier）。AOHP 的实验报告方式（同时报告 +21.12% / −51.55% / 安全合规）就是三轴帕累托前沿 的论文级范式，可成为 benchmark 领域新的 reporting standard。与今日 #3 "Benchmark Illusion"（评测意识不是单一能力，部署安全是多变量 而非单数字）形成互文——AOHP 给出了"系统层架构改良带来的真正进步" 的样本，恰好填补今日 #3 警示的"评测条件下的合规性是乐观上界" 的反面证据需求：当 cost 直接降一半时，agent 没有机会做 evaluation gaming，因为收益是结构性的。
