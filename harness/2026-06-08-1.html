首次把 Lean4 依赖类型语言变成 agent workflow / trajectory 的可机械验证 IR——形式化保障从"跑得通"升级为"类型检查通过"。
摘要（中文翻译，忠实原文）

赋予大语言模型以执行可靠多步工作流的能力，已成为 AI 的核心挑战。尽管近期 agent 能力快速跃迁，多数 agent 系统仍然缺乏形式化方法来规约、验证与调试自身的工作流和执行轨迹。我们的研究受到一个数学领域类比的启发：自然语言（NL）固有的歧义催生了形式语言（FL）的发展。

本文提出 Lean4Agent——据我们所知，这是第一个以 Lean4 这种依赖类型形式语言来对 agent 行为进行建模与验证的框架。它的两大组件是：(1) FormalAgentLib：一个可扩展的 Lean4 库，支持在显式假设下对 agent 工作流进行语义一致性的形式化建模与验证，并通过轨迹定位执行时故障；(2) LeanEvolve：在 FormalAgentLib 之上利用验证结果自动修订工作流，进一步提升 agent 能力。

在 SWE-Bench-Verified 困难子集与 ELAIP-Bench 子集上，对 5 个领先 LLM 的实验显示：通过验证的工作流相对未通过验证的工作流，平均提升 +11.94%；LeanEvolve 在 SWE 上再额外提升 +7.47%。这一结果不仅证明了形式化验证对 agent 行为的实质性增益，也为"用具有强表达力的依赖类型形式语言机械化建模和验证 agent 行为"这条新研究路线奠定了基础。

核心内容解读（FormalAgentLib / LeanEvolve / 与 SkillDAG 的差异 / 与 ANNEAL 的对照）

立论：把 agent 行为送进证明检查器。过去一年（2025-2026）harness 工程的主旋律是"把 IR 抬高一档"——Code as Agent Harness 把 IR 抬到代码、SkillDAG 把 IR 抬到类型化有向图、A Policy-Driven Runtime Layer 把 IR 抬到 observe/score/predict/act 四原语。Lean4Agent 把 IR 再抬一档——直接抬到依赖类型的形式证明系统。核心赌注：agent workflow 不仅是过程（process），也可以是命题（proposition）——前置条件、不变式、后置条件都能写成 Lean4 类型，trajectory 是这个类型的"居民"（inhabitant）。

FormalAgentLib：从规约到验证的可复用证明库。第一个组件是一个可扩展的 Lean4 库，作用如下表：

能力语义下游用法
显式假设下的工作流建模用 Lean4 类型刻画前置条件 / 不变式 / 后置条件把"这条 plan 是否合理"变成 type-checking 问题
语义一致性验证验证 plan 的子步骤组合是否仍满足全局规约排除"看似合理但条件不闭包"的虚假 plan
执行时故障定位当 trajectory 偏离规约，从证明义务（proof obligations）反推故障节点给出可机械验证的 root cause 而不是事后总结

这意味着"调试 agent 失误"第一次有了和形式化软件验证完全相同的工具链：plan failure 不再是日志里若隐若现的字符串，而是没有被居民填满的命题——找到那个洞，就是找到了 root cause。

LeanEvolve：让验证结果反哺 plan 改写。第二个组件 LeanEvolve 把 FormalAgentLib 的验证报告作为信号源，自动修订工作流——这是一种"结构性 self-improvement"：不是在 prompt 里写"请你检查并修改"，而是把"类型不通过"作为编译期错误反馈给 plan 重写器，逼迫候选 plan 闭包到一个可证明对的形态。在 SWE-Bench-Verified 困难子集上，LeanEvolve 在已经经过验证的 baseline 之上再补 +7.47 pp，说明形式验证既是过滤器、又是教师——这是与 5-19 ANNEAL（governed 符号补丁 + canary）相同审稿人偏好的连续演进：把"修"挂上"外部凭证"。

与 SkillDAG / Code as Agent Harness 的差异。SkillDAG 让 agent 在执行时调用结构化检索（vector + typed-edge + conflict 三路信号）；Code as Harness 把代码立成 IR；Lean4Agent 则把命题与证明立成 IR。差异在于"谁是裁判"：SkillDAG 的裁判是历史执行回放，Code as Harness 的裁判是 Python 解释器，Lean4Agent 的裁判是 Lean4 kernel。三者一旦组合：用代码作为执行物，用类型化图作为检索物，用 Lean4 类型作为可证明物——一条三档梯度的"从 token 到证明"IR 链路就成形了。这正是当前 harness 工程"越证越硬"的方向。

实验信号怎么读。论文的核心数据有两条：第一，"过验证的 plan" vs "没过验证的 plan"，平均成功率差 +11.94 pp——这是验证作为过滤器的价值。第二，LeanEvolve 进一步 +7.47 pp——这是验证作为反馈教师的边际价值。两者相加是 19.4 pp 的总体抬升，且来自同一类型化 IR 的两次复用，是少数能把"形式方法是否对 agent 实际有用"这一长期争议落地为正面数据的工作之一。SWE-Bench-Verified 困难子集与 ELAIP-Bench 都是有人盯着的硬测试床，这条收益不是温室增益。

研究路线意义。过去三十年形式化软件验证主要服务"程序员写出的程序"，而 Lean4Agent 把它的客户换成了"LLM 写出的工作流"——这是一次方法论降维：当 LLM 把 plan 当作 program 输出，program 验证学的全部基础设施都自动适用。后续两到三年，能否在 agent runtime 里嵌一个"常驻 Lean4 kernel"用作每一步 plan 的 type-checker，将决定 agentic 系统能不能拥有真正可证明的安全/正确性边界——而不是靠"多采样 + 投票"摘取经验主义的低果。

核心数据亮点 / 关键论点
  • +11.94 pp（5 模型平均）——通过 FormalAgentLib 验证的工作流，对未通过验证的工作流的纯过滤器收益。
  • +7.47 pp（SWE-Bench-Verified 困难子集）——LeanEvolve 在已过滤 baseline 之上的验证反馈教师收益，与第一项可叠加。
  • 5 个领先 LLM 通用——增益不绑定单一基模，证明 Lean4 IR 的"模型无关性"。
  • 可定位执行时故障——证明义务的"没填满的洞"直接对应 trajectory 中的失败节点，root cause 可机械化。
  • 测试床为难度子集——SWE-Bench-Verified 困难子集 + ELAIP-Bench，避免"简单题白送"的虚高。
  • 新研究领域——首次将依赖类型 FL 用作 agent 行为 IR，标记一条与"多采样 + 投票"完全正交的可证明性路线。

Harness Engineering（直击毕设）

Lean4Agent 与你毕设"稀疏注意力 KV-cache 预取 + 双水位线 GPU 热缓存"在 IR 抽象上具有可继承的位移逻辑：把"这次预取是否安全"从经验性命中率变成可形式化的不变式——例如"预取项必须落在当前 attention pattern 的传递闭包内"。建议：在毕设 Discussion 章引入 Lean4Agent 作为对照案例，论证你的 prefetcher 也能写成可机械验证的契约——例如 "load(block_i) ⇒ preimage(attention_pattern, i) 非空"——这远比单纯展示命中率更有评审说服力。同时 LeanEvolve 给"预取策略的自动修订"提供了一种纯结构性反馈方案：当某次预取的形式化前提失败时，由形式化错误信号反向修改策略，而不是依赖 RL 信号。这与今日方向 #2 的形式化攻击窗口互文——前者把"该预取啥"形式化，后者把"该攻击啥"形式化。

Agent Skills Safety

Lean4Agent 给"safety policy as type"提供了第一份可执行教材：把"不能把 PII 发到外部"、"API 调用必须先经审计"等政策写成 Lean4 类型而不是 prompt 字符串——任何 plan 的 type-checking 失败都等价于政策违反。这与今日方向 #2（Attack Selection）形成强对仗：当攻击者懂得"战略性地选择何时下手"，blue-team 也必须把合法操作窗口形式化为不变式，而不只靠 monitor 模型的判别能力。把 Lean4Agent 嵌入 control evaluation harness，能让 monitor 在每一步直接得到"当前 plan 类型化检查通过"或"违反第 X 条规约"的硬信号——这是让 control protocol 从经验性走向可证明性的钥匙。

Safety Benchmark

Lean4Agent 暴露了一个新的本征评估维度：verification yield——给定 N 条候选 plan，多少比例能通过形式化验证。它和 SkillDAG 的 Ret@K 一样，是独立于任务成功率的可机械化评估量。建议：未来 agent benchmark 应该同时报告 task-success 与 verification-yield——前者衡量"偶然走对"，后者衡量"结构上正确"。这恰好与今日 #3（AARRI-Bench）的"仅 68.3% 成功"形成补全：68.3% 当中有多少 plan 是能过类型检查的偶然走对，多少是压根写不出可证明 plan 的运气，二者对未来 frontier 模型评估意义截然不同。
