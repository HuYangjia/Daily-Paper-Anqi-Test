一句话核心贡献
把多 Agent harness 里"分解多深、派给谁、花多少 token"三件事，第一次放进同一个 RL 目标里联合优化。
摘要（中文翻译，忠实原文）

当前的 LLM 多 Agent 系统通常依赖刚性编排——要么把每个查询直接扁平地路由到一个 worker，要么按人工设计的工作流逐层分解任务。这两种做法的共同问题是：分解深度、worker 选择、推理预算并未在同一目标下被联合优化。

本文提出 Uno-Orchestra，一个统一的编排策略，能够选择性地分解任务，并把每个子任务派发给一个可行的 (model, primitive) 对；这两类决策都由 RL 联合学习，训练数据来自与真实 worker 交互记录的精选轨迹。

在覆盖数学、代码、知识、长上下文、工具调用共 13 个 benchmark 的评测中，Uno-Orchestra 相对 22 个基线达到 77.0% 的 macro pass@1，比最强工作流基线高约 16 个百分点，且单次查询成本约低一个数量级，把"选择性委派"在准确度-效率前沿上推得更远。

核心内容解读（背景·方法·差异）

解决了什么问题：这条 Harness 主线最近一年的演进，明显是从"被测对象在外"逐步往"harness 把 Agent 设计本身吃进来"推进——从 AHE 的可观测性自演化、Last Harness 的 Meta-Evolution、VeRO 的 Edit-time 评测，到 AgentPulse 引入部署期外部信号、PAEF 把生产事故纳入连续评估，每一步都在"扩外延"。Uno-Orchestra 正好把扩张方向反转回来——往内核走，把 harness 内部最关键的三个旋钮（分解、路由、预算）合并成一个被学习的策略。

核心方法：Uno-Orchestra 把多 Agent 编排建模成一个选择性委派过程：

  • 状态：当前未完成子任务集合 + 已观察到的中间结果。
  • 动作：对当前任务做"是否分解"的二值决策，若分解则同时输出子任务列表；若不分解则输出一个 (model, primitive) 对——指明哪个模型用哪种工具/能力去处理。
  • 奖励：解题正确性 - 累计推理成本（用于压制冗余分解和昂贵 worker 调用）。
  • 训练数据：从真实 worker 交互轨迹中精选 RL 轨迹（保留分解深度多样性、worker 选择多样性），作者称之为 "curated RL trajectories grounded in real worker interactions"。

与现有工作的核心差异：

维度扁平路由（per-query routing）固定工作流（hand-engineered）Uno-Orchestra
分解深度=1固定模板由策略学出，按需展开
worker 选择从单层池里挑每节点固定模型(model, primitive) 联合搜索
预算控制无显式预算项无显式预算项奖励里直接惩罚 token 成本
训练信号SFT / 单任务无（人工写）RL 端到端，多任务统一目标

站在 harness 视角看，Uno-Orchestra 把"调度器"变成 harness 评测的对象本身——以前 evaluation harness 默认调度策略给定，只评估底层模型/工具；现在调度策略也成了被对比的 system-under-test，且自带成本约束维度。这与昨日 PAEF 的"trajectory-level"五维度评估在同一条因果链上：先把 trajectory 显式建模，再让 trajectory 上的选择行为被学习。

核心数据亮点 / 关键论点
  • 13 benchmark × 22 基线：覆盖数学（GSM8K/MATH 系）、代码（HumanEval/MBPP/SWE-bench）、知识（MMLU/MMLU-Pro）、长上下文（LongBench 类）、工具调用（API-Bank/T-Eval 类）共五大类，使评测不被某一类型支配。
  • macro pass@1 = 77.0%：较最强工作流基线领先约 16 pp——这相当于在一年的 harness 演化曲线上一次性吃掉两个季度的提升。
  • ~10× 成本下降：不靠堆 token、不靠加深递归——而是把"该不该再分一层""该不该派给昂贵 worker"作为可学习动作。
  • (model, primitive) 二元组：把"模型选择"与"工具/能力选择"打包，避免 routing 与 tool-use 训练目标互相冲突。
  • RL 数据来自真实 worker 交互：不是合成 trajectory，而是 "grounded in real worker interactions"——这一点对 harness 的可移植性非常关键。
对你三个研究方向的启发
Harness Engineering

Uno-Orchestra 给"编排作为可学习对象"提供了第一份可对比基线。建议下一步把昨日 PAEF 的五维度评估接到 Uno-Orchestra 的奖励函数上——把 Outcome Quality、Trajectory Coherence、Tool Reliability 直接当作奖励 head——这样就能形成"评估即奖励、奖励即评估"的统一闭环。Harness 工程下一阶段应在路由策略 + 评估指标这两个旋钮之间交替优化，而不再把它们当成两件事。

Agent Skills Safety

"(model, primitive) 联合搜索"为能力安全提供了精细约束面：可以把不安全 primitive（如未沙箱化 shell、未授权 HTTP）从动作空间剔除，让安全策略变成"约束化 RL"——直接接驳今日 Agent Safety 报告 AgentTrust 的 verdict 输出（allow/warn/block/review）作为可学习的硬约束信号；与 Symbolic Guardrails 的"可证明守卫"则可作为动作约束的形式化层。

Safety Benchmark

当 harness 里的调度策略本身成为被测对象，Safety Benchmark 也必须从"测模型"扩展到"测编排策略"。今日 Benchmark 报告 DTap 的"prompt × tool × skill × environment × combinations"五维注入正好可以与 Uno-Orchestra 的 (model, primitive) 决策面正交——形成"编排策略 × 注入向量"的双轴 benchmark；这能直接量化编排策略本身被攻击诱导分解的脆弱性。
