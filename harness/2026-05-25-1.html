摘要（中文翻译，忠实原文）

近期 LLM 在理解和生成代码方面表现出强大能力——从竞赛编程一直到 repo 级软件工程。在新兴的 agentic 系统中，代码不再仅仅是目标输出，它越来越多地充当 agent 推理、行动、环境建模与基于执行的验证的"执行基质（operational substrate）"。我们通过 agent harness 的视角刻画这一转变，并提出 "code as agent harness" 这一统一视角——以代码作为 agent 基础设施的中心。

为系统化研究这一视角，本文围绕三个相互连接的层组织：第一层研究 harness 接口——代码如何把 agent 与推理、行动、环境建模相连接；第二层研究 harness 机制——长期执行所需的规划、记忆、工具使用，以及让 harness 可靠 / 自适应的反馈驱动控制与优化；第三层讨论 harness 从单 agent 到 多 agent 的扩展，共享代码工件如何支撑多 agent 协调、审查和验证。

跨越这三层，本文汇总了"代码即 harness"的代表性方法和实际应用，覆盖：编码助手、GUI/OS 自动化、具身 agent、科学发现、个性化推荐、DevOps 与企业工作流。文末进一步梳理了 harness 工程的开放挑战：(1) 超越最终任务成功的评估、(2) 不完整反馈下的验证、(3) 无回归（regression-free）的 harness 改进、(4) 多 agent 间的一致共享状态、(5) 安全关键动作的人类监督、(6) 对多模态环境的扩展。这份综述把"代码作为 agent harness"作为通向可执行、可验证、有状态的 AI agent 系统的统一路线图。

核心内容解读（视角·三层·开放挑战）

解决了什么问题（why a survey now）：2026-04 至 5-23 这一波 harness 研究线（Evaluation Differential · ChromaFlow · ANNEAL · AgentAtlas · Life-Harness）反复指向同一件事——能力差距大量来自"接口"而非"权重"。但社区一直缺一份"地图"：哪些工作算 harness？harness 由哪几个机制构成？多 agent 场景下 harness 如何扩展？本文给出的答案是把"代码"立成 harness 的物理载体——所有 agent 工件（plan / skill / memory snapshot / tool spec / verifier）都可以表达为可执行代码，从而获得统一的形式化处理。

三层组织结构——把 harness 工程拆成可独立研究的子领域：

七大应用域——综述把"代码即 harness"的实际落地分布在：

  • 编码助手——Codex / Claude Code / Cursor 等的 harness 是 IDE-rooted code 工件
  • GUI / OS 自动化——把 GUI 操作编译成可回放的 Python script，OSWorld 等环境
  • 具身 agent——把"行动序列"用代码表达，便于物理仿真器 verify
  • 科学发现——AI co-scientist 把假设 / 数据探索写成 notebook 单元，verifier 自动运行
  • 个性化与推荐——把用户偏好建成可执行 policy
  • DevOps——CI/CD 流水线本身就是天然的 harness
  • 企业工作流——RPA + LLM agent 的混合调度

六大开放挑战——这份综述真正的研究价值在最后这一章：

  • 超越最终任务成功的评估——光看 success rate 不足以衡量 harness 的可靠性，需要轨迹层 / 接口层指标（呼应 5-13 Evaluation Differential）
  • 不完整反馈下的验证——真实环境往往无法即时返回 ground-truth，verifier 必须在不完整 / 嘈杂反馈下工作
  • 无回归 harness 改进——演化 harness 时不能让原有任务退化，与 SkillOpt（5-22）的"严格改进保留集"思想一致
  • 多 agent 一致共享状态——多 agent 系统下"一份代码、多个 agent"的版本一致性
  • 安全关键动作的人类监督——把"必须 escalate to human"做成 harness 的内建检查点（与 CUGA 5-21 / SafeHarbor 系列一脉相承）
  • 多模态环境扩展——把视觉 / 音频 / 物理动作纳入 code-based harness

与"训模型派"对照：这份综述实际上是过去 12 个月 harness 研究路线的官方版图。Life-Harness（5-23）只动接口拿到 88.5% 相对提升，AgentAtlas（5-21）发现 14–40 pp 来自 prompt 监督，本综述则给出方法论解释——这些工作都属于把"代码"作为 harness 物理载体的早期实践。其真正贡献不是新方法，而是把分散的 harness 研究重新编组成一份可继续推进的 research agenda，并以 GitHub Awesome 列表形式提供持续维护的活地图。
