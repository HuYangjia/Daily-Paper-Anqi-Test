摘要（中文翻译，忠实原文）

我们提出 DecisionBench——长程 Agent 工作流中"涌现式委派"的评测底座。该底座固定五件事：（1）任务集合（GAIA、τ-bench、BFCL multi-turn）；（2）peer 模型池（11 模型，7 厂家）；（3）委派接口（call_model 加可选的 read_profile 通道）；（4）确定性的 skill-annotation 层；（5）多轴评测面——质量、成本、延迟、委派率、routing fidelity@k、厂家自偏、以及一个反事实委派上限（counterfactual ceiling）。底座对"peer 信息如何生成或传递"不做假设——所以可学习的路由器、更丰富的 peer 记忆、自适应 profile 构造、以及多步委派都能在同一把尺下评测。

作者用一次5 条件参考扫（n = 23,375 任务实例）刻画该底座，得到三个 benchmark-level 发现：

  • (i) Mean end-task quality 在四种 awareness 条件下统计不可分辨（|β| ≤ 0.010, p ≥ 0.21）——这意味着只看质量的评测会错过整条编排信号。
  • (ii) Routing fidelity@1 在 7.5%–29.5% 区间，且在质量相同的情况下也呈现这一巨大跨度——delivery channel（按需 tool vs 预加载 description）的影响压倒了 description 的内容本身。
  • (iii) 反事实上限把"完美委派"的极限放在所有任务集测得性能之上 15–31 个百分点——给未来的编排方法留出明确的、可被基准量化的进步空间。

作者公开 substrate、annotation layer、reference intervention suite、analysis pipeline、以及 220 个 per-condition run archives。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去三周的 harness 主线刻画了一个清晰的趋势——5-13 的 declarative semantics（Evaluation Differential）、5-14 的 prospective metacognition（TRIAGE）、5-15 的 negative ablation（ChromaFlow）、5-18 的 recovery ladder（CAX-Agent）、5-19 的 governed symbolic patch（ANNEAL）——"加 orchestration 不一定更好"已经被反复证伪。但"谁该做 orchestration 这件事"还没有公共底座。DecisionBench 的关键贡献是把"emergent delegation"做成跟 GAIA/τ-bench/BFCL 同一档的 first-class 评测面，并明确指出"只看 mean quality 会错过整条信号"。这就是为什么 4 个 awareness 条件下 |β| ≤ 0.010 的同时，routing fidelity@1 拉开了 7.5%–29.5% 的跨度——传统 quality benchmark 在这个 substrate 上是盲的。

核心方法：5 件可插拔零件 + 多轴评测 + 反事实上限

  • 固定底座：3 任务集 + 11 模型 + 1 委派接口 + 1 skill-annotation 层 + 多轴指标——所有变量被显式列在 substrate 边界上，让任何路由器/peer memory/profile 构造方法都能"换零件"地参与评测。
  • 多轴指标 7 维同测：quality / cost / latency / delegation rate / routing fidelity@k / vendor self-preference / counterfactual-delegation ceiling——后两项是新引入的两条 first-order 维度，用来区分"碰巧好"和"真知道该派给谁"。
  • 反事实委派上限：每条任务都计算"如果完美委派会是什么质量"——给出"上限-当前"差距，把进步空间量化到任务级。
  • 5 条件参考扫（reference sweep）：no-awareness / on-demand / preloaded-name / preloaded-description / oracle-routing——同一 substrate 跑 5 次，观察"哪条信号能被切掉而 quality 不动"。
  • 220 run archives 公开：不只是数据，整条 reference intervention suite 与 analysis pipeline 都开源——任何后续路由器可以直接对接、无需重建评测面。

核心数据亮点 / 关键论点
  • 23,375 任务实例 × 11 peer 模型 × 5 条件参考扫：把 substrate 跑到了"统计可声明"的规模——任何后续路由器都可以直接对接而不必自己再扫一遍。
  • 4 awareness 条件下 mean quality 不可分辨（|β| ≤ 0.010, p ≥ 0.21）——铁证"只看 quality 的 benchmark 是盲的"，编排信号必须看 routing 维度。
  • routing fidelity@1 ∈ [7.5%, 29.5%]（4× 跨度）：在 quality 几乎相同的情况下，路由对错差 4 倍——这个差恰好藏在delivery channel（按需 vs 预加载）里，而非 description 内容。
  • 反事实委派上限：完美 routing 比当前测得高 15–31 pp（任务集间）——给未来路由器一个明确的"还有多少肉可以咬"。
  • vendor self-preference 维度：把"模型偏向自家厂家"的 bias 第一次纳入正式评测面——这是 multi-vendor agent ecosystem 的bias-aware metric。
  • 220 per-condition run archives + analysis pipeline 全开源（HF）：substrate 真正可复现，不是"附录里给个 prompt"——这一条本身就是 harness engineering 实践的高水位线。
对你三个研究方向的启发
Harness Engineering（直击毕设）

DecisionBench 给了你 KV-cache offload 评测一个直接对位的"substrate 思维"：你目前的 GPU 缓存仿真本质上就是一个委派问题——"哪些 K 行该被 prefetch、哪些该 keep on GPU、哪些该让它走 H2D"。模仿 DecisionBench 的反事实上限，给毕设加一条"完美 prefetch 上限"曲线——任何驱逐+预取策略组合都可以被这个上限标尺度量"还差多少 pp"。同样，把 quality（throughput）与 routing fidelity（命中率/带宽利用率）两轴同测——不让 throughput 单一指标盖住 prefetch 决策的好坏。再加一个 delivery channel 类比：on-demand fetch vs preloaded prefetch_k，哪条路径让带宽更宽——这就是你已经在做的，但 DecisionBench 给了你一份"标准化报告模板"。

Agent Skills Safety

对 agent safety 的迁移路径：把 skill 安全也做成 5 条件参考扫——无 awareness（盲调用）/ on-demand（执行时审）/ preloaded name（仅看 skill ID）/ preloaded description（看 SKILL.md）/ oracle authorization（人工金标）。沿用 DecisionBench 的 mean quality 不可分辨而 routing 拉开的发现，可以预测：很多 skill registry 的"安全性能"在 mean quality 下看不见，但在"是否调对了正确 skill"维度上拉开极大跨度——这跟 5-13 SKILL.md 攻击（86%/77.6%/100% 绕过）完全同一调子，DecisionBench 给了一个量化它的工具。

Safety Benchmark

对 benchmark 方向的启发更直接：反事实上限是一个普适 first-order 评测维度——任何 benchmark 都应该报告 "完美策略上限-当前最强模型"差距，而不只是 "当前最强 / leaderboard"。这个习惯一旦推开，会立刻让 BenchJack（5-14 报告）的 reward-hack taxonomy 升级——hacking 现在可以被"反事实上限是否被超过"判定。同样，把 vendor self-preference 加为标配维度，会改变所有"GPT 评 GPT、Claude 评 Claude"的 judge 设计：DTap、ML-Bench&Guard 等 benchmark 都该补这一项。
