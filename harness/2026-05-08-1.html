一句话核心贡献
把推理 benchmark 的评测单元从"模型"下沉到"endpoint=（provider, model, SKU）"，并连续测量"焦耳 / 正确答案"。
摘要（中文翻译，忠实原文）

公开的推理 benchmark 大多在模型与提供商层面对 AI 系统作横向比较，但实际部署决策的最小单位是 endpoint——即（provider, model, SKU）三元组：暴露了某种特定的量化方案、解码策略、区域、推理服务栈的 SKU。

本文提出 Token Arena：以 endpoint 颗粒度连续评测推理，沿 5 个核心轴（输出速度、首 token 时间、工作负载混合价格、有效上下文、live endpoint 上的质量）测量，并联合一个建模能耗估计，合成出 3 个头条复合指标：每正确答案的焦耳数、每正确答案的美元数、endpoint 保真度（输出分布与第一方参考的相似度）。

该框架的新颖性既是经验也是方法论上的。在 78 个 endpoint × 12 个模型家族上：同一模型在不同 endpoint 上数学/代码任务的平均准确率最大差距 12.5 个百分点，与第一方的指纹相似度最大相差 12 个百分点，尾延迟相差 1 个数量级，建模焦耳/正确答案相差 6.2 倍。进一步表明工作负载感知的混合定价会显著重排榜单：在 chat 预设（输入:输出 = 3:1）下排名前 10 的 endpoint，有 7 个在检索增强预设（20:1）下跌出前 10；reasoning 预设（1:5）反过来抬高了那些被 chat 预设惩罚的前沿闭源模型。

作者以 CC BY 4.0 释出整套框架、schema、probe、eval harness 与 leaderboard v1.0 快照。Token Arena 不是单一榜单，而是一种方法论，作者公布完整溯源信息与限制条件，并欢迎外部复现。

核心内容解读（背景·方法·差异）

解决了什么问题：过去一年的 evaluation harness 演化曲线主要在做"外延扩张"——从 VeRO 把 Edit-time 接进来、AgentPulse 把 deployment 信号接进来、PAEF 把生产事故接进来，到昨日 Uno-Orchestra 把"调度策略"作为被测对象内化。Token Arena 是这条主线上的另一种"内化"——把 serving 栈本身（量化、解码、区域、服务实现）也当作评测变量。它指出：当前公开榜单的最大盲点是把（provider, model, SKU）扁平化看作"同一个模型"，因而忽略了同一模型在不同 endpoint 上 12.5 pp 的准确率漂移与一个数量级的尾延迟差异——这些差异恰好是真实部署 ROI 的支配项。

核心方法（5 轴 → 3 复合）：

  • 核心轴 1（速度）：output speed = tokens/s（持续区间）。
  • 核心轴 2（延迟）：time to first token（TTFT 与尾分位）。
  • 核心轴 3（价格）：workload-blended price（混合输入/输出比例计费，三种预设：chat 3:1、retrieval-aug 20:1、reasoning 1:5）。
  • 核心轴 4（容量）：effective context（实测有效长度，非 spec context）。
  • 核心轴 5（质量）：live endpoint quality（在线测试集分数 + 输出分布与 first-party reference 的指纹相似度）。
  • 建模能耗：从硬件 SKU、量化、并行度反推 J/token。
  • 3 个复合指标：① 焦耳/正确答案；② 美元/正确答案；③ endpoint fidelity（与 first-party reference 的输出分布相似度）。

站在 harness 工程的视角，Token Arena 的最尖锐贡献是承认 SKU 漂移并把它量化：榜单上的"GPT-4o"或"Claude Sonnet"在不同区域、不同量化方案、不同 serving 栈下，输出分布会显著漂移，且漂移幅度足以吞掉"前 10/前 11"这种细粒度排名差异。这与昨日 Uno-Orchestra 的 (model, primitive) 决策面是正交互补的——前者把"用哪个 endpoint"作为 harness 测量目标，后者把"派给哪个 worker"作为 harness 学习目标，二者一起把 harness 的量纲推进到 deployment-fidelity 时代。

核心数据亮点 / 关键论点
  • 78 endpoint × 12 模型家族：覆盖前沿闭源模型与多个开源家族，每个家族跨多 SKU（量化级、区域、serving 栈不同）。
  • 同模型 endpoint 漂移：数学/代码任务平均准确率差距最高 12.5 pp；与第一方参考的指纹相似度差距 12 pp；尾延迟差距 1 个数量级。
  • 能耗-质量耦合："建模焦耳 / 正确答案"指标在 endpoint 间相差 6.2×——同样问题答对一次的能耗最坏比最好高 6 倍。

对你三个研究方向的启发
Harness Engineering

Token Arena 给 harness 工程加了三件被长期忽略的东西：（1）把 SKU 当成评测变量，意味着今后任何 evaluation harness 都应该带 endpoint metadata；（2）"焦耳/正确答案"把能耗第一次和任务质量耦合，这正好是你做 W4A4 量化时端到端 benchmark 的一个理想 north-star——可以把 ZipperEngine 的 e2e 能耗除以正确率作为真正的 ROI 指标；（3）workload preset 把 "prefill : decode 比例" 显式化，这与你做 prefill_seq=2048 / decode_seq=1 的 BF16 benchmark 框架在同一条因果链上：harness 必须按 workload 预设报告，不要再给单一吞吐数字。建议把 endpoint fidelity 作为 ZipperQuant 论文 Section 4.2 实验报告的额外列。

Agent Skills Safety

"endpoint 指纹漂移高达 12 个百分点" 直接揭示了 agent safety 评测的一个隐患：用第一方 model 训练的 guardrail，在第三方 endpoint 上可能因为 SKU 漂移而失效。建议把今日 Agent Safety 报告 SafeHarbor 的 guardrail 评测扩展为 endpoint 矩阵——同一模型在不同 SKU 下的 refusal-rate 和 utility 差异，可能比模型间差异更大。

Safety Benchmark

Token Arena 的方法论可直接迁移到 safety benchmark：safety 表现也应该按 endpoint 报告而不是按 model 报告。今日 Benchmark 报告 XL-SafetyBench 已经把"语言/文化"作为评测正交轴，下一步自然就是叠加 "(provider, model, SKU)" 第三轴——这能区分"模型本身存在文化盲区"与"特定 endpoint 的安全栈过度过滤"。
