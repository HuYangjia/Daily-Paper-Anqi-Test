摘要（中文翻译，忠实原文）

LLM Agent 系统的搭建方式是叠加多种 scaffolding 组件（规划、工具、记忆、自反思、检索），背后假设是"越多越好"。本文研究跨组件干扰（Cross-Component Interference, CCI）：当组件之间相互作用具有破坏性时所导致的能力退化。

我们对 5 个组件的全部 2⁵ = 32 个子集，在 HotpotQA 和 GSM8K 上以 Llama-3.1-8B/70B 跑了一次全因子实验（96 个条件，每个最多 10 个 random seed）。"All-In"系统持续次优：在 HotpotQA 上单工具 Agent 比 All-In 高 32%（F1 0.233 vs 0.177，p = 0.023）；在 GSM8K 上一个 3 组件子集比 All-In 高 79%（0.43 vs 0.24，p = 0.010）。最优组件数随任务而变（k* = 1–4）且与规模相关：在 70B 上，那些在 8B 上有害的组合反而成为增益项；尽管如此，All-In 仍落后于最佳子集。

我们拟合了一个主效应回归（R² = 0.916，adj-R² = 0.899，LOOCV = 0.872），计算了精确的 Shapley 值，发现 183/325 个子模性违例（56.3%），说明贪心选择不可靠。在工具使用、自反思与检索三者之间存在一个三体协同（INT₃ = +0.175，95% CI [+0.003, +0.351]），但被作为探索性结果报告。CCI 在跨模型家族（Qwen2.5）和提示改写下都能复现。我们的结论是："最大装备"的 Agent 默认配置应当被基于交互感知分析的"任务特定子集选择"取代。

核心内容解读（背景·方法·差异）

解决了什么问题：过去一年的 harness engineering 演化，在外延上一路扩张——从 Token Arena 把 SKU/能耗内化、Uno-Orchestra 把 (model, primitive) 决策面内化、PAEF 把生产 trajectory 接进来——所有这些工作都在加东西。Ming Liu 这篇是反方向打的一拳：它把 Agent 自身的 scaffolding 5 个组件（planning / tools / memory / self-reflection / retrieval）当作 5 个二值因子，做一次真正的 2⁵ 全因子实验，第一次给社区一个干净的负证据："加得越多越好"是一个被高度反复但从未被严谨证明的传言；当你真的跑全因子的时候，超过一半的子集对存在子模性违例。这是 harness 工程从"经验主义堆栈"走向"实验设计学（DOE）"的一个关键节点。

核心方法（DOE × Shapley）：

  • 5 个组件全因子：planning / tool use / memory / self-reflection / retrieval，每个 binary on/off → 2⁵ = 32 子集。
  • 2 个任务 × 2 个模型 × 多 seed：HotpotQA（multi-hop QA） + GSM8K（推理），Llama-3.1-8B / 70B，每条件最多 10 seed → 96 条件。
  • 主效应回归：把每个组件当作 0/1 自变量预测最终分数，R² = 0.916（adj 0.899，LOOCV 0.872）说明大部分方差能被一阶解释——但仍剩下不可忽略的非加性。
  • 精确 Shapley 值：计算每个组件的 Shapley contribution，作为"在多大子集大小上贡献多少"的边际效用图。
  • 子模性违例统计：对所有 5×|S\{i}| 对 (S, i) 检查 v(S∪{i}) − v(S) ≤ v(T∪{i}) − v(T) for T⊆S，得 183/325 = 56.3% 违例——这是"贪心选组件"在数学上不可信任的证据。
  • 三体协同：tool use × self-reflection × retrieval 的 INT₃ = +0.175（95% CI 不跨 0），但作者克制地标为 exploratory（避免 multiple comparison 偏差）。
  • 跨模型家族复现：Qwen2.5 上重跑骨干结论一致；提示改写后稳定。

站在 harness engineering 的视角，这篇论文最尖锐的一句话是："主效应回归 R² = 0.916，但仍有 56% 的子集对存在子模性违例。" 翻译过来就是：组件之间的"主效应"足以解释大部分性能差异，但谁要是想用贪心一个一个加组件、希望单调上升，会有一半的概率撞上违例。这一句话把 LangChain 时代"先把全套工具塞给 Agent，再调 prompt"的工程范式判了死刑。它和 "Last Harness You'll Ever Build" 的 Meta-Evolution 是互补的——后者是"自动搜出 harness"的方法论，前者给"为什么必须自动搜（手动堆不行）"的硬证据。

核心数据亮点 / 关键论点
  • HotpotQA：单工具完胜 All-In——单 tool 子集 F1 0.233 vs 全 5 件套 All-In F1 0.177，提升 +32%，p = 0.023。
  • GSM8K：3 组件子集打趴 All-In——3-component subset 0.43 vs All-In 0.24，提升 +79%，p = 0.010。
  • 56% 子模性违例——183/325 对 (S, i) 违反 submodular，意味着"再加一个组件，平均贡献会因子集变大而下降"这一基本假设在过半场景下不成立。

对你三个研究方向的启发
Harness Engineering

这篇文章给 harness 工程的方法论标尺添了一根硬骨头：下一个 evaluation harness 必须自带"组件子集选择"这一维，不要再以"all components on"作为默认报告点。可以直接借鉴到你正在做的 ZipperQuant W4A4 量化 e2e benchmark：把 prefill / decode / KV-cache 优化、QKV fusion、sparse 补偿分支等也当作 binary 组件做小规模因子实验，每个 (Qwen3-8B, prefill_len, decode_len) 配置点用 Shapley 给"哪些 kernel 优化才有收益"做归因——避免你把全套优化堆上去后误判 ZipperEngine 单独贡献。另一个外推：你写 NeurIPS submission 的 ablation table，可以参考它的"k* 随任务变化"图，把 ZipperQuant 不同子集（dense base only / dense + sparse / dense + sparse + fused）单独跑，这是更能说服审稿人的证据形式。

Agent Skills Safety

CCI 对 agent safety 的直接含义是：组件越多，安全表面越大，但安全能力却不单调上升——你不能假设"加上 self-reflection 和 retrieval 就更安全"。结合今日 Agent Safety 报告 Chain of Risk 提到的"reasoning trace 中泄漏 / final answer 看起来安全"的 leak case，CCI 给出的解释是：当 reasoning + retrieval + self-reflection 三件套同时打开时，三体协同的+0.175 也可能用于错的方向（生成看似自洽的错误推理）。建议在做 agent safety 评测时报告 (component subset, ASR) 二维表，而不是单一数字。

Safety Benchmark

CCI 直接挑战了 agent benchmark 设计：主流 agent benchmark（如 AgentBench、HarmBench）默认评测 All-In 配置，因此报告的 "Agent X 在 benchmark Y 上得分 Z" 是 All-In 子集的局部最优，并不能反映该 Agent 在子集选择优化下的能力上限。今日 Benchmark 报告 Partial Evidence Bench 已经在做 ACL-partitioned 评测——下一步自然是叠加 component subset 维度，做"权限 × scaffolding 子集"的二维报告，让评测真正变成 capability frontier 而不是单点。
