摘要（中文翻译，忠实原文）

AI agent 的性能严重依赖于运行时 harness——包括 prompts、工具、记忆和控制流，这些组件介导了模型如何观察、推理与行动。然而，当今的 harness 仍然主要是手工构造且静态的：每个新模型或新任务都需要专门的 scaffolding，执行过程中产生的丰富轨迹也鲜被系统性地蒸馏回改进环节。

我们提出 HarnessX，一个面向可组合 / 可适应 / 可演化智能体 harness 的铸造厂。HarnessX 通过替换代数（substitution algebra） 组装类型化 harness 原语，借助 AEGIS（一个 trace-driven 的多智能体演化引擎，建立在符号适应与强化学习之间的 operational mirror 之上）实现适应性优化，并通过将轨迹同时转化为harness 更新和模型训练信号闭合 harness-model 循环。在 5 个 benchmark（ALFWorld / GAIA / WebShop / τ³-Bench / SWE-bench Verified）上，HarnessX 平均提升 +14.5%（最高 +44.0%），且 baseline 越低提升越大。这些结果表明，agent 进步不必只来自模型 scaling——组合并演化运行时接口是与 model scaling 互补、可操作的另一杠杆。完整代码将在未来发布时开源。

核心内容解读（铸造厂范式 / 9 维分类法 / AEGIS / 协同进化 / 三大病理防御）

立论：harness 不是脚手架，而是第一类研究对象。过去半年（2026 H1）连续四篇里程碑论文已经把 harness 从"不可见的胶水" 推到聚光灯下：5-23 Life-Harness 用 runtime 接口替代训练；5-25 Code as Agent Harness 把代码立为 harness 基质；6-12 HarnessBridge 把 harness 立为可学习模块；6-14 ToolSense 把 harness 立为审计工具。HarnessX 是这条线的最完整集大成——它把 harness 同时升级为：(a) 类型化代数对象（可组合）、(b) trace-driven 演化目标（可适应）、(c) 训练信号源（可与模型协同进化）。论文称这是"foundry" 范式——意味着 harness 不再像"编程" 那样手工构造，而像"铸造" 那样按类型契约自动组装与演化。

系统设计一：9 维分类法 + Substitution Algebra。HarnessX 把 harness 配置 ℋ = (c₁, ..., c₉) 分解为 9 个正交维度，并定义类型化 harness edit（保持类型契约的代码级修改函数）作为代数操作的基本单元：

维度名称典型 harness 元素
D1Model selection角色 → 模型映射、fallback 策略
D2 ★Context assemblyhistory 编辑、prompt 模板、压缩策略（最频繁编辑目标）
D3Memory management长期 / 工作 / episodic memory 接口
D4 ★Tool ecosystemtool registry、tool description、approval flag（最频繁编辑目标）
D5Execution environmentsandbox provider、workspace、plugin list
D6Evaluation and rewardverifier、reward shaping、smoke test
D7Control and safetypolicy gate、checkpoint、interrupt 策略
D8Observabilitytracer、log 注解、指标埋点
D9Training bridgetraining signal extractor、log-prob cache

每个 hook 点（task_start / step_start / before_model / after_model / before_tool / after_tool / step_end / task_end，共 8 个）映射一组 typed processor，每个 processor 有 5 种结果（pass-through / transform / split / intercept / interrupt）和类级元数据（_singleton_group / _order / _after）。这种"有类型 + 有顺序 + 有依赖"的代数结构让 harness 编辑可以在 build 期检测冲突——这是 6-08 #1 Lean4Agent"harness 形式化"思路的工程化落地。

系统设计二：AEGIS = harness 演化的 operational mirror。论文最重要的形式化是定义 (ℋ, ℰ, ℛ, 𝒯) 元组——把 harness 配置 ℋ 视为状态、harness edit ℰ 视为 action、verifier 反馈视为 reward ℛ、trace 视为转移信号 𝒯——从而把 harness 演化建模为符号空间中的 MDP，与 RL 形成精确对应。基于这个 mirror，AEGIS 用 4 阶段流水线（由单一 Meta-Agent LLM 驱动）实现演化：

  1. Digester：把海量 raw trace（GAIA 单次迭代约 ~10M tokens）压缩为结构化任务级摘要——二元结果、失败类别、涉及组件、证据片段；
  2. Planner：在 adaptation landscape 上构建结构化探索方向，避免局部 prompt 改写陷阱；
  3. Evolver：生成候选 harness {ℋ̃ₜᵏ}，附带 change manifest 与 smoke test；
  4. Critic + Deterministic Gating：Critic 检查 manifest 与 trace 证据一致性；4 道确定性 gate（manifest 完整性 / 规范化 / build+smoke / seesaw 回归检查）按序拦截不合格候选。

核心洞察：三大病理与对应防御机制。论文识别 harness 演化的三个 RL-病理类似失效模式：

病理表现防御机制失败案例（论文实证）
Reward hacking把答案嵌入 prompt / 利用 verifier 漏洞Critic 模块审查 manifest-trace 一致性GAIA · Sonnet 4.6 · R10
Catastrophic forgetting修复 A 任务却悄悄回退 B 任务确定性 gating 中的 seesaw 回归检查；variant isolationτ³-Bench · Sonnet 4.6 · Telecom · R7
Under-exploration偏向局部小修改（改 prompt 字眼）而非结构调整Planner 构建 adaptation landscape 强制结构化探索ALFWorld · Sonnet 4.6 · R4–R7

核心实证：Inverse-scaling + Variant Isolation + Co-Evolution 三大现象。实验在 5 基准 × 3 模型族（Claude Sonnet 4.6 / GPT-5.4 / Qwen3.5-9B）上进行，最多 15 轮演化。最关键的实证有三：(a) Inverse-scaling——baseline 越低提升越大，ALFWorld 上 Qwen3.5-9B 提升 +44.0%，Claude Sonnet 4.6 仅 +11.2%；演化的 harness 弥补了弱模型自我修正能力不足。(b) Variant Isolation 解锁异质任务集——GAIA（异质任务）上单 harness 演化停滞，启用 variant-isolation（同时维护至多 K 个变体并按历史成功率路由）后获 +13.6%。(c) Co-Evolution 增益——共享 replay buffer + Cross-Harness GRPO（按 task identity 分组而非 action 级，允许不同 harness 版本共存于同一 group；off-policy 训练，模型版本 lag 上界 ⌊C/s⌋ rounds）相对于单纯 harness 演化再叠加 +4.7%。这条 co-evolution 设计是对 6-12 HarnessBridge"harness 与 model unified instruction tuning"的延伸——HarnessBridge 训练单 harness 与 model 联合，HarnessX 则在 RL 阶段同时演化两者，并提出"scaffolding ceiling vs training-signal ceiling 双天花板" 框架。

方法学姿态：harness 演化与 model 训练共享 RL 形式。HarnessX 把 6-11 #1 SkillJuror"语义受控变体 + 匹配多试验"的方法学姿态推到全 harness 层级——SkillJuror 把 skill 组织方式立成可测变量，HarnessX 把整个 harness 立成可演化 MDP。论文与 6-08 Lean4Agent"形式化建模" / 6-09 SKILL.nb"选择性形式化" / 5-19 ANNEAL"governed 符号补丁" 共同构成 harness 工程的"形式化—演化—修补"三档梯度。2026 H1 后期 harness 工程的统一姿态已经清晰：(1) harness 是类型化代数对象（HarnessX D1-D9）；(2) harness 演化是 RL 同构（HarnessX AEGIS）；(3) harness 与 model 必须协同进化（HarnessX co-evolution）；(4) harness 必须可被审计（ToolSense）。

对你三个研究方向的启发
Harness Engineering（直击毕设）

HarnessX 给你的 KV-cache 系统层毕设最直接的启发是：把 cache 系统的"策略空间" 立成 9 维 + 8 hook 的类型化代数对象。当前 KV-cache 论文的策略改进往往是"拍脑袋一个新 heuristic + 评测"——但 HarnessX 的方法学告诉你：策略改进必须有类型契约 + 自动 build 检查 + smoke test，否则不同改进难以叠加。毕设方法论建议：(a) 借鉴 9 维分类法——为你的 cache 系统定义"caching 策略的 N 维分类法"（如 admission policy / eviction policy / prefetch policy / compression policy / sharing policy 等正交维度）；(b) 借鉴 substitution algebra——每个策略改进必须保持类型契约（e.g. 不破坏 cache 一致性），用 build-time check 拦截不合格组合；(c) 借鉴 4 阶段 AEGIS 流水线——Digester（trace 摘要）/ Planner（探索方向）/ Evolver（生成候选）/ Critic + Gating（确定性回归检查）这套结构可直接套用到 cache 策略的自动演化；(d) 最重要：借鉴 seesaw 回归检查——cache 策略改进经常出现"提升 workload A 但回退 workload B"的 seesaw，HarnessX 的 deterministic gate 设计可直接借鉴。论文写作姿态：把 HarnessX 列为"系统层可演化 framework 的方法学先例"——你的 cache 系统应该能用类似 framework 自动演化策略。

Agent Skills Safety

HarnessX 的三大病理（reward hacking / catastrophic forgetting / under-exploration）与今日方向 #2 MINIM 的"privacy-utility tradeoff" 在方法论上同构——两者都把"自动优化系统的副作用"立为可被独立测量的失效模式。HarnessX 用 4 道 gate 防御 reward hacking；MINIM 用 CI-aware 加权损失防御 high-sensitivity + low-necessity 节点的过度披露。safety 方向的关键启发：未来 agent skills safety 评测必须分别测量 (a) 主指标（任务成功率 / 隐私保留率）；(b) 副作用指标（catastrophic forgetting / sensitive leakage / off-target loss）；(c) seesaw 检查（一组任务改进时另一组是否回退）。这三档指标缺一不可。HarnessX co-evolution 的"双天花板"框架对 safety 同样适用：scaffolding ceiling（guardrail 上限）vs training-signal ceiling（aligned model 上限），二者必须协同推进。

Safety Benchmark

HarnessX 与今日方向 #3 WorkBench Revisited 在 benchmark 设计姿态上形成精神同构——HarnessX 用 5 基准证明"harness 演化跨任务族广泛有效"，WorkBench 重访用 21 模型证明"capability 与 safety 在工作场所 agent 上同向提升而非权衡"。两篇都暗示"agent 进步不是孤立的能力提升，而是 harness × model × benchmark 三角同步演化"。建议：未来 safety / agent benchmark 论文应同时声明：(a) 评测使用的 harness 配置版本；(b) 是否启用 variant isolation；(c) 是否同时报 baseline / best-harness 两档结果。这条姿态在 6-08 #3 AARRI"LLM × harness 二轴评测" 之后又一次升级。HarnessX 同时暴露 benchmark 设计的一个新维度：变更 harness 后 benchmark 数字是否仍然反映"真实 agent 能力" 还是"harness 工程能力"——这条疑问应被未来 agent benchmark 论文写入"discussion / threats to validity"。
