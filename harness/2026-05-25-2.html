一句话核心贡献
把 LLM router 评测从"one-shot 提示"升级为"step-level + 真 agent 链"——以 970 个 router-visible prefix 做 deterministic 静态打分，以 SWE-bench Verified 做 100 case 端到端 live harness 验证，同时锚定离线迭代速度与在线任务成功率。
摘要（中文翻译，忠实原文）

LLM 路由在长程应用（编码 agent / 深度研究系统 / computer-use agent）中最为关键——单次用户请求会触发多次模型调用，把每次调用都路由到"足够便宜又满足质量"的模型可以在不牺牲效果的前提下显著降本。然而现有 router benchmark 只在 one-shot 提示上评估路由器：它们从不暴露中间 agent 步上 router 可见的前缀，从不验证更便宜的替换是否保留下游任务成功率，并且常常依赖在线 LLM judge。

本文提出 TwinRouterBench，一个 step-level 路由 benchmark，包含两条评测轨道：

静态轨道提供 970 条 router-visible 前缀，来自 SWE-bench、BFCL、mtRAG、QMSum、PinchBench 五个数据集中 520 个 instance。每条前缀都配有一个"execution-verified target tier"标签——通过开源的 downgrade-and-cascade 协议估出，并以确定性的算术对 tier 标签 / 轨迹归属 / token 成本打分，不依赖任何 online LLM judge。

动态轨道则提供一个 harness，在完整 500 case 的 SWE-bench Verified 上跑 router；本文报告的是与静态 SWE 监督切片不交叠的 100 case 留出评测。每次 LLM 调用，router 从锁定的模型池中选一个具体模型，成功由官方任务解决判定 + 真实 API 花销共同度量。两条轨道使"快速离线迭代 → 端到端 live agent 执行验证"形成闭环。

核心内容解读（双轨设计 · 确定性打分 · 与"训路由"派的对比）

解决了什么问题（why now）：过去一年内 LLM 路由研究从"单模型 vs 单模型"快速走向"多 tier 路由 + agent 长链"。但所有 router benchmark（RouterBench / RouteLLM / GraphRouter 等）都共用一个隐含假设——router 看到的输入和最终用户提示是同一份。这在 agent 场景里完全不成立：当 SWE-Agent 已经跑了 12 步、上下文里有 8K tokens 的工具响应时，再决定下一步用 GPT-5.4 还是 Sonnet 4.6，需要的评测样本本身必须是这种带前缀的中间快照。TwinRouterBench 的第一性原理就是把"router-visible prefix"作为基本评测单元，从此把 router 评测和 agent 评测在数据形态上对齐。

双轨设计的工程含义：

属性静态轨道动态轨道（harness）
样本形态 970 个 router-visible prefix（已固化） SWE-bench Verified 100 case（live agent 跑完整轨迹）
label 来源 execution-verified target tier（用 downgrade-and-cascade 协议反推） 官方 task resolution + 实际 API 花销（pass/fail × $）
评分函数 deterministic 算术：tier 标签 + 轨迹归属 + token cost 真实任务成功率 × 实际 API 支出
是否依赖 online LLM judge 否（核心卖点之一） 否（用官方 SWE-bench evaluator）
用途 快速离线迭代（router 设计 / 调参） 端到端 sanity check（防止离线-在线漂移）

downgrade-and-cascade 协议：论文把如何决定一个前缀的"target tier"也开源了——给一个前缀依次从最便宜模型试到最贵模型，找到第一个能保持下游任务成功的最便宜模型，作为该前缀的 ground-truth tier。这一反向标注机制比"用 LLM judge 当场打分"更稳定，也是 deterministic scoring 的前提。

5 个上游数据生态共建：SWE-bench（软件工程）、BFCL（工具调用）、mtRAG（多轮检索）、QMSum（会议摘要）、PinchBench（多模态）——这五个数据集横跨"代码 / 工具 / 检索 / 长文本 / 多模态"，覆盖了 router 在真实部署里最常见的负载形态。这种"静态轨道是横切 5 个 benchmark 的步级快照、动态轨道收口到 SWE-bench"的双重对齐，才是 TwinRouterBench 区别于过去 router 评测最大的工程贡献。

与"训路由派"的对比：过去半年路由研究主要是 RouterDC / GraphRouter / Eagle 等"训一个分类器"思路。TwinRouterBench 不否认这条路线，但把它放在更小的位置——路由的瓶颈不在分类器精度，而在评测数据形态。当评测样本本身缺少 router-visible prefix 时，无论训得多准都会与真实部署收益脱节。这个观察呼应了今天 #1 的 Code as Agent Harness 综述里 L1 接口层的核心论点——能力差距大量来自接口而非权重。

核心数据亮点 / 关键论点
  • 970 router-visible prefixes / 520 instances / 5 上游 benchmark——把 router 评测的样本规模从"单 prompt"推到"step-level 中间快照"。
  • 动态 harness × SWE-bench Verified 100 case——首个把 router 决策放回真 agent 链端到端跑出 task resolution + API 花销的双指标。
  • deterministic arithmetic scoring——彻底去掉 online LLM judge，使评测可重复、可审计、可放进 CI。
  • downgrade-and-cascade target-tier 协议开源——任何想造自己 router benchmark 的人都可以直接复用这套反向标注机制。
  • locked model pool——避免"今天换了模型就不能复现昨天的结果"的常见 router 评测陷阱。
  • 静态切分 vs 动态切分严格不交叠——100 case 留出用于动态轨道，避免 overfitting 到静态轨道。

Harness Engineering（直击毕设）

对你毕设最直接的启发是"评测数据形态决定 harness 收益的天花板"。你的 KV-cache prefetch + 双水位线工作如果想在更大尺度证明价值，评测样本必须是 router-visible prefix 这种"中间快照"——也就是说，要在 LongBench-v2 之上构造一个"已跑了 N 个 prefill+decode token 时是否值得 prefetch 下一段"的 step-level eval set。这恰好对应 Q.md 评审里"为什么不直接看长文本端到端 latency"——答案是：端到端只能给最终数字，step-level harness 才能给你"在某一步 cache 状态下迁移决策的边际收益"。把 TwinRouterBench 的 deterministic scoring 思路类比到 KV-cache：把"router 选 tier"换成"cache 选驻留 / 卸载 / 预取"，把"task resolution + API cost"换成"hit rate + bandwidth + latency"，方法论几乎一一映射。

Agent Skills Safety

"router-visible prefix" 这一抽象在 safety 方向的对应物是 "safety-checker-visible prefix"——也就是说，agent 安全检查器在 12 步对话之后看到的上下文，与第一轮 prompt 的形态完全不同。今天 #1 的 SafeHarbor（分层记忆 guardrail）和昨天的 Latent-space Refusal Evasion 都暗示——safety check 的 step-level prefix 才是真正攻击面。TwinRouterBench 的 970 prefix 数据集形态，可以直接被 borrowed 用于 safety check 的离线测试。

Safety Benchmark

对 benchmark 设计的启发：双轨（静态快 + 动态真）是未来 safety benchmark 的标准模板。今天 #1 的 Boiling the Frog（多轮有状态 ASR）已是 dynamic safety 评测的雏形，但缺一条"快速离线静态轨道"。如果把 Boiling the Frog 的 9-step 轨迹切成 step-level prefix + label，再用 deterministic scoring 评判每一步是否触发了拒绝，就能得到一个"safety 版 TwinRouter"——10× 离线迭代速度 + 端到端真实成功率。
