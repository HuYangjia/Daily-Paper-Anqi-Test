摘要（中文翻译，忠实原文）

LLM agent 不只由语言模型本身塑造，还由 runtime harness 塑造——后者负责调节观察、工具使用、动作执行、反馈解释和轨迹控制。现有的 agent 适配方法主要更新模型参数，但在确定性、规则治理的领域中，许多失败其实并不发源于模型，而是源于模型–环境接口的不匹配。

本文提出 Life-Harness：一个生命周期感知的 runtime 框架，可以在不改动模型权重、不改动评估环境的前提下，改进冻结的 LLM agent。Life-Harness 从训练轨迹中演化出来，把反复出现的交互失败翻译成可跨环境复用的"契约 / 技能 / 动作实现 / 轨迹调节"四类干预。

在 7 个确定性环境上跑 126 个 model–environment 设置——Life-Harness 改进了其中 116 个，跨 18 个模型骨干平均相对提升 88.5%；并且，从 Qwen3-4B-Instruct 一条轨迹演化出来的 harness，可零额外成本转移到剩下 17 个模型骨干上。论文用 τ-bench / τ²-bench / AgentBench 等基准做了可复现验证。作者把这一方法论定位为"对模型中心训练的补充而非替代"。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：2026-04 至 5-21 这条研究线已经在反复说同一件事——5-13 Evaluation Differential 把"模型 vs harness"分开记账，5-15 ChromaFlow 用 negative ablation 否定盲目堆 orchestration，5-19 ANNEAL 把 recurring failure 翻成 governed 符号补丁，5-21 AgentAtlas 揭示模型表观能力中 14–40 pp 来自 prompt 监督而非模型本身。Life-Harness 是这条线的正面落地版——既然能力差距大量来自接口而不是权重，那就只动接口：在 7 个确定性环境 × 18 模型 = 126 设置中改进 116 个、相对提升 88.5%，并把"从 4B 模型一条轨迹演化的 harness 可转移到 17 个其他模型"作为可证伪的强主张。

核心方法：四件可独立演化的接口干预

  • 环境契约（Environment contracts）——把环境对 agent 的"观察格式 / 反馈语义 / 错误码含义"显式写成契约文档塞进 prompt。把 agent 看错的失败从"模型理解不了"翻译成"接口没说清"。
  • 程序技能（Procedural skills）——从训练轨迹里挖出反复使用的子流程（先 search 再 read / 先校验再写入）固化为 named skill，让冻结模型走熟路而非每次重新规划。
  • 动作实现（Action realization）——把 LLM 输出的高阶动作落到具体 API 调用的"翻译层"做一遍打磨：参数补全、副作用幂等化、合法/非法集合裁剪——把"格式错"和"语义错"在接口侧拦截。
  • 轨迹调节（Trajectory regulation）——在跨步层面注入"什么时候要 stop / 什么时候要复盘 / 什么时候要 escalate"的 meta-rules——把 trajectory shape 也接口化、让 harness 接管轨迹政策。

四个维度正交可独立增删，构成一份可复用的 "跨环境 / 跨模型契约-技能-实现-轨迹" 资产清单——这就是论文反复强调的 "life-cycle aware" 含义：harness 自己有版本演化史。

核心数据亮点 / 关键论点
  • 116 / 126 设置改进——7 确定性环境 × 18 模型骨干，几乎全覆盖；改进率约 92%。
  • 平均相对提升 88.5%——不动权重、只动接口的提升幅度，与一次中等规模 SFT 的提升相当甚至更高。
  • 从 Qwen3-4B-Instruct 一条训练轨迹演化的 harness 零成本转移到 17 个其他模型——证伪"harness 必然过拟合到一个模型"的隐含假设。
  • 四维正交干预（契约 / 技能 / 动作 / 轨迹）——可单独消融的设计，把"接口工程"做成可量化的科研对象。
  • 明确定位"补充而非替代"——作者并没有反对模型训练，而是把 runtime harness 立成与权重训练对等的二元结构。
  • 评测使用 τ-bench / τ²-bench / AgentBench——三个公开 deterministic 环境构成可复现基线。

Agent Skills Safety

对 agent safety 的迁移：把"安全"也做成 runtime 接口的四维干预——安全契约（什么是 unsafe 动作的形式定义）、安全程序技能（敏感请求的 ask/refuse/confirm 模板）、安全动作实现（参数审计、副作用确认）、安全轨迹调节（meta-rule "高风险动作必经 review"）。这跟 5-21 CUGA Policy System（5 个结构化执行检查点）、5-13 SKILL.md（86%/77.6%/100% 绕过）、5-20 ECA（authorization-failure gate）一脉相承——Life-Harness 提供了"不改模型权重就能拿到安全"的方法论骨架。

Safety Benchmark

对 benchmark 设计的启发：评测协议必须同时报告"模型骨干 vs 接口干预"两条曲线，否则混淆能力归因。具体可借鉴：(1) 把 7 环境 × 18 模型 = 126 设置的强网格作为"benchmark 充分性"的最小尺度；(2) 把"4B 演化 harness 转移到 17 模型"做成 transferability 矩阵——任何号称泛化的方法都该补这一栏；(3) 把 "116 / 126 改进" 作为 "不退化保证" 这一新 desideratum 写进 5-21 GrandGuard、5-20 POLAR-Bench 一类细粒度 benchmark 的报告模板。
