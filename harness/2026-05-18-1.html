摘要（中文翻译，忠实原文）

把大语言模型部署在 MAPDL 有限元仿真任务上时面临实际可靠性挑战：没有结构化的执行控制、工具封装与故障恢复，输出会不一致、任务失败常见。Agent Harness 范式通过在中间插入 domain-specific 的 orchestration middleware 来管理工具生命周期、工作流状态与恢复升级，正是为了解决这一问题。本论文呈现 CAX-Agent——一个面向 MAPDL 自动化的轻量级 agent harness 架构，并实证评估了它的核心组件：recovery policy。

CAX-Agent 把执行分为三层——LLM service / agent harness / solver backend——并配以一个 recovery ladder：从确定性 rule patching 升级到 model-driven regeneration，再到 context enrichment，最后到人工干预。评测覆盖 3 种 recovery 策略（no_recovery、rule_only、model_only）在 50 道标准结构 benchmark 上各跑 3 次，共 450 case-runs。

model_only 拿到最佳完成率 0.9267、任务分 3.59/4、综合分 9.16/10、零干预率 0.84——大幅领先 rule_only 与 no_recovery，Cliff's δ ∈ [0.81, 0.87]（大效应量）。inter-rater agreement κ = 0.84。

核心内容解读（背景·方法·差异）

解决了什么问题（why now）：过去三周 harness 主线一直在外化决策面：5-08 endpoint（Token Arena）、5-09 scaffold（More Is Not Always Better）、5-12 工具拓扑（ComplexMCP）、5-13 声明语义（Evaluation Differential）、5-14 前瞻 metacognition（TRIAGE）、5-15 负向消融（ChromaFlow）。这些工作回答的都是"评测面要看什么"。CAX-Agent 把视角换到"运行时要修什么"——它把 agent 在工具调用失败后该怎么爬出来，写成有梯度的 recovery 升级路径，而不是"出错就 retry / 出错就降级到 human"。这是 harness 工程从"评测层"下沉到"运行时控制层"的一个干净样本。

核心方法：三层架构 + 四阶 recovery ladder

  • 三层架构：LLM service（推理）↔ agent harness（编排/工具/状态/恢复）↔ solver backend（MAPDL）。关键设计：harness 不是单纯的"调度胶水"——它持有 workflow 状态、tool lifecycle 和 recovery 决策树，把 LLM 的不确定性吸收在 harness 这一层，不让它穿透到 solver。
  • recovery ladder（4 阶梯子）：①deterministic rule patching——能用规则修就不调模型；②model-driven regeneration——让 LLM 看错误信息重生成；③context enrichment——把更多 solver 状态、工具文档、上一次失败的 trace 注入再让模型来；④human intervention——以上都失败才告警。这一阶梯把"修错成本"做成单调递增的可观测变量。
  • 定级实验：消融了梯子上的不同分段——no_recovery（梯子全关）/ rule_only（只第一阶）/ model_only（只第二阶）。每策略 50 题 × 3 重复 = 150 跑次，共 450 跑次。这是 harness 论文里少见的"按 recovery 强度做 ablation"。
  • 评分协议：完成率（task is solvable）+ 任务分（4 分制）+ 综合分（10 分制）+ 零干预率（不需要人介入的比例）。inter-rater κ = 0.84，证明评分稳定。

核心数据亮点 / 关键论点
  • 完成率 0.9267 / 任务分 3.59/4 / 综合 9.16/10（model_only）：相比 no_recovery 与 rule_only 大效应量提升，Cliff's δ ∈ [0.81, 0.87]——这是非参数效应量大档，说明梯子第二阶的收益不是噪声。
  • 零干预率 0.84：约 84% 的 case 完全不需要人类介入即被 harness 自动恢复——这把 "agent 自治度" 做成了一个可量化的 harness 出口指标，而不是 vibe-checked claim。
  • 450 case-runs · κ = 0.84（rater agreement）：这是少数报告了"评分协议本身被验证"的 harness 论文，意味着论文对自己的评测面也下了 audit——属于"harness 内的 harness"。
  • recovery 梯子的"梯度"本身是论点：作者明确把恢复升级写成线性偏序，不让"恢复"塌缩成黑盒 retry——这是把 reliability engineering 的 SRE 思路（incident severity ladder、escalation policy）搬到了 LLM-agent harness。
  • scope 透明：论文把自己定位为 "lightweight"，不试图扩到任意工程仿真，而是固定 MAPDL 做 deep-dive。这种"先在窄域内做完整 recovery 工程，再向其他域迁移"的写法，是 harness 论文应有的科学诚实。
  • 方法学外推：4 阶梯子可直接迁到 git agent / CAD agent / 数据库 agent / KV-cache offload runtime——只要有一个确定性后端 + LLM 决策面，recovery ladder 都成立。
对你三个研究方向的启发
Harness Engineering（直击毕设）

CAX-Agent 给了一份把"修错"做成可消融变量的工程蓝本——这正是你 KV-cache offload 实验里现在缺的东西。你的 prefetch + Top-K 索引体系本质上就是一份"recovery ladder"：①规则触发的预取（top-k indexing pre-load）→②基于命中预测的 model-style 预取（ask predictor）→③context enrichment（提前一层把 query 信息塞进 prefetch decision）→④fallback（HBM miss 时 sync H2D）。建议在毕设 v3 实验章直接借 CAX-Agent 的写法做"recovery 强度"ablation：no_prefetch / rule_only（top-k）/ model_only（learned predictor）/ context_enrichment（query-aware）。这把现在分散的几条曲线整合成一条单调升级的工程论证，并配上 Cliff's δ 这种非参数效应量——比单看 hit-rate 数字更可发表。

Agent Skills Safety

recovery ladder 的镜像形式可以做成 safety escalation ladder：①确定性 policy 检查（regex / schema）→②model-as-judge → ③ask-clarification 注入更多 user context → ④ block + human review。今日选定的 Ensemble Monitoring 给出 "diversity 比 scale 更重要" 的 monitor-集成证据；把它接到 CAX-Agent 的梯子里——monitor 不再是单一关卡，而是阶梯上的不同等级，可以按"危险度"自适应升级。这是把 AI control 从"yes/no 闸门"升级到"分级响应"的一种结构化做法。

Safety Benchmark

对 benchmark 设计的启示：每个 agent benchmark 都应该附 recovery 强度坐标轴。今日选定的 SaaS-Bench 报告"strongest model 仅 <4% e2e completion"——这个数字本身没意义，因为没说"agent 在 recovery 上能爬几阶梯子"。CAX-Agent 的 4 阶坐标可以直接搬到 benchmark：报告"在 rule_only/model_only/context_enrichment 三档下的 completion 曲线"——把"无 recovery 时的 raw 能力"和"配齐 harness 后的有效能力"分开评，避免 benchmark 数字被 harness 工程"白嫖"。
