一句话核心贡献
用 append-only 事件日志 作 agent 状态的确定性投影，把"诊断 → 补丁 → 留出集门控 → 提升 / 丢弃"做成每一步本身就是事件的可审计闭环。
摘要（中文翻译，忠实原文）

自主改进闭环之所以难以信任，是因为改进过程通常以"外挂脚手架"形式螺接到 agent 上——失败不被记录、诊断无法回放、提升或丢弃决策落到旁路数据库而不是 agent 自己的历史里。我们展示：事件溯源（event-sourced）的 agent runtime 能消除这种摩擦，把"受控改进"变成一等工作流。

当 agent 状态是追加式（append-only）事件日志的确定性投影时，失败会被记录、运行能从其日志精确回放、候选补丁的作用域被限制在类型化的管线接缝（typed pipeline seams）、所有 gate 可审计，且每一次提升或丢弃都是一个事件。

我们在 ActiveGraph runtime 上实现了名为 Regimes 的闭环：诊断失败的评估 → 在管线某点提出修复 → 仅在通过静态检查、沙箱执行、样本内评估、留出集验证这四道门之后才提升。该闭环 target-agnostic——同一控制流经统一接口可作用于不同任务。

在 LongMemEval-S 上，主要的失败模式不是检索，而是调和（reconciliation）——证据已在组装好的上下文里，但 reader 仍给出错答。在 5 个 seeded held-out 划分上，Regimes 发现的reader-prompt 修复使最终留出集准确率在 4 个划分上提升 +0.05~+0.10，在 1 个 over-promotion 划分上提升 +0.01；其中 2 个划分单独显著（seed 5 未对其顺序提升结构进行调整）；汇总计数仅作描述性参考，因为 5 个划分共享同一 500 题池。

持久贡献为：(1) ActiveGraph 作为可审计基板使受控改进闭环工程化可行；(2) 留出集门控的闭环本身；(3) failure-regime taxonomy——把每种失败路由到管线某具体位置（其相对无路由基线的边际价值是主要开放问题）；(4) prompt-as-discovery-probe 假设。

核心内容解读（事件溯源 / 类型化接缝 / 留出集门控 / failure-regime taxonomy）

立论：自主改进的真问题不在"会不会改"，而在"改完算不算数"。2026 H1 自主改进 agent 已经百花齐放——5-19 ANNEAL 把修复立成 governed 符号补丁、5-26 Gu 的 system scaling 把 verification cost 立成评测维度、6-08 Lean4Agent 把工作流抬到证明系统、6-09 SKILL.nb 把 lifecycle 治理立成评测维度。但所有这些都默认了一个事：改完之后的 agent 状态是权威的、不可回放的、不可问责的。Regimes 的赌注是反向的：如果 agent 状态本身就是 append-only 事件日志的确定性投影，那么"提升 / 丢弃"决策就一定能被审计、被回放、被驳回。

事件溯源 runtime 的工程哲学：状态 = 事件流的投影。这条思路在金融系统、CQRS 架构、Git 等已成熟二十年，但首次被用作 agent 自主改进的工程根基。在 ActiveGraph 中：

事件类别记录内容下游意义
RUN（一次任务执行）任务输入、agent 决策序列、工具调用、最终输出、评估分数失败分析的 ground truth；可从日志逐步重放
FAILURE（评估失败）失败实例 + 路由到 failure-regime 的某一位置（管线 N）定义"补丁应作用在哪一接缝"
PATCH_PROPOSED（候选补丁）类型化的管线接缝、修改前后版本、修改理由限定补丁的作用域，避免"全局乱改"
GATE_PASSED / GATE_FAILED（门控）四道 gate 各自结果（static / sandbox / in-sample / held-out）每道门是独立的 commit point，单一 gate 失败 → 自动回滚
PROMOTION / DISCARD（最终决策）是否把补丁纳入 agent 主流程本身就是事件——后续可回滚、可审计

四道门控的真正价值：分离"代码错误 / 局部成功 / 全局回归"三种风险。静态检查（编译 / 类型 / 静态规范）拦下平凡补丁错误；沙箱执行验证不会破坏环境；样本内评估验证补丁能修当前失败；留出集（held-out）才决定是否对未见数据有效。这一层设计直接对位 ML 圈的 train / val / test，但首次用在 agent 改进闭环——"补丁过 in-sample 就 promote"是过去半年最流行也最隐蔽的 p-hacking 来源，它直接驱动 6-09 候选 #2 PACE 的 anytime-valid 提交检验。Regimes 的 held-out 门控走的是经典 frequentist 路线，它和 PACE 形成了"prompt-level 改进 vs 系统级评测公允"的两条互补防线。

Failure-regime taxonomy：把"它哪里错了"路由到"修在哪一步"。论文最具普适价值的洞见是：在 LongMemEval-S 上，主导失败不是检索失败，而是 reconciliation 失败——证据已在 reader 的输入里，但 reader 给出了错答。这个发现把 RAG / 长记忆系统圈子的研究方向重新校准：把研究投入压在 retrieval 上是错配 — reader 才是瓶颈。Regimes 把这条洞察工程化成了 taxonomy：每个失败实例都被映射到管线某一具体位置（retriever / 上下文组装 / reader 等），失败位置决定补丁作用域。这是一种"把诊断学嵌入运行时"的设计哲学，与 6-08 Lean4Agent "类型作为外部凭证" 在精神上同源。

"Prompt 作为 discovery probe"：把 prompt 改写从"trick"升级为"研究方法"。多年来 prompt engineering 一直被视为缺乏理论的 trick；Regimes 提出一种新的解释：prompt 改写就是对 LLM 行为流形的探针实验。在 LongMemEval-S 上，所有的修复都集中在 reader prompt，5 个划分中 4 个划分能稳定带来 +0.05~+0.10 的准确率提升——这暗示"哪一类 prompt 微调有效"本身是一个可被实验、可被分类、可被复现的研究对象。这一点对应于 5-25 Code as Agent Harness "代码作为推理基质" 的另一极：不是用代码取代 prompt，而是反过来用结构化方法研究 prompt。

实验信号怎么读：4 / 5 划分稳定提升，1 个 over-promotion——把"诚实声明负面信号"立成范例。论文显式承认：5 个 splits 共享 500 题池，所以汇总计数仅作描述性参考；其中 1 个 split 仅 +0.01 是因为 over-promotion；2 个 splits 单独显著，但 seed 5 没有针对顺序提升做调整。这种对统计弱点的诚实声明本身就是 agent 评测圈子稀缺的研究品质——和 6-08 AARR/AARRI-Bench "把 harness 与模型解耦报告" 共同构成 2026 年中"评测要老老实实承认混淆变量"的姿态。

研究路线意义：把"agent 改进"从行为学问题变成系统学问题。Regimes 之前，自主改进 agent 的主要研究范式是 RLHF / RL post-training / Reflexion 等"行为层"方法；Regimes 把范式拉到"系统层"——你需要的不是更强的反思器，而是把每次反思的输入 / 输出 / 决策都立成可回放事件。这条路径直接对位软件工程圈过去十年的 trace-based observability（OpenTelemetry / Jaeger 等）——当 agent 进入生产，可观测性必须先于智能性。

核心数据亮点 / 关键论点
  • 4 / 5 splits 留出集准确率提升 +0.05~+0.10——稳定的边际改进，验证 prompt-as-discovery-probe 假设。
  • 1 / 5 splits 仅 +0.01（over-promotion）——论文显式承认负面信号；4 道 gate 仍能拦下大多数过度提升。
  • 主导失败模式：reader reconciliation 而非 retrieval——把 RAG 圈研究重心从检索器拨向 reader。
  • 500 题共享池——5 个划分非独立，论文显式声明汇总计数仅描述性。
  • 4 道 gate（static / sandbox / in-sample / held-out）——把"p-hacking-by-loop"问题工程化分离。
  • Append-only 事件日志——agent 状态成为可回放、可审计、可回滚对象。
  • Failure-regime taxonomy——把"它错在哪"路由到"补丁作用在哪一接缝"，统一改进闭环的语义层。
  • Target-agnostic——同一控制流通过统一接口对接不同任务，迁移成本接近零。

对你三个研究方向的启发
Harness Engineering（直击毕设）

Regimes 给你的"稀疏注意力 KV-cache 预取 + 双水位线 GPU 热缓存"毕设最直接的工程范式启发是把每次预取决策、每次水位调整、每次驱逐都立成事件流——而不是 just-in-time 计算的隐式状态。建议：(1) 在毕设 runtime 里加入append-only 决策事件日志——每次 prefetch / evict / 水位调整都写一条事件，含触发条件 / 当时 attention pattern hash / KV-cache 命中率快照；(2) 评测层引入4 道 gate对位 — static gate 验证策略本身合法（cache 总量上限）、sandbox gate 验证策略不破坏正确性（与 baseline 对照 logits 一致）、in-sample gate 验证 hit-rate 提升、held-out gate 验证未见 prompt 上仍然 hit-rate 提升；(3) 最重要的：把"策略提升"本身立成事件——这意味着你的毕设可以诚实承认 some workloads regress，而不是被迫只报"平均 hit-rate 提升 X%"。这条诚实姿态在 ML systems 论文里少见且非常加分。

Agent Skills Safety

Regimes 对今日方向 #2 Arbiter Agent（多智能体 misalignment 检测）形成天然耦合：Arbiter 在对话级工作，但当 misalignment 被检测出来需要修时，Regimes 提供的事件溯源 + held-out gate 就是修复的工程基底。建议：把 Arbiter 的 inspection budget 决策、Regimes 的 promotion / discard 决策合并到同一事件流——所有"检测 / 干预 / 修复 / 提升"四类操作变成统一事件，安全审计与改进审计共享 trace。这条路径直接对位 5-19 memory laundering：当 sub-threshold 状态污染发生时，事件溯源是唯一能让"修复回到第一次污染源"的工程基底。

Safety Benchmark

Regimes 给 agent benchmark 引入了一个新的报告维度：报告 over-promotion 比例。当一个改进闭环在 N 个 held-out splits 上仅在 N-1 个有提升、1 个 over-promote 时，这个 1 个就是 benchmark 报告的关键数据。建议：未来 agent benchmark（包括今日 #3 PreAct-Bench、5-26 AgentHijack、6-08 AARR/AARRI-Bench）应同时报告 (a) 平均提升 / (b) over-promotion 比例 / (c) failure-regime 分布。这与 6-03 What Benchmarks Don't Measure 提出的"safety-usability tunable"在精神上同源——评测面必须从"能不能做对"扩展到"提升是不是诚实的"。
