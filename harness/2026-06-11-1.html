一句话核心贡献
把 agent skill 的组织方式（"渐进式披露 vs 扁平化"）从内容里剥离出来作为独立可测变量，证明组织方式先改变运行时行为再改变结果——SkillsBench 82 任务上轨迹接触资源数 1.18 → 3.85、验证器通过率 +4.1%。
摘要（中文翻译，忠实原文）

Agent Skills 在推理时为大语言模型 agent 注入程序性知识，但当前 benchmark 极少区分一个 Skill 说了什么 与 它如何被组织。我们通过 Progressive Disclosure（渐进式披露）研究这一区分：让一个简洁的根文件按需把 agent 引导到支持资源；并将其与归一化的扁平化基线对比。

我们提出 SkillJuror，一个评估 Skill 写作范式的框架，通过 语义受控变体、匹配的多试验评估 与 轨迹证据，在保持任务知识不变的条件下进行测量。

在 82 任务的 SkillsBench 研究中，Progressive Disclosure 在聚合结果之前先改变了运行时行为：每条轨迹接触的不同 Skill 资源数从 1.18 上升到 3.85，有效的 uptake 事件从 1.33 上升到 3.92。它也带来了相对扁平化基线的 +17 / 410 匹配试验 (+4.1%) 的额外验证器通过试验。

该收益取决于任务。Progressive Disclosure 在支持资源用于指导实现 / 检查 / 修复时有帮助，但在成功取决于精确输出约定、数值阈值或长产物生成管线时表现较弱。这些结果表明：Skill 组织不只是"呈现形式"——它会改变 agent 搜索与应用程序性知识的方式，而结果上的收益取决于暴露的资源对当前任务是否可操作。

核心内容解读（语义受控变体 / 匹配多试验 / Progressive Disclosure / 行为先于结果）

立论：当前 benchmark 把"说了什么"和"怎么组织"混为一谈。2026 H1 已经形成一个明确共识：Anthropic 5-13 SKILL.md（arXiv 2605.11418）把 Skill 立为"声明型可注册的程序知识资产"；6-03 SkillDAG（arXiv 2606.03056）把 skill 检索从相似度匹配升级为类型化有向图；6-09 SKILL.nb（arXiv 2606.08049）把 lifecycle 治理立成评测维度。然而所有这些工作都默认了同一件事：写得好的 Skill 自然会被 agent 用得好。SkillJuror 的赌注是对这一假设直接做反事实——同一份程序性知识，在两种组织形态下，agent 的运行时行为差异是否大到足以与"内容"分开测量？答案是是。

方法学贡献：把"组织方式"从"内容"中剥离的三件套。SkillJuror 的工程难点是如何在保持任务知识不变的前提下只改变组织，作者给出三件套：

工具作用对应解决的混淆
语义受控变体（semantically controlled variants）对同一组 skill 资源做归一化重写——内容字面表述、知识量、token 数全部对齐，仅"是否一上来就把所有内容塞给 agent"这一组织维度变化排除"Progressive Disclosure 写得更短/更详细"造成的伪相关
匹配多试验评估（matched multi-trial evaluations）每个 (任务 × 变体) 对都跑多次试验，用同一组 seed 与采样策略——410 试验对作为统计基线排除采样噪声"偶然让一种形态看起来更好"
轨迹证据（trajectory evidence）除了最终是否通过验证器，还记录"每条轨迹接触了哪些 skill 资源 / 资源被实际使用还是只被读取"排除"组织变化只让 agent 看起来不同但其实做了同一件事"的虚假改进

核心发现一：组织方式先改变行为，再改变结果。这是论文最重要的方法学发现：在 82 任务 SkillsBench 上，聚合结果（验证器通过率）只增 4.1%，但每条轨迹接触的不同 skill 资源数从 1.18 跳到 3.85（+226%）、有效的 uptake 事件从 1.33 跳到 3.92（+195%）。这说明 Progressive Disclosure 在很大程度上改变了 agent 与知识库的交互方式，而这些行为变化只有一部分翻译成了最终结果改进。SkillJuror 因此立成了一种"行为指标先于结果指标"的评测范式——和 6-10 Regimes（arXiv 2606.10241）的 4 道 gate 并列：Regimes 在纵向用事件溯源把改进闭环立成事件流，SkillJuror 在横向用轨迹证据把"知识被如何使用"立成可测量行为。

核心发现二：Progressive Disclosure 的收益是 task-dependent。论文用大量篇幅证明这不是一句简单的"有时管用有时不管用"——而是有具体边界条件：当支持资源用于指导实现、检查、修复时，渐进式披露能帮 agent 找到合适的指引；但当成功取决于精确输出约定、数值阈值、长产物生成管线时，渐进式披露反而较弱——因为这些场景需要一次性把所有约束摆在 agent 面前，按需暴露反而让 agent 错过关键约束。这一发现把"Progressive Disclosure 是否更好"这一直白问题升级成"哪些任务面应该用 Progressive，哪些任务面应该用 flat"的方法学决策问题——几乎可以预见会成为下半年 agent skill 写作圈的引用标杆。

哲学定位：把 SKILL.md 圈子从"怎么写"推向"怎么测"。过去半年大量工作教 agent 团队"如何写出可复用、可注册、可治理的 SKILL.md"（5-13 SKILL.md / 5-25 SkillSafetyBench / 6-09 SKILL.nb），但"怎么知道我写得对"这一元层一直缺位。SkillJuror 是首个把"写作范式选择本身"立成可受控对照实验的框架——它对位 ML 圈的 ablation study 文化，把 agent skill 写作从"艺术"推向"实验科学"。这与 6-08 AARR/AARRI-Bench（arXiv 2606.07462）"把 harness 与模型解耦报告" 在精神上同源——都是把"写法 vs 模型"两条混淆变量切开。

开源价值：可复现的 ablation 平台。GitHub zhiyuchen-ai/skill-juror 提供 82 任务 SkillsBench + 语义受控变体 + 匹配多试验 runner。任何在生产中维护 SKILL.md 的团队（包括 Anthropic Claude Code、Devin、Manus 等）都可以把自家 skill 文件接入做组织变体 ablation——这条复现链是过去半年缺位的工程基础设施。

对你三个研究方向的启发
Harness Engineering（直击毕设）

SkillJuror 给你的 KV-cache 毕设的最直接启发是：把"系统行为"与"系统结果"分两层报告。当前 ML systems 论文绝大多数只报告 hit-rate / latency 等结果指标；SkillJuror 强调行为指标先于结果指标。建议在毕设里同时报告：(a) 结果层—— hit-rate、平均延迟、p99；(b) 行为层—— 平均每个 token 实际访问的 KV-cache 块数 / 预取触发率 / 驱逐前的平均访问次数 / 双水位线越线频率；(c) 组织层 ablation—— 用 SkillJuror 的"语义受控变体"思路，把 cache 组织方式（线性 / 分块 / hierarchical）做成剥离出来的独立变量，与 cache 大小 / 算法分开报告。这条做法直接对位 SkillJuror 的"把组织方式从内容剥离"哲学，能让你的论文具备 Anthropic / Google 系统圈子的方法学严谨度。

Agent Skills Safety

SkillJuror 与今日方向 #2 Impossibility of ELK 的耦合最深远：当 ELK 不可能性定理证明仅靠行为反馈无法保证 agent honest，自然推论是——必须把行为内部结构纳入评测。SkillJuror 的"轨迹证据"恰好是这条路径的工程化：不只看 agent 输出对不对，还看 agent 内部访问了哪些资源、用了哪些资源、依赖哪些 skill。把这两者合起来：未来 agent safety benchmark 必须同时报告 (a) 行为输出 / (b) 程序性轨迹 / (c) 内部表征——三层证据缺一不可。这条结合也直接呼应 5-19 memory laundering 的"sub-threshold 状态污染需要轨迹层证据"。

Safety Benchmark

SkillJuror 的方法学三件套（语义受控变体 / 匹配多试验 / 轨迹证据）是所有未来 agent benchmark 都应该采纳的实验设计原则。今日方向 #3 SciConBench / SciConHarness 的"clean-room evaluation" 与 SkillJuror 的"语义受控变体"形成对位——两者都试图把变量从混淆中分离出来。建议：未来 agent benchmark 报告时必须显式声明：(a) 哪些维度做了对照（控制了什么）；(b) 哪些维度仍是混淆（可能解释看到的提升）；(c) 结果层 vs 行为层的指标分离。这一姿态在 6-03 What Benchmarks Don't Measure "safety-usability tunable"、6-09 MAC-Bench "CSR / MG 双指标"、6-10 PreAct-Bench "Prefix Foresight F1" 之后，构成 2026 H2 benchmark 方法学的统一姿态。
