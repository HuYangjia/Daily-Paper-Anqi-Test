一句话核心贡献
把可复用 agent 产物（代码 / 工作流 / 程序性记忆）从"一次性技能"升级为"带证据、带门、带回退、带版本"的生命周期治理对象。
摘要（中文翻译，忠实原文）

AI agent 越来越倾向于把过往经验转换成可复用产物——代码、工作流、程序性记忆。复用能提升效率，但也带来一个生命周期可靠性问题：曾经成功一次的产物，会在环境漂移、任务欠规约、任务分布变化下失效，尤其是在 web 自动化场景中。

我们提出 SKILL.nb——一个用证据校准的生命周期策略来治理可复用 agent 工作流的框架。SKILL.nb 采用选择性形式化（selective formalization）：由执行证据决定哪些工作流步骤应当被固化为可执行代码、哪些应当保留为自然语言引导，以及何时应该修订这些选择。工作流被存为可审计、可版本化的 notebook，其中交织自然语言指引、多语言可执行单元、验证门、回退路径，以及输出 / 截图 / 错误轨迹等多模态证据。

在运行时，门控执行（gate-conditioned execution）让每一步在 gate 通过时执行代码，在漂移使可执行实现失效时本地回退到自然语言路径。

实验结果：在 WebArena-Verified 上单轮成功率 53.7%（领先最强基线 +3.9 pp），三次重执行后保留初始成功任务 91.7%（次优 +15.5 pp），受限修复下后续失败恢复率 72.9%，修复后回归率仅 4.2%（持久化基线 15.0–17.0%）；在 Mind2Web 的 cross-website / cross-domain 划分上同时领先；在 GitLab 真实版本迁移（15.7→16.11→18.9）上保持基本持平（−1.7pp / +0.6pp）。这些结果把生命周期治理与门控执行立成一次性任务成功率之外的可靠性新维度。

核心内容解读（"产物的两种命运" / 选择性形式化 / 门控执行 / 与 SKILL.md 的延续）

立论：可复用产物的"第二次失败"问题。过去半年（2025 H2 → 2026 Q2）"把经验当代码用"已经成为 agent 工程的主旋律：5-13 SKILL.md 把技能立成可声明 artifact、5-25 Code as Agent Harness 把代码立成 agent 推理基质、6-03 SkillDAG 把技能选择改成类型化有向图、6-08 Lean4Agent 把工作流抬到证明系统。但所有这些都默认产物"第一次成功 = 长期可用"。SKILL.nb 第一次直面"第二次为什么失败"。

"选择性形式化" vs "全量形式化"。SKILL.nb 不是把所有步骤都写死成 Python 代码——那条路 5-25 Code as Harness 已经走过，问题是环境漂移会把代码立刻打回原形（GitLab 升级到 16.11，按钮 selector 变了，硬编码全废）。它的核心赌注是：每个步骤是否值得形式化，由它过去的执行证据决定。

形式化候选证据信号命运
多次执行成功 / 输出结构稳定 / 环境无漂移"固化"为可执行代码 cell（Python / Bash / SQL 等）下次执行直接运行，省 token、可审计
偶发漂移 / 多模态证据不一致 / 输出格式不可重现"留"在自然语言指引 cell下次由 LLM 重新决策（保留弹性）
原本固化的代码 cell 连续触发 gate 失败"降级"回自然语言（可后续再升级）避免"已死代码"长期占据资源

这种"每个 cell 自己有命运"的工程哲学，在 ML pipeline 里早就成熟（DVC / MLflow），但首次被搬进 agent workflow，并且把"升级 / 降级"的决策权从开发者交还给执行证据。

门控执行（gate-conditioned execution）：让每一步带契约。每个 cell 都关联一组 gate——可以是结构化输出 schema、关键截图相似度、错误轨迹模式匹配等。Gate 通过 → 跑代码；Gate 失败 → 落到自然语言 fallback。这把"把代码 + LLM 缝起来"这件事从 prompt-spaghetti 变成了形式化的 if-then 控制流，且整张 notebook 是可审计、可版本化的——这与 5-19 ANNEAL（governed 符号补丁 + canary）、6-08 Lean4Agent（Lean4 类型作为外部凭证）共同构成"外部凭证驱动的 agent 工程"梯度。

多模态证据驻留：bug repro 第一次成为 agent 一等公民。每次执行的输出、截图、错误堆栈都被驻留在 notebook 里——这意味着调试 agent 失败不再是看一行 trace string、要回到工作流第 N 步的当时上下文就能恢复。这条对应于 5-13 TRIVIA+ "desiderata 显式化" 的延续：当评测要求验证 agent 的可恢复性，证据本身必须可重放。

实验信号怎么读。论文最值得划重点的不是 53.7% 单轮 SOTA（领先 +3.9 pp 在 WebArena 上是常规水平），而是三个跨期指标： (1) 三次重跑保留 91.7%——比次优持久化基线高 15.5 pp，说明"不在第二次掉线"是 SKILL.nb 真正打开的维度； (2) 修复后回归率 4.2% vs 基线 15–17%——意味着"修了之后不二次坏"，这是 ANNEAL 之后又一篇正面验证 governed repair 的工作； (3) GitLab 跨版本（15.7→18.9）持平——是 web agent 圈第一次在真实版本漂移环境测试持久化的工作。 整套 90.x 区间报数本来在 web automation 评测里都偏高，但 SKILL.nb 的真正贡献是把"横切的、不止报告 single-run accuracy"立成 agent 评测的第六维度——和 5-26 Gu 的 system scaling 六维评测无缝对齐。

研究路线意义：从 IR 升档到 lifecycle 升档。2026 年 H1 三条 harness 主线终于收口：第一条是 IR 抬升（代码 → 类型图 → 类型证明），第二条是系统级编排（CheetahClaws / OpenClaw / Claude Code 等参考实现），第三条是 SKILL.nb 给出的生命周期治理。三条线叠加，agent 评测的"单轮成功率"将像 ML 圈的"单 epoch test acc"一样被永久淘汰，取而代之的是 (成功率 × 可保留率 × 修复后稳定率 × 跨环境鲁棒率) 的多维量表——这正是 SKILL.nb 在 4 个数据点上同时给出贡献的原因。

对你三个研究方向的启发
Harness Engineering（直击毕设）

SKILL.nb 给你的"稀疏注意力 KV-cache 预取 + 双水位线 GPU 热缓存"毕设提供了一种非 RL、非微调的策略升级路径：把"哪类 attention pattern 适合命中预取"从一次性 heuristic 升级为带证据校准的 cache 策略 cell——多次命中的模式固化为静态策略，偶发模式留作 LLM 在线决策；当 attention pattern 漂移导致连续 miss，触发 gate 自动降级回自适应水位。建议：在毕设的 ablation 章引入"策略持久率"指标——经过 N 次 inference 后仍保留有效的预取策略比例。这与 SKILL.nb 的"三次重跑保留 91.7%"指标同源。同时，"修复后回归率 4.2%"启发 cache 自适应：当一个水位线被自动调整后，该水位线在新工作集上是否引发新 miss。这条评测维度比命中率更能体现 cache 策略的"稳态性"。

Agent Skills Safety

SKILL.nb 与今日方向 #2 的 VATS（错误路径注入）形成强对仗：VATS 揭示错误消息是隐式权威攻击面，SKILL.nb 的 gate 机制恰好提供了形式化反制——所有"工具错误返回"必须先过 gate 校验（schema / 关键字白名单 / 长度约束），失败则强制 fallback 到自然语言路径而不是直接喂给纠错推理回路。建议：把 VATS 的"夹心定位"攻击面纳入 SKILL.nb 的 gate 设计——error-cell 必须独立 gate。这相当于把今日方向 #2 暴露的攻击面用方向 #1 的工程范式天然中和。SKILL.nb 的版本化 notebook 也直接为 5-19 memory laundering 提供 audit 起点：所有跨步骤记忆传递都驻留在 notebook 里、可审。

Safety Benchmark

SKILL.nb 给 agent benchmark 提了两个新维度：(1) 三次重跑保留率——agent 第一次跑通的任务，经过环境状态轻微漂移后仍能成功的比例；(2) 修复后回归率——某个失败任务被修复后，是否破坏了原先成功的任务。这两条都不是单轮成功率能捕捉的。建议：未来 agent benchmark（包括今日 #3 MAC-Bench 的合规性评测）应同时报告 task-success / hold-rate / regression-rate 三件套。这与 6-03 What Benchmarks Don't Measure 提出的"safety-usability tunable"在精神上同源——评测面必须从"会不会做"扩展到"是否能保持做对"。
